## VERDICT: RATIFY-WITH-AMENDMENTS

The disposition primitive is the right law. It is the house calculus applied to the one place it was missing (debt-until-answered, record-forever, supersedable, actor-stamped), it preserves both s31 ratified forks untouched, and it beats the alternative that already exists silently: today an agent can clear three of the four orphan shapes by superseding the orphaned act itself — retracting a genuine historical fact from current truth to make debt disappear. The spec's premise "NOTHING can ever remove them" is therefore slightly overstated, but in the direction that strengthens its case: the only existing discharge path is a destructive one, and the disposition act replaces evidence-retraction with an honest, attributable record. That is the correct shape.

But the spec's closure statement is **false as written**, and I witnessed the counterexample live.

### The witnessed refutation (question C's verification, and it changes the scope)

The closure section claims `orphaned_by_retraction` is the ONLY `work_item_violations` member class with no answering act. I enumerated the current view (s33: eleven arms, nine classes) and probed the suspect member on a scratch schema (toy db, full s15→s33 chain applied, schema `adr14rev`, torn down after):

- **WITNESSED:** `INSERT ... work_depends_on, slug='probe-a', depends_on='probe-a'` with no edge type — the *default*, `informs` — was **accepted** (s30/s35 refuse self-edges and cycles only for `blocks-close`; "informs edges are not subject to this refusal" is the trigger's own text). `work_item_violations` immediately returned `dependency_cycle | probe-a`.
- **WITNESSED:** superseding that edge removed it from `ledger_current` (in-force edge count 0) — and the violation **persisted**, because the `deps` CTE reads raw `ledger`, retraction-blind, by declared history design.

So `dependency_cycle` is reachable in one ordinary CLI-shaped insert and is permanently undischargeable — a second member of exactly the class the spec claims has one member. `hooks/stop_clean_exit.py` counts it as blocking debt forever and its hint text ("record a corrected dependency") names a discharge that does not mechanically exist. This is the ADR-0000 2026-07-02 amendment's named failure verbatim: *the class gets named at exactly the scope of the fix the executor has already built.* The incident produced orphans, so the class was named "orphans." The actual class is: **violations-view members with no answering act, counted as blocking debt.** That is the heat-of-the-moment evidence the maintainer asked me to look for — the mechanism design is calm and sound; the *closure quantification* is incident-shaped.

The five defense-in-depth members (`duplicate_open`, `shipped_without_witness`, `dangling_parent`, `parent_cycle`, `blocks_close_cycle`) are construction-refused and vacuous under ordinary operation, but if a trigger bypass ever writes one, it too is permanent debt with only the fail-open breaker — worth one sentence in the spec, not a mechanism.

### Amendments (enumerated, concrete)

**A1 — Re-scope the act or the claim.** Either (a) generalize the kind to `work_violation_disposition`, answering any in-force violations member with the same validation posture — this forecloses the class per ADR-0011 Rule 4 and makes the spec's own "future members must state their discharge path" precedent mechanical — or (b) keep the orphan-specific kind and correct the closure statement to name `dependency_cycle` as a known second undischargeable member, routed separately. If (b), the sibling hazard needs its own disposition: post-s30, an informs cycle is arguably a false-positive class (informs never gates; a mutual informs citation is legitimate), so narrowing `dependency_cycle` to blocks-close edges — which are already cycle-refused at construction, making the member properly vacuous — may be the true type fix. That narrows a ratified view's semantics and routes to the maintainer either way. I lean (a) for the act plus the informs-narrowing as a sibling delta; but both routes are defensible.

**A2 — The ASP twin is missing from the spec.** `engine/lp/work_items.lp` mirrors the four orphan CTEs exactly (its own header, line 114) and `engine/ledger_floor.py` twins them. Narrowing the SQL member on in-force dispositions without the matching `.lp`/floor change breaks the SQL/ASP differential — the project's standing drift detector, and per project memory the deductive layer is the point. s31 shipped its engine companion in the same delta; s37 must too. The spec's element list has no engine item. This is its largest concrete omission.

**A3 — "One disposition per orphan" must be in-force-scoped, and the spec must say so.** If the builder copies the slug-burn idiom (raw-history uniqueness read), a superseded wrong disposition burns the orphan forever and the trap re-mints one level up — the exact incident shape again. The refusal must be "a second disposition while one is IN FORCE"; after superseding a wrong one, a new one is writable. One sentence forecloses a likely mis-build.

**A4 — Disposition validity must be bounded (the second addendum's I7, made mechanical).** A disposition whose basis lapses must stop answering. Concretely: `retired` on an `orphan_children` member whose basis is "child closed and settled" — if that child's close is later defeated (which the maintainer's constraint requires to stay possible, and which s31 permits — see below), the child is live again, its parent-edge dangles into a burned slug, and a stale disposition waves the debt through. Same for `reissued`: if the cited successor row is later retracted, the answer has lapsed. Fix inside the view narrowing the spec already writes: an orphan drops out only while an in-force disposition answers it AND the disposition's checkable condition still holds (`retired`: the orphaned item's own acts still read settled; `reissued`: the cited witness row is in force). This is BRIEF invariant I7 (validity bounds on every discharged obligation) applied to the new obligation kind, and it costs one join.

**A5 — Disposition acts should carry the house review discipline.** Every other debt-removing act in this calculus is review-disciplined: `work_closed` *requires* witnessed-or-deferred, and deferred creates countersignable debt. A disposition removes stop-gate debt with no review hook at all — below the house bar for exactly the act an agent under gate pressure has the most incentive to abuse (ADR-0013's self-serving-faculty warning). Reuse the existing machinery: dispositions accept the same witnessed/deferred posture, deferred landing in the existing review-gap path. No new mechanism; one column's discipline extended to one new kind.

**A6 — The stop-gate hook's teach-text, and the reissue cascade, need naming.** (i) `hooks/stop_clean_exit.py` line ~660: orphan rows currently fall into the else-branch that says "normally refused at INSERT... kernel/trigger anomaly; escalate" — wrong and misleading for a class that is legal-and-surfaced by design; the spec patches `led work violations` output but not this hook. (ii) Dispositioning an orphaned OPEN child as `reissued` entails retracting/re-opening that child under a new slug, which orphans the child's own surviving claims/edges one level down. The mechanism is closed under this recursion (each new orphan is dispositionable), but the spec should say so, because it is exactly where the maintainer's subtree intuition is right — see below.

Minor builder notes, not amendments: the validator should establish "is an in-force orphan" by re-deriving the orphan predicate for the given ledger id, never by parsing the view's `detail` string; `work_orphan_history` needs its GRANT and its `gates/ledger_reader_allowlist.py` classification; `judge.tmpl` also reads the view (RED on non-empty) and inherits the fix silently — worth adding to the spec's consumer list, which currently names only the stop-gate and distance-to-clean.

## The maintainer's subtree question, answered plainly

You asked what the problem is with superseding the entire subtree rooted at the superseded node, suspecting combinatorial explosion but noting the semantics is clean. The combinatorics is real but it is the *third* problem, not the first. Two things refute subtree re-issue as the primitive:

**1. The subtree is not closed under reference.** Retract-and-re-issue with new slugs throughout only works if everything pointing INTO the subtree moves with it. It doesn't: `blocks-close` and `informs` edges from items *outside* the subtree name inside slugs; `estimate:` rows join on slug; decisions and refs cite slugs and row ids; reviews cite close-row ids. Those rows are owned by other acts (append-only — you cannot edit them), so an "atomic" subtree re-issue either expands to supersede every referencing row too — the cone grows until it is not a subtree operation anymore — or it leaves dangling references. Either way "no orphans ever exist" is not achieved; the orphan class just moves from the typed parent-edge (visible, and now dispositionable) to untyped references (invisible). Clean semantics on the tree, at the price of silent breakage off the tree.

**2. Re-issuing a closed child cannot re-issue its reviews without forging agency.** A re-issued close is a new row — stamped by whoever runs the migration, not by the original closer. The review that countersigned the old close regards the old row id; it cannot regard the new one, and writing a new review row in the reviewer's name is exactly the "trigger-authored rows forge agency" move your own composite-discharge spec rejected. So re-issued closed children either arrive unreviewed (their review debt resurrects — churn) or the migration exempts them from review (a laundering hole). The recursion you asked about — "does a re-issued closed item need its closes, reviews, attests re-issued too?" — has no floor that is both honest and quiet. The disposition design sidesteps this entirely: settled acts stay where they are, and one small recorded act answers the dangling edge.

**3. Combinatorics/churn** — your own suspicion — is real (N mechanical rows per repair, burying the one-column signal in N−1 non-changes, against ADR-0000's spirit that the record states the actual delta) but is secondary to the two above.

**Coexistence: yes, and I recommend it.** The disposition act is the correct kernel primitive. Subtree re-issue is the correct *operator convenience* for the open-descendants cascade (A6.ii): a `led work supersede --cascade` style verb that re-opens surviving open children under the replacement slug and writes their `reissued` dispositions citing the successors, one witnessed act per step, all built ON the primitive. Closed/settled descendants stay put with `retired` dispositions. Kernel law stays small; the ergonomics live in the CLI where they belong. One honest cost of the disposition path to note: for settled children the tree topology to the new parent is not restored (no reparent kind exists; parent is fixed at birth) — the linkage lives in the disposition rows and the successor's refs, which is auditable but not walkable by `work_item_descendants`. I judge that acceptable for settled items; if rollup-walkability over repaired trees ever matters, that is a separate, deliberate spec.

## The addenda (closed items must stay supersedable/invalidatable)

The constraint is already satisfied by the landed kernel, and the disposition spec does not restrain it anywhere. **WITNESSED** on the same scratch schema: `work_closed` row inserted (deferred review) → item reads `closed`, appears in `work_review_gap`; the close superseded → item reads `open` in the same read, review-gap row gone; the defeated close remains in raw history, linked by the `supersedes` FK. That is BRIEF I3 exactly (append-only correction, linked, prior state never obscured), and s33 §3b gives the same for reviews: a superseded attest un-discharges its leaf on the next read and every derived ancestor re-opens — IEC 60880's change-through-re-verification linkage as a derived fact rather than a procedure. Note the weighing consequence: the maintainer's constraint cuts *against* subtree re-issue, not for it — re-issued copies carry defeat-linkage to the originals only as prose, and re-issued reviews are unobtainable without forged agency, so bulk re-issue degrades precisely the I3/60880 linkage the addendum names. `retired` in the disposition spec is a description, never a protection — nothing in it makes any close un-supersedable — but amendment A4 is required to make the disposition itself defeasible-in-effect when a later defeat invalidates its basis; without A4 the spec meets the constraint's letter and misses its I7 spirit.
