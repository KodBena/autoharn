## 0. The one-sentence diagnosis

The substrate is not badly built — almost every repetition is *named*, which is rare. What makes it haphazard in the categorical sense is that its recurring morphisms (the in-force projection, edge-closure, column extension, the validation fold, program stacking, the discharge join) are each **re-instantiated by hand per delta, with coherence asserted in comment prose** ("UNCHANGED, byte-for-byte"; "re-verified NOT members"; load-order paragraphs) rather than **stated once and held by construction or by gate**. ADR-0000 Rule 2(a) read at the class level: the recurring defect class here is "a coherence condition that lives only in prose" — cancer G of ADR-0012's checklist, applied to the kernel's own authoring idiom. The refactor is to move each such condition to (a) a single definition it holds by construction, or (b) a mechanical gate (ADR-0011 Rule 2), never more prose.

## 1. Findings — where, specifically (suspects confirmed or refuted)

**F1 — The in-force projection is re-authored per reader. CONFIRMED, and already ratified for fixing.** Mechanical count: ~52 inline `NOT EXISTS (… supersedes = …)` copies across `kernel/lineage/*.sql` (7 in s15 alone, 3–5 per later delta). Worse, per-reader *discretion* exists: `engine/lp/review_gap_audit.lp` declares its superseded check "SINGLE-HOP … a different consumer's semantics"; `work_item_current` and `work_item_strict_blockers` quantify over raw `ledger` (the composite-discharge spec §3b names this the "unenumerated matrix"). This is the springboard's naturality failure, and `design/FABLE-SUPERSESSION-UNIFORM-RETRACTION-SPEC.md` §2/§4 is already the correct categorical answer: one arrow `ledger → in_force` that every current-truth reader factors through, a two-valued reader typing, and an allowlist gate to foreclose the class. Nothing I propose below fights it; several items should ride the same delta.

**F2 — Per-delta re-issue of column-complete views. CONFIRMED as mechanizable, REFUTED as collapsible-by-`*`.** `ledger_current`/`countersigned_in_force` are re-issued in all 15 lineage files; each re-issue hand-copies a now-31-column list, and each closure statement hand-re-verifies the "NOT members" universe. The repetition is *forced by Postgres* (a `SELECT l.*` view freezes its column set at creation — the s20 defect), so the collapse target is not the view text but the *authoring morphism*: the map "append column c" ⟹ "re-issued views" is a functor that can be computed from the catalog. Mechanism: a generator that emits the re-issue DDL from `information_schema` + a declared exclusion manifest, and a `.detect` gate asserting view columns = table columns minus declared exclusions. The gate converts the closure statement's hand enumeration from prose into checked output while leaving the frozen files alone.

**F3 — The hand-rolled walker family. CONFIRMED, with a refinement.** Ten SQL `WITH RECURSIVE` instances and four ASP `_star` pairs are instances of one transitive-closure construction (s22 `reach`, s28 `anc`/`parent_anc`/`tree`, s29 `tree`, s30 `reach`/`bc_reach`, plus `ledger_floor.py`'s `sup_star`/`support_star`/`reach`/`tree`). But the deeper P1 violation is not the recursion skeleton — it is the **edge-source semantics** re-derived per walker. The parent/depends-on **direction-flip comment block** ("a work_depends_on edge walks OPPOSITE its own column names…") is copied verbatim in at least three homes: `s29-…sql:466-473`, `s30-…sql:382-391`, `engine/ledger_floor.py` (work_review_floor_atoms), and `work_review.lp`. That is the fact "what is an obligation edge, and which way does it point" with four hand-authored homes — exactly ADR-0012 P1/B. And the substrate is honest about what it can abstract: **SQL can state the objects once (named edge views) but cannot parameterize the recursion over a relation name; ASP can state both** (a kind-indexed `star(K,X,Y) :- edge(K,X,Y). star(K,X,Z) :- edge(K,X,Y), star(K,Y,Z).` — `ledger_support.lp`'s kind-tagged `support_edge/3` already proves the idiom in-house). `ledger_floor.py`'s `_base_ctes` is the existing stated-once precedent on the SQL floor side; the kernel views deserve the same move at the edge-source level.

**F4 — `validate_work_item` as an accreting monolith. CONFIRMED.** Defined in s22, wholly re-issued in s28, s29 (twice, with the sec-10 amendment), and s30 — each time copying every prior branch with only a prose assertion of byte-identity. That is a hand-maintained fold over the delta chain; the coherence condition ("prior branches unchanged") is checkable by no mechanism today. One silent mutation in a re-issue would ship. Same shape, smaller: `validate_independence()` (s17→s21→s29).

**F5 — Kind-scoped shape CHECKs. CONFIRMED as a pattern; the variants are load-bearing.** Ten-plus constraints of the form `(kind='X') = (col IS NOT NULL)` or `col IS NULL OR kind='X'` encode a discriminated union (the ledger row as a sum type over kinds) with hand-written injectivity conditions per column. The one-way/two-way split is *not* accidental: s30's one-way form is forced by append-only history, s28's by nullable-legal roots. So do not uniformize the shapes; state the (kind, column, arity) table once as a manifest and gate the catalog against it — an unlicensed payload column, or a payload column with no shape CHECK, is refused mechanically.

**F6 — The discharge join has ≥4 SQL homes. CONFIRMED (a suspect the brief did not list).** "An un-superseded, distinct-actor attest review regarding row R" is hand-copied in `review_gap` (s15), `countersigned_in_force`, `work_review_gap` (s29, whose own header says "mirrors review_gap's own join shape exactly"), `work_item_strict_blockers`' `review_unresolved`, and `ledger_floor.py`'s `discharged`. One derived view (`discharging_attest(regards_id, reviewer)`) is the single home; every consumer joins it.

**F7 — .lp composition conventions. CONFIRMED, with a real hazard.** Load order lives in per-file comment prose; each producer pair has "its own bespoke scratch-differential script" (s22's header, verbatim); `judge` hardwires `ledger_tnow.lp` only. The `#defined` guards are deliberately fail-soft — but that means **mis-stacking grounds silently**: load `ledger_support.lp` without `ledger_tnow.lp` and `in_force/1` is empty, so `exposure/2` is empty, which reads as "no exposure" — the corpus's own F49 not-shown-conflated-with-false class, at the module seam. Also, the supersession spec §3 supersedes `work_items.lp`'s "never whole-row superseded by design" standing-alone premise on the record, so the stacking graph is about to change anyway; the module registry should land with it.

## 2. The abstraction ladder (what the substrate can and cannot say point-free)

- **Can state once by construction (SQL):** the in-force projection (one view); the edge relations with their directions (named edge views, including the union edge `obligation = parent ∪ blocks-close`); the discharge join (one view). Consumers become compositions of these.
- **Can state once by construction (ASP):** both edges *and* closure — one kind-indexed `closure.lp` over `edge/3`, following `support_edge/3`'s existing idiom.
- **Cannot state once in SQL; mechanize as a gate instead:** the closure *functor* itself (no parametric recursion over relation names without dynamic SQL — leave per-consumer `WITH RECURSIVE` skeletons, but over the one edge view); the column-complete re-issue (generator + detect gate); byte-identity of re-issued function branches (detect gate); the kind-shape manifest (catalog gate).
- **Not expressible without changing the system's character (aspiration or anti-indicated):** genuine reinstatement (recursion through negation — the supersession spec correctly parks it as "a new spec, not a patch"); generating both producers from one source — this one is not parked but **anti-indicated** for the judged predicates: the SQL/ASP differential's entire value is two independent hand-encodings (I6; `ledger_floor.py` even duplicates `_wi_quote` on purpose). Abstracting *up* must stop at the producer boundary: shared EDB, shared harness, independent logic.

## 3. Ordered plan — smallest first, each independently shippable

All targets are LIVE definitions and going-forward idiom; frozen sNN files are never touched (ADR-0005 Rule 8). New constructions enter via the next deltas in the birth chain.

1. **(a) The in-force factoring delta** — build the supersession spec's §4.1 delta *as* the factoring refactor: first CREATE the one in-force projection, then define every re-issued reader ON it (`work_item_current`, `strict_blockers`' tree/closes, `question_status`, `work_review_gap`), plus `orphaned_by_retraction`. Acceptance: the spec's own §5 list, plus a grep-level assertion that the delta introduces zero new inline `NOT EXISTS (supersedes)` outside the projection.
2. **(b) The raw-reader allowlist gate** (spec §4.3). This forecloses the class (ADR-0011 Rule 4); everything else in this plan is easier once it stands. Acceptance: red-polarity fixture refused with teach-text naming both discharge paths.
3. **(a) Named edge views + `discharging_attest`** — one home each for `work_edge_parent`, `work_edge_blocks_close`, `work_edge_obligation` (direction semantics documented once, at the definition), and the discharge join. Re-point `strict_blockers`, the `would_cycle` functions (keep their seeded per-call walk shape — it is O(chain) at trigger time where a full-star view is O(edges²); collapse the edge *source*, not the walk), and `work_item_violations`' members. Rides naturally in or immediately after delta 1, since those objects are re-issued there anyway. Acceptance: before/after output equality on a fixture world; all existing refusal polarities re-witnessed; `./judge` AGREE.
4. **(a) `closure.lp`** — one kind-indexed closure module; `work_items.lp`/`ordering_violations.lp` successors express their stars over it when the work layer joins the tnow stack (the supersession spec's §4.2 engine companion forces re-issue anyway). Acceptance: banked derivations byte-identical (the §1.6 idiom).
5. **(b) Column-complete mechanization** — the re-issue generator + `.detect` gate (F2). Acceptance: gate red on a synthetic missing column, green across the scratch chain; the next delta's closure statement cites gate output instead of hand enumeration.
6. **(b) Kind-shape manifest + catalog gate** (F5). Acceptance: red on a synthetic payload column lacking its licensing CHECK.
7. **(b) Validation decomposition for s31+** — per-concern trigger functions behind a thin dispatcher (each future delta replaces only its own leaf), plus a byte-identity `.detect` for whatever must still be re-issued wholesale. One caution to witness on scratch first: s30's default-then-check ordering (`NEW.edge_type := 'informs'` before the blocks-close refusals) must stay within one function or under enforced trigger naming. Acceptance: every existing refusal polarity re-witnessed with unchanged error text.
8. **(b) The .lp module registry + one generic differential harness** — a declared `provides/requires/stands_alone` registry consumed by one runner and by `judge`; a mis-stacked invocation gets a typed refusal instead of silent empty grounding (closing the F49-shaped hazard in F7). Acceptance: judge runs the work-layer differential per the supersession spec; red polarity = deliberately missing `requires` refused.
9. **(c) Parked, named:** delta-as-data (generating whole sNN deltas from a typed kernel manifest); reinstatement semantics (already parked by the supersession spec §1); cross-producer codegen (anti-indicated, see ladder — record it as a standing refusal, not a someday).

## 4. What should NOT be abstracted

- **The two producers.** SQL floor vs .lp duplication is the differential's substance (I6/ADR-0009-composed P6). Collapse the *harnesses*, never the producers.
- **The two edge relations at the fact layer.** `work_parent` vs `work_depends_on` stay separate base facts (s28's own "deliberately DIFFERENT relations"; ADR-0008's fuzzy-match warning). The generic `edge` object of step 3 is a *derived* view over them — collapse in the derived layer only.
- **`work_witness` vs `work_review_ref`** (s29's named choice) — two facts, two columns.
- **One-way vs two-way shape CHECKs** — the variance is forced by append-only history; uniformizing would break history-safe applies (s30's HISTORY section is the proof).
- **The belt-and-braces provably-vacuous violation members** — deliberate defense in depth in both producers; keep.
- **Frozen lineage and per-delta closure statements as *records*** — the re-verification evidence stays per-delta; only its production becomes gate output instead of hand prose.
- **`review_gap_audit.lp`'s single-hop semantics** — it deliberately mirrors the kernel view it audits; under the new reader typing it gets *declared* (history-reader with reason), not "fixed."

## 5. Uncertainties

- Postgres `CREATE OR REPLACE VIEW` restrictions on dependent views may constrain how `countersigned_in_force` composes over `ledger_current`; the generator approach sidesteps this but needs a scratch-chain witness before the idiom is committed.
- Multiple-trigger ordering (alphabetical firing) for step 7 needs a scratch witness; the dispatcher-with-leaf-functions shape avoids it but adds one indirection — pick on evidence, not preference.
- I read s26/s27 only at header depth; if the hash-chain trigger reads any structure steps 1/3 re-home, that is an unswept corner to add to the supersession delta's enumerated universe (its §6 already reserves the mechanism for exactly this).
- Whether the springboard's "one two-valued typing judgment" should also bind the *engine* readers (each .lp's EDB exporter) is not settled here; the module registry (step 8) is where that declaration would naturally live.
