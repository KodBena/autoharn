# PART 1 — CORRESPONDENCE

## 1(a) Delta-by-delta, s44–s49

**A preliminary divergence in the model's own honesty apparatus.** The header's disclosure (`design/Autoharn.idr:29`, `:77-95`) names exactly one lagging delta: "LAGGING: s45-standing-lifecycle not yet transcribed." The actual gap is six deltas (s44–s49). The freshness gate still fires its WARN (it computes the actual head mechanically), but the gate's own header admits it checks presence-of-marker, never content — and the model's HONESTY CONTRACT (`Autoharn.idr:23-27`) is precisely that gaps are *named*, not silent. Five of the six are currently silent. The line "AS-OF stays s43 until a future pass pays this down" (`Autoharn.idr:93`) is now itself a stale claim about the size of the debt.

**s44 — model-identity attestation** (`kernel/lineage/s44-model-identity-attestation.sql`). What it adds: a 25th kind `model_identity_attested` (lines 239-250); seven `attest_*` columns with two-way kind-shape CHECKs, closed grade/verdict vocabularies (lines 306-347); a self-referencing FK `attest_row_id REFERENCES ledger(id)` (lines 262-263, "the target's existence is thereby structural, never CLI-side"); the structural expectation/verdict coupling `(attest_expected IS NULL) = (attest_verdict = 'unevaluated')` (lines 350-353); hash re-issue to 65 columns; the `model_attestations` view; supersession deliberately allowed (contrast s43 R6). **Model: absent entirely.** No payload constructor, no derived read, no mention. Notably, every mechanism here is squarely inside the model's demonstrated idiom: the FK is exactly `Fin n` (the model's signature move); the closed grade/verdict vocabularies are closed sums; the expectation/verdict coupling is a one-line type function in the `Gated`/`witnessTy` family — e.g. an index `Maybe expected` determining whether the verdict type is `{Match, Mismatch}` or `{Unevaluated}` would render the coupling as unrepresentability, *stronger* than the CHECK, the same relationship `CompetenceValueF` (`Autoharn.idr:444-446`) already has to its CHECK pair. This is the cheapest of the six to transcribe faithfully.

**s45 — standing lifecycle** (`kernel/lineage/s45-standing-lifecycle.sql`). Named as lagging, so not a silent gap — but the divergence is worse than "not yet transcribed": on two points the model now **permits what the kernel refuses**, and on one it **cannot represent what the kernel accepts**:

- *Lift is unrepresentable, and its nearest encoding is wrong.* `PPrincipalSuspended` (`Autoharn.idr:475`) carries no active flag. s45 Element 1 (SQL lines 362-367) licenses `principal_binding_active` on suspensions; a lift is a superseding same-kind `active=false` row. In the model, the only way to write a same-kind superseding row is another `PPrincipalSuspended`, which `principalStanding` (`Autoharn.idr:845-869`) reads as an in-force suspension — the model reproduces exactly the pre-s45 defect s45's own header calls "worse than unbuilt" (SQL lines 110-133). Same for `PStandingDeclared` and the unbind.
- *Revocation is not terminal in the model.* s45 Element 4 (SQL lines 491-536) refuses cross-kind supersession of the three lifecycle kinds. The model's `boundaryOk` (`Autoharn.idr:1180-1207`) checks supersession targets only for `PWriteRefused` (`unretractableOk`); a `PProse KNote` row superseding a `PPrincipalRevoked` row constructs green, and `supersededIn`'s monotone read then flips the principal's standing back to `PsActive` — the exact silent-reinstatement hole s45 closes, still open in the model.
- *The three-valued governing read* (superseded / unsuperseded-inactive / unsuperseded-active; the resurrection-proof `kernel.principal_role`, SQL lines 382-403) has no model counterpart — `inForce : ... -> Bool` (`Autoharn.idr:589-590`) is two-valued. The LAGGING note itself names this correctly (`Autoharn.idr:88-92`).

**s46 — credited views** (`kernel/lineage/s46-credited-views.sql`). `model_defeated_rows` (lines 167-186: mismatch attestation joined to an active `model-identity-attestation` competence grant, defeat-input exclusion) and `credited_current` (lines 201-223). **Model: absent.** No defeat calculus, no credited projection. The model has all inputs (`PCompetenceGranted` with activity string; s44's attestation once transcribed), and §6's `DualProducers` (`Autoharn.idr:1354-1368`) does not know about `./judge --layer defeat` — the differential the model renders is the in-force layer only. The delta's own named divergence (ledger_current-scoped vs. full-history defeat-input exclusion, SQL lines 55-70) is exactly the kind of two-quantification-domains fact the model elsewhere makes its central subject (raw vs. in-force, §3 header) — currently unmodeled. See Part 2, item 4.

**s47 — claim-on-closed refusal** (`kernel/lineage/s47-claim-on-closed-refusal.sql`). A third claim-time precondition prepended to `validate_work_item_claim` (SQL lines 99-109): a slug with an in-force `work_closed` row is not claimable. **Model: diverges, green-vs-red.** `VClaim` (`Autoharn.idr:1072-1074`) carries exactly two premises (`opened`, `startOk`). Concretely: `ValidPayload worldA (PWorkClaimed "a")` elaborates in the model today — worldA (`Autoharn.idr:1383-1385`) carries an in-force close of "a" — while the s47 kernel refuses that same act. This is the sharpest correspondence defect found in this consult: not lag on new machinery, but a live legality claim the kernel contradicts, in the write-boundary section the model presents as its most faithful. The fix is one erased premise, `So (not (hasCloseCur l s))`, and the model *already owns* `hasCloseCur` (`Autoharn.idr:710-718`) — the same in-force read the SQL re-derives inline. Per CLAUDE.md's hazard rule I flag this loudly: until fixed, a reader using the model as the architecture tour will learn a false write-boundary fact, mitigated only by a WARN whose prose names a different delta.

**s48 — review-witness row existence** (`kernel/lineage/s48-review-witness-existence.sql`). A standalone trigger regex-extracts `row:<id>` tokens from `work_review_ref` on the two close-family kinds and refuses dangling ids (raw existence, not in-force; SQL lines 215-230). **Model: diverges by silence.** `reviewRefTy DWitnessed = NonEmptyText` (`Autoharn.idr:391-394`) — free text, no row-token semantics. The interesting fact: the model's own signature idiom (`Fin n` — "a reference to a later or nonexistent row is UNREPRESENTABLE," `Autoharn.idr:427-432`) is *exactly* what s48 enforces for the `row:` sub-shape, since existence-at-insert-time of a server-assigned id is precisely "earlier row." The faithful transcription is a three-armed sum (`RowRef (Fin n) | Commit CommitRef | Artifact NonEmptyText`), and writing it exposes an implementation smell — see Part 2, item 1.

**s49 — journaler overflow guard** (`kernel/lineage/s49-journaler-overflow-guard.sql`). Totalizes one partial cast inside `kernel.journal_write_refusal` (SQL lines 242-248): an over-bigint digit string now yields `v_attempted := NULL` instead of aborting the journaler. **Model: the post-s49 semantics is captured trivially and the defect class was never representable** — `PWriteRefused`'s `attemptedActor : Maybe PrincipalId` (`Autoharn.idr:508-511`) with `PrincipalId = Nat` (unbounded) has no range to overflow. Two honest notes the model should carry rather than leave to extrapolation: (i) the s43 out-of-model boundary note at `PWriteRefused` (`Autoharn.idr:500-507`, "boundary control flow, not row algebra") arguably covers s49, but the AS-OF bump needs to *say* s49 lands under that existing carve-out, not absorb it silently; (ii) the model's `Nat` vs. SQL's `bigint` is itself an undisclosed idealization — s49 is a defect that lived precisely in that gap (regex domain wider than bigint range), and nothing in the PRESERVED list names numeric range as an out-of-model boundary.

## 1(b) Steelman: the model as a fancier Haskell — and my judgment of it

**The steelman, made as strongly as honesty allows.**

1. *A large fraction of the file needs no Idris.* §1's vocabulary sums (`AgentClass`, `Relation`, `EdgeType`, `ReviewDisposition`, `ProseKind`, `Resolution`), the hand-rolled `Eq` instances, `isShipped`/`isWitnessed`/`isAttest`, and the whole of §2c's derived reads (`standingDecisionIds`, `danglingDepDebt/History`, `principalStanding`, `classOf`, `startBlockers`) are sums, records, and boolean folds over `allIds` — expressible identically in Haskell 98, close to it in OCaml, and the folds even in SQL. The dependent machinery in §2c is limited to `Fin n` as a list element type.

2. *The write boundary is proof-demanding, not verdict-producing — the kernel outgrew the model here.* `append` (`Autoharn.idr:1216-1220`) takes the validity proofs as erased *arguments*: the caller must already possess them. The kernel's own s43 revolution was the opposite shape — a **total** function returning a typed verdict, where refusal is a value and every refusal is a committed row. The model renders the refusal *row* (`PWriteRefused`) but not the refusal *judgment*: `boundaryOk` is a bare `Bool` (`Autoharn.idr:1180`), erasing which premise failed — precisely the information s43's `refusal_surface`/`sqlstate` columns exist to carry, and `RefusalSurface` (`Autoharn.idr:452`) already sits in the file unused by any function. Every premise in `ValidPayload`/`boundaryOk` is decidable, so the total form is cheap and beautiful: `write : Ledger n -> Entry Draft n -> Ledger (S n)` — the ledger grows by exactly one row on *both* arms (the accepted entry, or the journaled `PWriteRefused` row). That signature IS s43's "a refusal verdict cannot be delivered unjournaled," the very invariant the header declares out-of-model (`Autoharn.idr:37-41`, `:500-507`). The declared reason ("operational control flow, not row algebra") is half-true — the refusal_seq oracle and the journal-INSERT-failure leg genuinely aren't row algebra — but the core totality fact is typeable with constructors the file already has, and totality checking is the one discipline Idris enforces that nothing in the SQL stack can.

3. *Supersession discipline is a relation the kernel now has and the model does not.* s43 R6 and s45 §3.4 are two instances of one concept: *which payload kind may supersede which, under what identity continuity*. The model renders R6 as one ad-hoc case inside a Bool (`unretractableOk`, `Autoharn.idr:1187-1193`). A first-class `MaySupersede : Payload Recorded m -> Payload Draft n -> Type` would have made "what may supersede a revocation?" an unavoidable, enumerable question at s43 time — i.e., the model, sharpened, *predicts s45 Element 4*, the "conversion-found closure" the SQL side only discovered in the field (s45 lines 135-149). This is the steelman's best exhibit: a place where a stronger type in the model would have been a design instrument, and its absence coincided with the kernel discovering the hole late.

4. *The `Bool`-collapse recurs at standing.* `inForce : Bool` cannot express s45's three-valued governing read; the LAGGING note admits it (`Autoharn.idr:88-92`). The active-flag family (`CompetenceValueF`) exists for one kind but was not generalized to the lifecycle kinds when s41 landed it.

5. *Linearity and the rest of QTT are essentially unused* beyond erasure (`0` multiplicities on proofs — which the file does use well). No linear ledger handle, no session-typed slug lifecycle.

**My judgment of the steelman: mostly right about the gaps, wrong about the frame, and two of its upgrades are ceremony.**

The "fancier Haskell" charge understates what is there. The load-bearing dependent core is real and is exactly where the kernel's own invariants live: `Fin n` back-references make acyclicity and earlier-row reference structural (§1); `Stage`/`GradeF` make writer-supplied grades unrepresentable (r2a); `Gated`/`witnessTy`/`reviewRefTy`/`CompetenceValueF` are the CHECK-pair idiom rendered as Pi-types; `Projection` carries erased soundness (R6); `supersededStable` (`Autoharn.idr:1314-1321`) is a genuine machine-checked theorem — reinstatement-freedom as monotonicity — with the honest §5 note that Idris makes the *choice* legible rather than the rejected semantics unrepresentable; and the `failing` blocks are compile-time red fixtures, a both-polarity discipline no Haskell encoding gives you at elaboration time. The plain-sum sections are plain because the substrate is plain — that is the HONESTY CONTRACT working, not a deficiency.

Load-bearing upgrades, in priority order:
1. **The s47 `VClaim` premise** — not an upgrade, a correctness fix. One line.
2. **The total `write` verdict function** (steelman point 2). Genuinely load-bearing: it internalizes s43's central idea, gives `RefusalSurface` a job, and turns the model's premise list into an enumerable datum auditable against the SQL's refusal list — the correspondence surface that would have made the s47 gap visible as a diff between two closed lists.
3. **`MaySupersede` as a typed relation** (steelman point 3), which is also most of the s45 transcription: it carries R6, the three lifecycle kinds' same-kind/identity-continuity rules, and leaves everything else uniformly retractable.
4. **Active-flag indexing on the two lifecycle kinds + a three-valued `Governs` read** — the filed s45 debt, real but already honestly filed.

Ceremony, declined with reasons: linear/QTT ledger handles (the ledger is a persistent value being *described*, not a resource being *consumed*; linearity would encode nothing the substrate has); refactoring to the RefUniverse GADT-vs-manifest form (already adjudicated in the header, `Autoharn.idr:130-145`, on grounds that still hold); deriving/removing the hand-written `Eq`s (noise either way); modeling the hash chain or byte serialization (correctly ruled out at s42, `Autoharn.idr:43-51`); making every `boundaryOk` conjunct a named proof constructor *without* building the verdict function (names without totality just add reading cost — the value is in the total function, not the naming).

# PART 2 — LOWERING OPPORTUNITIES

1. **`work_review_ref` should be a typed sum, not regex-parsed free text — and s44 already proved the better factoring in this same kernel.** Restating s48 in the model forces the choice between transcribing a string-parse (ugly, and against the grain of the file's own `Fin n` idiom) or the honest type `RowRef (Fin n) | Commit CommitRef | Artifact NonEmptyText` — and the moment you write the sum, the implementation's shape is exposed: `s48-review-witness-existence.sql:217-229` regex-scans one text column holding three semantically distinct witness forms, checking only the one form it can, with the other two disclaimed in LIMITS (lines 133-138). Meanwhile `s44-model-identity-attestation.sql:58-59, 262-263` solved the identical problem the structural way the same week: `attest_row_id bigint REFERENCES ledger(id)` — "the FK is to the SAME table — the target's existence is thereby structural, never CLI-side." A future delta giving the row-witness its own FK column (with the existing text column retained for commit/artifact forms, discriminated by the house kind-shape CHECK idiom) would delete the regex, make s48's refusal a plain FK violation, and let the `led` verb stop encoding a typed fact as a string convention. This also rhymes with ADR-0000's 2026-07-18 amendment: a value (`row id`) currently crosses into a parsed textual position instead of a bound, typed carrier. **Kernel-lineage semantics — maintainer-ratification territory; proposed, not instructed.**

2. **The claim leaf's inline re-derivation of "closed" marks a missing one-home predicate.** In the model, "slug has an in-force close" is one function, `hasCloseCur` (`Autoharn.idr:710-718`), consumed by `strictBlockers`, `startBlockers`, and (after the s47 fix) `VClaim` — three readers, one home. In SQL, `s47-claim-on-closed-refusal.sql:103-106` re-derives `work_item_current`'s `closed` CTE inline, and its own closure statement spends nine lines (142-148) arguing this is not a second mechanism because "a cross-function call into work_item_current's own CTE is not representable in SQL without a second view/function of its own." The model's restatement answers that argument directly: that second function is not the feared second mechanism — it *is* the ADR-0012 P1 one home, and the trigger and the view would both become its readers. A small `STABLE` function (`work_item_closed(slug) RETURNS bigint`, trigger-safe raw-vs-current posture decided once at its definition) would collapse the duplication before a third reader appears (the s47 LIMITS already forecast one: "depends-edges on closed items, double-close" as the next axes). **Kernel semantics — propose only.**

3. **The LAGGING disclosure should be checked against the actual head — a gate improvement the model's current state motivates.** Witnessed in this consult: the LAGGING prose names s45 while the real gap is s44–s49, and `gates/idris_model_freshness.py` (its own header, lines ~40-50) checks marker presence, never content. A one-line tightening — require the LAGGING window to mention `s{actual}` (the mechanically-computed head) — converts "the disclosure has not silently under-enumerated" from hope into a checked property, exactly the gate's own stated class ("the claim that decays silently and unwitnessed"). This touches `gates/` only, not kernel or law — implementable under ordinary discipline.

4. **The s46 defeat-input-exclusion divergence should be forced to one quantification domain before it ships in a birth chain.** `s46-credited-views.sql:55-70` names, honestly, that the kernel view tests defeat-input membership over `ledger_current` while both engine producers test over full raw history, diverging on a kind-changing supersession chain. Rendering the defeat calculus in the model requires writing one `defeatInputs : Ledger n -> List (Fin n)` and *picking a domain* — the model cannot transcribe an unadjudicated fork, which is the point: the restatement turns a "named, vanishingly narrow" divergence into a decision that must be made. Since the engine differential is authoritative and this view is display-only, the cheap resolution is re-issuing the view's exclusion subquery to the engine's raw-history domain (the exclusion is the one place the s31 reader discipline arguably mis-applies: "was this row ever machinery input" is a history question, the same class as `LDuplicateOpen`, `Autoharn.idr:629`). **Kernel view semantics under a ratified spec — propose to the maintainer, never instruct.**

5. **The s45 transcription, done properly, doubles as a design probe on "correction" rows.** Indexing the lifecycle payloads by their active flag (the `CompetenceValueF` pattern) forces the model to enumerate what an *active* same-kind superseding suspension/revocation means (a rationale-correction) versus inactive (a lift/unbind), and what identity must be restated in each arm. s45's trigger (SQL lines 513-529) enforces subject/role continuity but treats correction and lift uniformly except on declarations; writing the typed arms will either confirm that uniformity is intended or surface a missing arm — cheap to find in the model, expensive to find in a born world. This is exactly the "model as design instrument" use the commission asks about, available as a by-product of paying the filed s45 debt.

**Summary of hazards flagged (per CLAUDE.md):** (1) the model's `VClaim` asserts a legality the s49-head kernel refuses (s47) — the one place the model is currently *wrong* rather than behind; (2) the LAGGING disclosure under-enumerates the gap by five deltas, and the gate cannot see that; (3) `idris2` is not on PATH (the freshness gate survives via its hardcoded fallback, but any hand-run `idris2 --check` fails with command-not-found — worth knowing before trusting a "gate green" that a human could not reproduce at a shell).
