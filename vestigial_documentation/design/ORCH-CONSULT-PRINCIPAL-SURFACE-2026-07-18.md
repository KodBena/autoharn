## 0. What the principal surface currently is (verified, not summarized from prose)

`kernel.principal` — four columns: `id`, `name` (UNIQUE), `agent_class` (CHECK ∈ human/model/subagent/tool), `acts_for` (nullable self-FK, "delegation; NULL = own right"). Created in s13, re-instanced s14/s15, and **touched by no lineage file since** — verified by grep across all of `kernel/lineage/` (references everywhere; `ALTER TABLE ... principal` nowhere). Registration is `INSERT ... ON CONFLICT (name) DO NOTHING`. Companion `principal_role` maps db_role → principal (other-assigned attribution). The `design/Autoharn.idr` model carries `PrincipalId = Nat` — a bare natural, faithfully mirroring the thinness.

The load this identity carries, meanwhile, is the heaviest in the kernel: `ledger.actor` NOT NULL FK on every row; `set_actor`'s connection-default stamping; `validate_review`'s same-actor refusal; the s17/s21 independence machinery; s29's computed `discharge_grade`; `obligation.assigned_by`/`obliges_actor` and the s32 `countersign_obligation`/`review_gap` debt views; s25 commission signing modes; and the `LED_ACTOR`/`resolve_actor` CLI surface, which by its own header comments covers four write paths and honestly names three (`work depends`/`close`/`resolve-violation`) as unwired — the exact gap the panel deployment witnessed live (five closes dispatched as `item-countersign`, all landing `actor=author`; backflow Finding 1 addendum, rows 1710/1716/1719/1732/1746).

Two live witnesses worth having in front of you before any design talk:

- The panel roster (queried this consult): 13 principals, ids 1–27 **with gaps** (5–6, 8–14, 16–20) — sequence values burned by silent `ON CONFLICT DO NOTHING` re-registrations. The silent-duplicate defect is visible in the id column itself.
- Three of the 13 are the same human (`commissioner`, `maintainer`, `bork` — all `human`), `acts_for` is NULL on every row in the deployment's history, and `reviewer` is `subagent` while `reviewer2` — minted a day later for the same job shape — is `model`. The relationship column exists and has never once been used; the class column disagrees with itself across siblings.

One structural observation that frames everything: **the principal table is the only load-bearing surface in the kernel that violates the kernel's own record discipline.** Every ledger row is append-only, attributed, timestamped, hash-chained. A principal row is mutable in principle, carries no registrar, no timestamp, no rationale, and its creation produces no ledger event. The identity layer that anchors ALCOA attributability is itself the one non-ALCOA artifact in the system. That is the type-level diagnosis ADR-0000 asks for, and most of the design space in §2 follows from it.

## 1. What the standards families actually say, mapped against this surface

The commission is right that this had to start from the standards. Method note: per ADR-0000 Revisit #4 Clause 2, this runs registry-rooted (from the standard toward the project). The registry (`law/STANDARDS-REGISTRY.md`) carries exactly one entry — NIST SP 800-53, entered 2026-07-12, marked NOT YET OPERATIONALIZED. This consult is, in effect, the first registry-rooted audit slice for the IA/AC families. The maintainer's surprise that principal-granularity "never came up in law/brief audits" has a precise answer: the founding BRIEF *does* oblige the relevant records (its I2, I6, G7, G9, G13), but every prior audit ran corpus-rooted and the principal table is exactly the artifact no document cites — the same silent-omission mechanism the what-did-we-miss RCA already named for read-logging.

### 1.1 NIST SP 800-53 rev 5 — IA and AC families

Researched this session ([IA-4](https://csf.tools/reference/nist-sp-800-53/r5/ia/ia-4/), [AC-2](https://csf.tools/reference/nist-sp-800-53/r5/ac/ac-2/), [IA-5](https://csf.tools/reference/nist-sp-800-53/r5/ia/ia-5/), [AU-10](https://grcacademy.io/nist-800-53/controls/au-10/), [SP 800-53r5 full text](https://nvlpubs.nist.gov/nistpubs/SpecialPublications/NIST.SP.800-53r5.pdf)). Paraphrase-grade confidence: high for control intent; I did not retrieve exact clause text for quotation, and say so.

- **IA-4 (Identifier Management):** an identifier is assigned only with *authorization*; it must identify the intended entity; reuse is controlled. autoharn: `register-principal` requires no authorization of any kind — any writer with INSERT mints identities. The UNIQUE constraint prevents literal reuse forever (stronger than IA-4's minimum), but the silent no-op means a *second registration attempt aliases onto an existing identity with no signal* — the "identifies the intended entity" property fails silently, which is the worse polarity. The panel deployment tripped this in production.
- **AC-2 (Account Management):** account types defined; **approval required before creation**; accounts created/enabled/disabled/removed per procedure; use monitored; periodic review. autoharn has none of this at the schema level — no approval, no disable, no removal, no review cycle. Notably, the panel deployment's invented role-name-governance process (§3 below) is a hand-built AC-2 approval workflow, reinvented from felt need without knowing the control existed. That convergence is evidence the control is load-bearing, not bureaucratic.
- **IA-2 (unique identification and authentication of users and processes acting on their behalf):** this is the deepest gap. A principal *name* in autoharn is asserted (`LED_ACTOR`), never authenticated. Authentication actually exists in the system — but at two *other* layers: the db connection (`principal_role`, other-assigned) and the invocation (the s17 HMAC stamp, explicitly a "tripwire, not authentication" by its own header). The principal layer itself has no authenticator: any writer can sign as any registered principal. Finding 31 (one context registering "reviewer" and countersigning itself) was exactly this control failing, and s17/s21/s29 responded by *routing independence claims around the principal layer onto the stamp layer* — correct, and honest, but it means the principal table's names now carry social/audit weight the kernel deliberately does not trust. That asymmetry is currently documented nowhere a reader of the table would find it.
- **IA-5 (Authenticator Management):** lifecycle of authenticators — binding, rotation, revocation on compromise. Principals have no authenticators, so there is nothing to manage; the control is not "partially met," it is structurally inapplicable until §2's binding slot exists.
- **AC-5 (Separation of Duties):** comparatively the *strong* story — `validate_review`, stamp-distinctness, INSERT-only s18 reviewer grants (SoD by privilege, not politeness). Worth saying plainly: autoharn's SoD machinery is ahead of its identity machinery, which is the reverse of most systems.
- **AU-10 (Non-Repudiation) and its enhancements** — Association of Identities; **Validate Binding of Information Producer Identity**; Chain of Custody; **Validate Binding of Information Reviewer Identity**; Digital Signatures. This family reads like a review of autoharn's exact architecture: producer-identity binding = the stamp (invocation-grade, present); reviewer-identity binding = countersigns (stamp-grade, present); chain of custody = s26/s27 row-hash chain (present); digital signatures = the GPG layer (present) — **and the principal table references none of them.** Every AU-10 mechanism exists; none is joined to the identity record. The gap is not missing machinery, it is a missing join.

### 1.2 NIST SP 800-63-3 — the conceptual model this schema needs

([SP 800-63-3](https://pages.nist.gov/800-63-3/sp800-63-3.html), [800-63B](https://pages.nist.gov/800-63-3/sp800-63b.html); 800-63-4 now exists — noted, not read in depth.) The durable lesson for this design is not any assurance level; it is 800-63's *decomposition*: identity proofing (IAL — is this record about the real-world entity it claims?), authentication (AAL — is the actor in this transaction the holder of that record?), and federation (FAL) are **independent axes with independent lifecycles**, and an authenticator is **bound to** an identity record, never conflated with it, with explicit bind/rebind/revoke events. autoharn's four-column table collapses all of this into a name. The stamp is an AAL-like mechanism with no binding to the identity layer; the db_role map is a second, parallel authenticator also unnamed as one; a GPG key would be a third. The type that makes the current confusion unrepresentable is precisely "identity record ≠ authenticator; bindings are first-class, dated, revocable" — 800-63's spine, and it happens to also be ADR-0012 P1 (one fact, one home).

### 1.3 NRC / 10 CFR 50 Appendix B — organization, qualification, independence

([eCFR Appendix B](https://www.ecfr.gov/current/title-10/chapter-I/part-50/subject-group-ECFR89aa6ca4aada73c/appendix-Appendix%20B%20to%20Part%2050), [Cornell LII](https://www.law.cornell.edu/cfr/text/10/appendix-B_to_part_50).) Criterion I requires that persons performing QA functions have "sufficient authority and organizational freedom," including **independence from cost and schedule pressures**, with authority and duties *documented*; Criterion II requires the QA program be carried out by **trained and qualified personnel** (qualification is a recorded fact, not an assumption); Criterion XVII requires QA records be attributable. Mapped: the kernel records *that* a distinct identity countersigned, but records nowhere *what that identity is qualified to countersign*, or what its organizational position is. The BRIEF's own G13 (competence record — actor, safety activity, competence basis, integrity band; backed by IEC 61508-1 §6 and ISO 26262 Pt 8 §9) is the operational form of Criterion II, is already law-adjacent in this project — and has no column, table, or event kind anywhere in the kernel to land in. This is the "domains of competence" item in the maintainer's own words, and it is not exotic: it is the single most-repeated identity requirement across the nuclear/functional-safety cluster.

One honesty point the nuclear/IV&V cluster forces (IEEE 1012's independence dimensions — technical, managerial, financial — which s17's independence vocabulary imported verbatim): in a single-human deployment, **no schema can substantiate managerial or financial independence between model principals** — every agent shares one orchestrator and one payer. The stamp can witness *technical* distinctness (different invocation) and nothing more. The current vocabulary lets a review claim `managerial`/`financial` with the same stamp-distinctness evidence that supports `technical`. The standards answer is not to fake the evidence but to scope the claim: grades the schema can witness are computed (s29 already does this correctly); grades it cannot witness should either be refused or carried as explicitly human-attested. Right now the vocabulary promises more than any mechanism in the system can check — a quiet I9-style discharge-status dishonesty at the identity layer.

### 1.4 FAA / DO-178C-adjacent

([AdaCore on DO-178C reviews](https://www.adacore.com/blog/a-fresh-take-on-do-178c-software-reviews), [ISIT independence white paper](https://www.isit.fr/documents/2114/safety_assurance_through_independent_verification_white_paper_v3.3.pdf); paraphrase-grade, primary standard not retrieved — marked accordingly.) "With independence" objectives require that the verifier is not the author **and that this separation is documented in the evidence**; the Software Verification Plan documents the reporting structure and review authority; reviewer identity appears in verification records (DO-178C §11, per the BRIEF's own high-confidence entry). 21 CFR Part 11 §11.50/.70 (via the BRIEF, high confidence) adds the signature form: bound to the specific record version, with its *meaning* (author/reviewer/approver) explicit. Mapped: autoharn documents the separation (stamps, grades) but not the *authority* — nothing says which principals are entitled to act as reviewers at all, which is the backflow Suggestion's exact complaint restated in DO-178C vocabulary.

### 1.5 Cryptographic authentication — what the standards actually require, and the two rulings reconciled

What the standards say (against the maintainer's stated uncertainty): none of the families surveyed requires cryptographic authentication of *every actor*. What they require is (a) non-repudiation for **defined actions** that carry authority — approvals, sign-offs, releases (AU-10: "organization-defined actions"; Part 11: signatures on records, bound and meaning-typed; the PE-stamp tradition: the liability-transfer event); and (b) that where signatures exist, the **signer's identity and key are managed artifacts** — the record system knows which key speaks for which identity, from when to when. That maps exactly onto the design `MAINT-GPG-TRUST-LAYER.md` already ratified: sign the authority-carrying moments (ratifications, commissions, chain heads), refuse agent keys (§6 — an agent key on the host proves nothing the stamp doesn't; this is correct and standards-consistent), and — §5, filed-not-built — "fingerprints committed per principal" for the multi-human case. The defect the commission names is real and precise: **the trust layer's own design already assumes a principal↔key binding, and the principal table has no slot for it.** The signature exists; the record system cannot say whose it is.

Reconciling the two rulings, on the maintainer's own words: the standing deferral ("key generation/signing deferred until all else banked; never re-raise as recommendation") governs the *operational ceremony* — generating keys, adopting signing rituals, hardware tokens. The newer injection names cryptographic authentication a first-class concern *for the principal-surface design*. These do not conflict, and the reconciliation is clean: **modeling the binding is in scope now; performing the ceremony remains deferred.** A `principal ↔ key-fingerprint` binding (with bind/revoke dates) is a strictly additive schema fact that can sit empty indefinitely — exactly the STANDARDS-REGISTRY posture ("an entry may sit unoperationalized; what it cannot be is silently absent"). Designing the slot re-raises nothing; it makes the already-ratified GPG design's §5 representable instead of prose. I flag one residual tension honestly: the deferral said "never re-raise" — but the maintainer himself re-raised it in this commission, and the newer maintainer word governs. The spec that follows should carry this reconciliation in its own text so the next auditor doesn't re-litigate it.

**Section 1 confidence:** high on control-family intent and the mapping; medium on anything phrased as clause text (nothing above is a verbatim quotation; primary standards were not retrieved in full — DO-178C, IEEE 1012, and 800-63-4 especially are paraphrase-grade). UNVERIFIED: whether NRC regulatory practice anywhere requires cryptographic (as opposed to attributable-and-immutable) records — I found no such requirement, but absence-of-finding is not a finding of absence.

## 2. The design space: what a principal must BE in this system

ADR-0000 Rule 2(a), applied to identity itself. The defect class, named generally: **an authority-carrying identity whose authority, scope, and validity period are not representable, so every misuse of it is representable.** The candidate answers, with trade-offs stated:

### 2.1 The root move: identity events, not identity columns

The maintainer's instinct in row 1691 ("a boolean `revoked` column") is the instance-level fix. The class-level fix follows from §0's structural observation: the principal table is the kernel's one non-ALCOA surface. The type that forecloses the class is **event-sourced identity**: `principal` stays as the identity *anchor* (id + name, immutable), and everything else — registration, class assignment, role/purpose binding, competence grant, key binding, suspension, revocation, succession — becomes an append-only, actor-attributed, stamp-carrying event stream, with current state a *derived view* the triggers consult. This is not imported architecture; it is the kernel's own idiom (supersession s31, `effective_state`, composite discharge s33) applied to the one table that predates the discipline. Under it, "who registered this principal, when, on whose approval" stops being unanswerable, and a lifecycle transition is a row in the same tamper-evident chain as everything else.

Trade-off, honestly: this is a real delta — new table(s), a view, trigger extensions — and only partly class-ratifiable. The pragmatic ladder: **rung 1** (fail-safe, class-ratified shape: adds refusals only) — `revoked_at`/`revoked_by` on `principal` plus a write-path refusal in `set_actor`/`resolve_actor`, and `register-principal` refusing duplicates loudly (backflow asks #1 and #2, verbatim); **rung 2** — the event stream and derived state, superseding rung 1's columns into the view. Shipping rung 1 alone and stopping would be the ADR-0000 contrast-specimen patch; shipping it as the witnessed first rung of a filed rung 2 is legitimate (ADR-0013 Rule 4: filed, not buried).

### 2.2 Granularity: four facts, four homes (the 800-63 decomposition)

A principal today is one name carrying four conflated facts. The type-driven separation:

1. **Identity** — the durable anchor. Immutable name, immutable id. (Exists.)
2. **Role/purpose** — what job this name signs for. Today this lives, at best, in the prose of a panel-deployment decision row. Should be a typed, dated binding (role vocabulary can start closed and small: e.g. author / item-reviewer / whole-diff-reviewer / commissioner / orchestrator / scout — the panel's own purpose-named slate shows the natural grain). This is what lets the kernel eventually answer "was this countersign performed by an identity *entitled* to countersign this kind of act" — the review_gap machinery's missing premise, and the backflow Suggestion's fix.
3. **Competence/qualification** — the G13 record: integrity band, basis, granted-by, dates. For today's single-human deployment this can be a **named-not-built** slot (the registry posture); the point is that when the BRIEF's G13 obligation fires, there is a home. `agent_class` is a degenerate two-bit competence taxonomy already; it stays, but stops pretending to be sufficient.
4. **Authenticator bindings** — the joins to the three mechanisms that already exist: db_role (present, in `principal_role`), stamp identity (present per-invocation, deliberately *not* durable — keep it that way), and key fingerprint (absent; human principals only, per GPG spec §6). Each binding dated, each revocable.

Over-typing guard (ADR-0000's own "not type everything"): resist minting per-deployment qualification bureaucracy the single-human instance can't exercise. The discriminator is whether a class is foreclosed: role-binding forecloses a witnessed class (the reviewer2 ambiguity); competence machinery forecloses a class only once acts are graded against bands — so it earns a slot, not yet machinery.

### 2.3 Cross-principal relationships

`acts_for` is the wrong type: single-valued, undated, kind-less, and unused in 1800+ rows of live history. The kernel already owns the right pattern — s30's *typed dependency edges* for work items. The same move for principals: a typed, dated, attributed edge table with a small closed vocabulary of relations the record actually needs: `acts-for` (delegation), `dispatched-by` (supervision — today only inferable from stamp pairs), `same-natural-person` (the bork/maintainer/commissioner triple, currently invisible to any auditor), `succeeds` (reviewer2 taking over a role from reviewer — the panel's lived need), `holds-key` (if not modeled as a binding in 2.2.4). Each of these is a fact an auditor of the panel ledger needed *this week* and could not query. Retire `acts_for` by supersession, not deletion (lineage discipline).

This is also where row 1343's open taxonomy question (118 precedents) becomes answerable rather than case-by-case: the question "does a name minted for self-review disclosure taint later independent reuse" dissolves once identity (the name) is separated from role-binding (what it currently signs as) and the grade stays denominated in stamps (s29 already does the right thing — grades are computed from invocation pairs, never names). The honest general rule the schema should make expressible: **names carry discoverability; stamps carry independence; role-bindings carry entitlement.** Three facts, three homes; the 1343 confusion exists because all three currently ride one name.

### 2.4 Registration ceremony

Formalize AC-2's approval-before-creation at the write path, not in prose: a registration event that carries purpose text and an approver reference, with `register-principal` refusing (or grading LAZY, mirroring s25's commission-mode honesty) a bare registration. See §3 for what to take from the panel's process. Silent-duplicate no-op becomes a loud refusal in any variant — that one is unambiguous and cheap.

### 2.5 What stays deliberately un-typed

The stamp remains a tripwire, per its own header — nothing here should promote it to authentication. Agent keys stay refused (GPG §6). Managerial/financial independence stays unclaimable-by-computation (1.3); if the vocabulary keeps those values, the spec should require them to be human-attested or refuse them in single-human deployments.

**Section 2 confidence:** high on the diagnosis and the event-sourced direction (it is the kernel's own established idiom); medium on the specific role vocabulary and the rung split — those are design judgments the spec author and maintainer should own, and per the commission I have not written the spec.

## 3. The downstream deployment's invented practice, assessed

The role-name-governance procedure (row 1729, with the full commission at 1691 and the executed slate at 1736–1754) is, read against AC-2, a remarkably faithful reinvention: proposal with stated purpose → fresh live-roster walk with per-existing-principal distinctness argument → independent countersign → registration gated on the attestation → refusal ends the proposal. Four purpose-named principals were minted under it, each with a recorded job description and sign-off.

**What it gets right, worth formalizing:** approval precedes creation (AC-2's core, and the panel arrived at it from first principles — strong evidence of load-bearing-ness); the roster walk (an anti-duplication check the schema's silent no-op actively undermines — the procedure compensates for the kernel defect); purpose stated concretely enough that a later duplicate proposal is detectable; the conflict-of-interest carve-out (item-countersign may not countersign proposals about itself — a genuinely sophisticated touch, SoD applied reflexively); ledger-native record.

**What it gets wrong, or cannot get right procedurally:** (a) the kernel cannot distinguish a governed registration from a bare one — `register-principal` still works with no proposal row, so the procedure's guarantee is exactly the "verbal policy" the backflow indicts elsewhere; ADR-0011 Rule 2 says this recurrence-prone prose converts to a mechanism, and 2.4 is that mechanism. (b) The role/purpose it so carefully records lands in free prose of a decision row — unqueryable, un-joinable to the acts later signed under the name; 2.2.2 is the typed home. (c) STEP 4's ledger-only filing makes it a house rule invisible to every other deployment — right call for the panel, but the harness-level formalization belongs upstream (the spec this consult feeds). (d) It governs birth only — no revocation, succession, or same-person half, which the same session then needed (retiring `author`) and could not enforce. (e) The default sign-off reviewer is itself an identity with no bound entitlement — the procedure bootstraps on the very gap it works around. None of these are criticisms of the deployment; they are the boundary of what procedure can do without schema. As a candidate to formalize: adopt its *shape* (proposal/walk/countersign/gate), move its *enforcement* to the write path, and type its *purpose* field.

**Section 3 confidence:** high — this is direct-evidence assessment of artifacts read in full.

## 4. Closure-statement candidate (ADR-0000 Rule 2(a), 2026-07-02 form)

**Invariant (candidate, most general form):** every act the kernel records is attributable to an identity whose registration, current standing (active/suspended/revoked), role entitlement, and authenticator bindings are themselves recorded, attributed, append-only facts; no act is acceptable under an identity whose standing does not permit it at the act's time; and no identity fact is mutable or unattributed.

**Quantification universe — enumerated explicitly, so the later spec can be checked against it:**

*Axis A — every place the kernel asks "who" (the sibling surfaces):*
1. `ledger.actor` + `set_actor` connection-default; 2. `principal_role` (db_role axis); 3. the stamp pair (`stamp_session`,`stamp_agent`) + s23 per-invocation token; 4. `validate_review` same-actor refusal; 5. `validate_independence` + the s17 independence vocabulary; 6. s29 `discharge_grade` computation (and s34's writer-assertion refusal on it); 7. `obligation.assigned_by`/`obliges_actor`; 8. `countersign_obligation`/`review_gap` (s15/s32); 9. s25 commission signing modes (FULL's actor+absent-stamp presumption); 10. s18 criterion-reviewer roles (privilege-enforced identity); 11. work-item claim/close/depends actor attribution; 12. the CLI: `resolve_actor`'s four wired paths and the three unwired (`work depends`/`close`/`resolve-violation`) — the unwired three are in-universe and named not-covered by any schema fix (they are CLI debt, tracked upstream as write-path-actor-completion); 13. `register-principal` itself (the registrar's identity — currently unrecorded); 14. the GPG surfaces: ratification tags, commission signatures, chain-head signatures, the §5 session sign-off; 15. `nla-schema`'s deliberate FK-less actor (a named sibling that *opts out* — its exemption must be re-argued, not inherited silently); 16. the Idris model (`PrincipalId = Nat`, `Entry.actor`, `attests`' closer) — any principal enrichment owes an s-parity pass; 17. the SPA/read surfaces that render actor names (misattribution renders there; display-only, but in-universe for the audit); 18. every deployment's kernel instance (the fix travels by birth chain only — runs-are-linear; existing worlds' misattributed history is dust and stays dust, named as not-covered).

*Axis B — every lifecycle event an identity can undergo:* registration (governed / bare); duplicate-registration attempt; class assignment and any later class change (the reviewer/reviewer2 model-vs-subagent drift is the witnessed specimen); role/purpose bind, rebind, release; competence grant/withdrawal; authenticator bind/rotate/revoke per authenticator kind (db_role, key; stamps excluded by design); suspension; revocation; succession; merge of two names discovered to be one entity; rename — the spec must rule rename in or out explicitly (I recommend out: names immutable, succession instead); and, for every event above, the registrar/approver identity and time. An event kind deliberately unsupported is named as unsupported.

*Axis C — every relationship between identities:* delegation (acts-for); dispatch/supervision (orchestrator↔subagent, today stamp-inferable only); same-natural-person; role succession; review/countersign edges (`regards` — exists); obligation edges (exist); key custody (human↔fingerprint); and cross-deployment identity (the same "maintainer" across worlds) — **named as not covered** by this consult (see §5).

*Denomination check:* independence remains denominated in stamp pairs (the resource that actually witnesses distinctness), never names; entitlement in role-bindings; standing in dated lifecycle events; non-repudiation in key bindings — never a name standing proxy for any of the four.

**Presumption of narrowness, checked outward:** the sibling I most suspect this enumeration still misses is the *hooks/gate layer* (delegation observers, doc-attestation B-agents) which reasons about "who" outside the kernel entirely — flagged for the spec author to sweep; I did not audit hooks/ for identity assumptions.

**Section 4 confidence:** high on axes A and B (derived from grep-verified surfaces and witnessed events); medium on C's completeness — relationship kinds are the least mechanically enumerable axis, which is why the presumption-of-narrowness note is there.
