# Report 04 — chocofarm perf-investigation discipline (ADR-0009) + execution stamina (ADR-0013) + executor second-opinion (ADR-0014), and the throughput_research provenance DB that already mechanizes part of ADR-0009

## Summary

ADR-0009 mandates that any perf or equivalence claim is honest only when its substantiation (a captured, reproducible before/after) is attached, with a distinctive two-tier equivalence bar: bit-exact for logic invariants (== 0.0) and aggregate-statistical-within-MC-CI for float-sensitive numerics. The ADR is explicit that this is policy, not mechanism (Consequences/Negative: "No automated check verifies that a write-up's perf claim is backed by a referenced bench"; Revisit #4: a claim-token scanner is the mechanization trigger) — this is the central harness gap and exactly where a logic-gate plugs in. Crucially, ADR-0009's 2026-06-24 amendment already built half of the provenance ledger the harness wants: throughput-lab/harness/exp_db.py is a Postgres store (throughput_research) where every benchmark reading carries a code stamp (git_commit + git_tree clean/DIRTY), host, exact command, and tool, NOT-NULL by construction so an un-pinnable reading is unattributable; and it splits the immutable MEASUREMENT (tlab_reading) from the append-only, supersedable INTERPRETATION (tlab_finding) — plus a pre-registration layer (tlab_prereg / tlab_prereg_conclusion) that registers a typed decision criterion BEFORE data exists. ADR-0013 (execution integrity) and ADR-0014 (second opinion) are ADR-0011 instantiated to execution attrition and reasoning-anchoring respectively; both honestly declare themselves almost entirely review-only/self-applied, naming exactly the recurrence triggers that would convert them to mechanisms. ADR-0014's structured-second-opinion (trigger: >=2 mis-targeted attempts on the same symptom) is a genuine candidate harness gate — its named-but-unbuilt mechanism is a session-trace heuristic flagging N consecutive edits/commits citing the same diagnosis against the same unresolved symptom, which the provenance ledger's session-attribution is precisely positioned to feed.

## Key facts

- ADR-0009 mandated benchmark metadata, per the live tlab_reading schema (exp_db.py:191-230), is: git_commit (NOT NULL), git_tree (NOT NULL, CHECK in {'clean','DIRTY'}), recorded_at, host (NOT NULL), the exact command string (NOT NULL), the emitting tool (NOT NULL), an optional cohort tag, measurement-condition notes, replicate_idx, a config_id FK into tlab_config, and the metrics (wall_s + raw counts decisions/leaves/forwards ALONGSIDE derived rates leaf_rows_s/dps/server_util_pct/forwards_s/lpd so the row is self-checking), with metrics_extra jsonb for the long tail. A reading with NO metric at all is rejected by record_reading (ADR-0002 fail-loud).
- ADR-0009's per-component speedup substantiation is bench_hotpath.py: each hot-path component (env.marginals, features.build, the forward) timed in isolation on representative captured states (states.npz, beliefs from 15,504 worlds down to 1) so a speedup is validated one component at a time before/after, never an artifact of episode-level noise (ADR-0009 lines 38-43, 124-125).
- The two-tier equivalence bar (ADR-0009 lines 47-67, 169-185): (1) bit/exact for logic invariants e.g. 'exactly zero mass on illegal action slots' asserted == 0.0, NEVER relaxed to a tolerance; (2) aggregate behavioral for float32+numba numerics: fixed-lambda0 rate, mean E[T], action histogram must be statistically indistinguishable from the float64 baseline over N>=300 episodes across >=2 seeds within Monte-Carlo CI, with MC standard error reported so 'indistinguishable' is a number; (3) a tighter forward bar ABS_TOL=1e-4 in tests/test_jax_equivalence.py. Confusing the two kinds is the named failure.
- ADR-0009 acceptance criteria (lines 151-167): speedup claims attach before/after under the SAME reproducible scenario (same belief corpus, same net, same seeds); equivalence claims attach BOTH the == 0.0 exact invariant AND CI-overlapping behavioral metrics (or the forward ABS_TOL test); hot-path refactor PRs attach the pre-refactor baseline. Absence of substantiation does NOT block landing, but the write-up must state the absence explicitly ('defensively sound but not substantiated by a bench pair; speculative win is X, cost is Y').
- ADR-0009 is explicit that the discipline is currently UNMECHANIZED: Consequences/Negative (lines 207-210) 'No automated check verifies that a write-up's perf claim is backed by a referenced bench (ADR-0011 Rule 1: a declared review-only surface; a claim-token scanner would be the mechanization trigger)'. Revisit #4 (lines 253-254) names the same scanner as the conversion trigger.
- ADR-0009's 2026-06-24 amendment (lines 277-305) records that the substantiation requirement is NOW PARTIALLY MECHANIZED via throughput-lab/harness/exp_db.py (the throughput_research Postgres store): every reading persisted with code stamp (commit/tree), HP config, exact command, and replicates so a perf claim is code-addressable and aggregated (median/IQR), never a single eyeballed number. It distinguishes MEASUREMENT (tlab_reading, immutable fact) from INTERPRETATION (tlab_finding, an authored supersedable 'finding' e.g. 'X is faster', 'the +31% was an artifact').
- exp_db.py separates measurement from belief at the schema level: tlab_finding (exp_db.py:240-266) is APPEND-ONLY, carries status in CLOSED vocabulary {provisional, confirmed, retracted} (ADR-0008), the git_commit the belief was formed against (so you can travel back to what was believed at a commit), a scope (typically a tlab_reading.tag cohort), a 'supersedes' self-FK (a corrected belief is a NEW row; prior is NEVER mutated; 'is X current?' == 'no finding supersedes X'), and an optional motivated_commit linking interpretation -> code change (closes the loop).
- exp_db.py also has a PRE-REGISTRATION layer: tlab_prereg (lines 277+) registers a typed `criterion` (built/validated by a Criterion dataclass — a true partition, no gaps/no overlaps, not free prose) and the code stamp BEFORE any data exists, immutable; tlab_prereg_conclusion is a SEPARATE later verdict, UNIQUE per prereg_id so an experiment concludes AT MOST ONCE — making 'decisive' a falsifiable, code-stamped-earlier-than-result claim rather than a retro-fitted narrative.
- Provenance composes via code_stamp.code_stamp() (throughput-lab/harness/code_stamp.py, ADR-0012 P1 'one home, never re-author'): record_reading auto-stamps {commit, tree} unless the caller passes an explicit stamp; a DIRTY tree marks a NON-reproducible reading and is recorded as-is, never silently 'clean' (exp_db.py:75-77, 198-201).
- ADR-0013 (execution integrity) Rule 5 'Verify the artifact, not the claim' is named as THE ONE genuinely mechanized rule (test/CI gate + run-time invariant); it generalizes existing machinery — ADR-0002 fail-loud, ADR-0012 P8 mypy --strict, ADR-0009's equivalence/parity tests verify numbers against a measured baseline rather than trusting an 'equivalent' claim. Rules 1-4 are declared review-only; Rule 3 ('lower-ROI/invasive demurral is a tell') is the weakest, 'enforced by the faculty it most reliably corrupts'.
- ADR-0013 names its own ADR-0011 Rule-2 conversion triggers (lines 426-431): a structured commission/result-conformance diff (a checklist the result is mechanically diffed against — the move-by-move scorecard in the leaf-eval audit's 01-plan-vs-result.md is the hand-run prototype) for Rule 1; a staged-content verifier that fails a content-empty 'hollow' commit for Rule 5; an output-not-exit-code wrapper for the shell-misfire class.
- ADR-0014 (executor second opinion) is Provisional/emergency, ADR-0011 turned inward on the executor's own reasoning. Its mechanism is the structured second-opinion request, triggered (Rule 2) on an OBSERVABLE recurrence, not a feeling: any one of (>=2 attempts that each addressed the wrong target; a diagnosis that keeps proving partial; growing thrash). Enforcement is review-only/self-applied throughout, honestly 'enforced by the faculty most prone to the lock it guards against'.
- ADR-0014's named-but-unbuilt mechanization candidate (Rule 2 / Revisit #2, lines 217-219, 370-374): 'a session-trace heuristic that flags N consecutive commits/edits citing the SAME diagnosis on the SAME unresolved symptom' — the observable recurrence made mechanical. This is the closest thing the tenet has to a gate, currently 'conceivable and unbuilt'.
- ADR-0014 Rule 3 requires the second agent be briefed for INDEPENDENCE: give the problem + evidence but NOT the diagnosis/frame/suspected-call list; a leading brief collapses an independent perspective into 'one x M' (your single frame run twice). Rule 4 keeps the executor owning the result (a second opinion is a hypothesis to test against evidence, per ADR-0013 Rule 5, not a verdict to rubber-stamp). Both governed by ADR-0005 Rule 9: commission prompt + report recorded verbatim; a verdict whose artifact cannot be produced is no verdict.

## Gaps (where the haphazardness lives)

- ADR-0009 perf/equivalence claims in PROSE write-ups (docs/results/az-perf.md, az-jax-perf.md) are NOT mechanically tied to a bench run or git hash — the ledger only covers the throughput-lab register (tlab_reading), not the AlphaZero/numba/JAX bench register. The bench harnesses (bench_hotpath etc.) emit numbers but there is no record_reading equivalent persisting them with a code stamp.
- No claim-token scanner exists: nothing verifies a write-up asserting 'faster'/'regression'/'equivalent'/'matches the baseline' references a bench or a tlab_reading. ADR-0009 names this explicitly as the unmechanized residue (Negative + Revisit #4). This is the primary ADR-0009 logic-gate gap.
- The bug->session link is absent: tlab_finding.motivated_commit links interpretation->commit, but nothing links a git_commit (or a bench regression) back to the Claude Code SESSION that produced it. There is no session_id column anywhere in the ledger.
- ADR-0013 is 4/5 review-only and ADR-0014 is entirely review-only/self-applied — the attrition trigger (a commission/result-conformance diff, a hollow-commit content verifier) and the second-opinion trigger (a session-trace heuristic flagging N consecutive edits citing the same diagnosis against the same unresolved symptom) are BOTH named-but-unbuilt. These are the explicit recurrence->mechanism conversions the maintainer's harness can mint.
- The two-tier equivalence bar (== 0.0 logic-exact vs MC-CI behavioral) is a human judgment about which bar a quantity's KIND demands; nothing prevents a contributor from pinning a float-sensitive rate bit-exactly (forbids a legit optimization) or relaxing a logic invariant to a tolerance (admits a bug). A logic program over quantity-kind could gate this.
- Pre-registration (tlab_prereg) exists for the throughput register only; the AlphaZero equivalence register (where N>=300/>=2-seed is the substantiation floor) has no pre-registered criterion, so 'indistinguishable within MC CI' is asserted per-write-up rather than pre-committed.

## Harness hooks (where a registry / ledger / logic-gate plugs in)

- Generalize record_reading/tlab_reading beyond throughput-lab into the capability/provenance ledger so EVERY bench (bench_hotpath, bench_equivalence, test_jax_equivalence) writes a code-stamped reading — this is the existing schema the harness should adopt project-wide rather than reinvent.
- Add a session_id (Claude Code session) column to tlab_reading and tlab_finding (and a sessions table) so a benchmark result and an interpretation are attributable to the producing session — closes the maintainer's non-attributable-git-hash-vs-perf and bug->session gaps directly atop the existing provenance stamp.
- Build the ADR-0009 claim-token scanner as a logic-gate: scan perf write-ups for closed-vocabulary tokens ('faster'/'regression'/'equivalent'/'matches the baseline'); require each be backed by a referenced bench output or a tlab_reading.reading_id, else flagged as the loudly-marked-unsubstantiated case. This is the ADR-0009 Revisit #4 / Negative trigger named verbatim.
- Encode the two-tier equivalence bar as a logic rule over quantity-kind: kind=logic_invariant => assertion must be == 0.0 (Z3/classical: tolerance on a logic invariant is unsatisfiable); kind=numeric => assertion must be statistical-within-MC-CI with reported standard error and N>=300/>=2-seeds. A category error in either direction is a gate failure.
- Implement ADR-0014's named session-trace heuristic over the ledger: flag N consecutive edits/commits citing the SAME diagnosis against the SAME unresolved symptom -> emit a structured second-opinion trigger. The tlab_finding.scope + status + supersedes chain plus a session_id is exactly the substrate to detect 'a diagnosis that keeps proving partial'.
- Implement ADR-0013 Rule 1/Rule 5 mechanisms: a structured commission/result-conformance diff (checklist the result is mechanically diffed against — generalize the leaf-eval 01-plan-vs-result.md scorecard) and a staged-content verifier that fails a content-empty hollow commit; both are explicitly listed as the recurrence-conversion candidates.
- Surface the bench/perf tool surface in the CAPABILITY REGISTRY (bench_hotpath, bench_equivalence, bench_value_target, test_jax_equivalence, capture_states->states.npz, throughput-lab harness incl. CP-SAT topology_enum) plus the Criterion dataclass partition-validator — so agents reach for the provable bench/OR tools instead of eyeballing.
- Make the second-opinion brief an inspectable artifact gated by ADR-0014 Rule 3 + ADR-0005 Rule 9: persist the commission prompt and report verbatim, and lint a leading brief (one that names the conclusion it wants) as forfeiting independence.
