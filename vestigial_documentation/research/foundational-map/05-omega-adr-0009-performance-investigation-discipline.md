# Report 05 — omega/ADR-0009 Performance Investigation Discipline — perf-number provenance/attribution (harness Pillar 2)

## Summary

Omega's ADR-0009 is a rigorous *authoring* discipline: a perf claim (improvement/regression/null) is honest only when a captured before/after profile under a reproducible scenario is attached, else the claim must carry an explicit "[unsubstantiated]" qualifier. But the ADR itself concedes (Negative consequences, Revisit #4) that it is "policy, not mechanism" — no automated check links a perf claim to its evidence. The actual capture path proves this: `frontend/scripts/perf-capture.mjs` writes a Chrome trace named ONLY `<scenario>-<ISO-timestamp>.json` into a user-local, git-ignored dir, embedding no git hash, no scenario config (model/visits/adaptive/boards), no environment (headed/headless/connect/host), and no Claude Code session id — even though those config values exist in-process at capture time. The link from a perf number to a commit/scenario/env/session is therefore entirely manual prose (worklog path+timestamp+size references, an out-of-tree `perf-baseline` git worktree, and inline numbers like "17.1s->8.3s" baked into commit messages with no profile citation). Chocofarm's sibling ADR-0009 has already MECHANIZED exactly this gap (2026-06-24 amendment): `throughput-lab/harness/exp_db.py` persists every reading into a `throughput_research` Postgres store stamped with commit/tree, HP config, exact command, and replicates, and separates immutable measurement (`tlab_reading`) from authored, supersedable interpretation (`tlab_finding`) at the schema level — the ready-made blueprint for omega's provenance ledger.

## Key facts

- omega perf-capture.mjs builds the output filename as `outPath = join(outDir, `${scenario}-${stamp}.json`)` where stamp is just an ISO timestamp — frontend/scripts/perf-capture.mjs line ~115. The trace filename encodes ONLY scenario + timestamp.
- The capture config (model, visits, adaptive, boards, popoverTarget, proxyUrl, sgf) is assembled into `cfg` and passed to `window.__perfScenario.run()` (perf-capture.mjs lines ~74-90) but is NEVER written into the filename or any sidecar metadata — the parameters that define the scenario evaporate at capture time.
- The faithfulness/environment mode (default headless / --headed / --connect to a real desktop Chromium) materially changes felt latency (script header lines ~30-44) yet is not recorded with the trace either.
- No omega perf script invokes git or child_process: `grep -rniE 'rev-parse|git rev|child_process|execSync' frontend/scripts/perf-*.mjs` returns nothing. There is zero machine-readable link from a profile to the code state it measured.
- Traces land in `~/w/vdc/chromium_profiles/` (overriding ADR-0009's `~/perf-profiles/` default for space; ADR §Profile-share convention, lines 299-305); binaries are deliberately kept OUT of the repo, referenced by hand as `path + timestamp + gzipped size` from worklogs/PRs.
- Confirmed listing of `~/w/vdc/chromium_profiles/`: files like `close-at-scale-2026-06-22T15-29-35-262Z.json` — many same-scenario captures distinguishable only by timestamp, with no in-band way to know which commit/model/visits/env produced each.
- ADR-0009 explicitly names the mechanism gap in §Consequences/Negative (lines 440-443): 'Discipline is policy, not mechanism... No automated check verifies that a worklog's perf claim is backed by a referenced profile.' Revisit #4 (lines 524-529) flags a future linter/CI gate scanning worklog text for perf-claim tokens + adjacent profile markers as 'Not feasible today.'
- The before/after baseline is a manually-managed out-of-tree git worktree: `git worktree list` shows `/home/bork/w/omega/.claude/worktrees/perf-baseline` on branch `bork/perf/jank-extended-baseline` (commit 6e2444a3) — the 'baseline commit' is a human-curated worktree, not an attributable field on a reading.
- Perf commit messages embed the numbers inline but cite no profile: commit b3bfe8c1 body asserts 'close long-task 17.1s -> 8.3s (-51% ... original baseline was 43s)' with no profile path, no baseline commit hash, no scenario config reference.
- Capture commit d52769f7 explicitly says 'makes no perf claim (ADR-0009)' and files work-status items by name (fix-boardtab-vmemo-index-key, perf-tabstrip-virtualization, etc.) — showing the work-status layer is where perf intentions live, but it is textual, not joined to the trace artifacts.
- chocofarm ADR-0009 (2026-06-24 amendment, lines ~293-314) MECHANIZES the same gap: `throughput-lab/harness/exp_db.py` (a `throughput_research` Postgres store) 'persists every reading with its code stamp (commit/tree), HP config, exact command, and replicates — so a perf claim is code-addressable and aggregated (median/IQR), never a single eyeballed number'.
- chocofarm separates measurement from interpretation at the SCHEMA level: `tlab_reading` is an immutable fact; `tlab_finding` is an authored, supersedable claim about it — 'so an overturned claim is auditable, not lost' (chocofarm ADR-0009 lines ~308-314). Omega has no such measured-vs-interpreted separation.
- chocofarm's perf register is shared-tenet (Provenance line 14: 'Transferred from the LengYue ADR corpus') — both projects carry the identical tenet shape, but only chocofarm has executed the mechanization that ADR-0011 demands; omega is the lagging twin.

## Gaps (where the haphazardness lives)

- Trace filename is `<scenario>-<timestamp>.json` only — the single biggest non-attribution: a captured omega profile cannot be joined to a git commit, a scenario config (model/visits/adaptive/boards), an environment (headless/headed/connect/host/CPU), or a Claude Code session by any machine-readable means. The timestamp is the only key.
- Scenario config exists in-process (cfg object, perf-capture.mjs lines 74-90) at the exact moment of capture but is thrown away — a near-free metadata write is simply not done.
- No measurement-vs-interpretation separation: omega has no equivalent of chocofarm's tlab_reading/tlab_finding split. A perf claim ('-51%', 'no regression') and the raw trace it rests on are not linked, so an overturned interpretation is lost rather than auditable.
- Perf numbers live as free prose in commit bodies (b3bfe8c1: '17.1s -> 8.3s') and worklogs with no profile-path citation and no baseline-commit hash — exactly the manual, non-attributable git-hash-vs-perf the harness exists to kill.
- ADR-0009's non-blocking '[unsubstantiated]' qualifier is policy-only: nothing detects a perf-claim token in a worklog/PR/commit and verifies an adjacent profile reference (ADR explicitly flags this as Revisit #4, not feasible today).
- Environment faithfulness (headless vs headed vs real-desktop --connect) materially changes felt latency per the script's own header, yet is uncaptured — two traces of the same scenario may be incomparable for an undocumented reason.
- No bug→session link: when a perf regression (or a measurement-confounding bug) is later found, there is no record of which Claude Code session and which commit produced the capture or the change.
- Baseline identity is encoded in a git worktree's checked-out branch, not as data — fragile and not queryable alongside the readings.

## Harness hooks (where a registry / ledger / logic-gate plugs in)

- At the write site in perf-capture.mjs (just before/after `writeFile(outPath, data)`, lines ~196-200): emit a sidecar `<trace>.meta.json` (or insert a ledger row) capturing git rev-parse HEAD + dirty flag, the full cfg object (scenario/model/visits/adaptive/boards/sgf), the env mode (headless/headed/connect URL), hostname/CPU/OS, trace size, and the Claude Code session id — turning every capture into an attributable reading. This is Pillar 2's primary insertion point.
- Encode the same fields into the filename or a content-addressed id so `~/w/vdc/chromium_profiles/` stops being timestamp-keyed: e.g. `<scenario>__<shorthash>__<model>-v<visits>__<env>__<stamp>.json`.
- Port chocofarm's exp_db.py schema to an omega `perf_research` (or reuse omega's existing Postgres anti-corruption layer): a `reading` table (immutable: commit/tree, scenario cfg, env, metric values from perf-trace-parse, replicate set) and a `finding` table (authored, supersedable perf claim referencing reading ids) — the measured-vs-interpreted split made structural.
- Have perf-trace-parse.mjs write its computed metric vocabulary (render/patch counts+ratio, LongTask count/cumulative, frame p50/p99) straight into the reading row keyed by the trace's content id, so a claim is code-addressable and aggregated (median/IQR) rather than a single eyeballed number.
- A logic-gate / linter (ADR-0009 Revisit #4) that scans a commit/PR/worklog for perf-claim tokens ('faster','-NN%','regress','no change') and FAILS unless either a ledger reading id is referenced or an explicit '[unsubstantiated]' qualifier is present — mechanizing ADR-0009's honesty rule as the ADR-0011 safety net.
- Register the perf scenario set (window.__perfScenario.list()) and the capture tools (perf-capture/-trace-parse/-heap) in the Capability Registry (Pillar 1) so an agent knows the reproducible-scenario harness exists rather than re-deriving ad-hoc jq investigations — the exact re-derivation ADR-0009 §Context names.
- Encode the normalization protocol (cold cache, scenario-proxy comparability asserted before cost comparison; counts strict vs latency best-effort) as a check that runs before two readings are allowed to be compared as before/after.
