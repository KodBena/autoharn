# Sweep: Aviation/aerospace software assurance — DO-178C (life-cycle data), DO-333 (formal methods supplement), DO-330 (tool qualification), ARP4754A (system-level development assurance)

## Standards cited

### RTCA/EUROCAE DO-178C — Software Considerations in Airborne Systems and Equipment Certification
- **Clause/locus:** §11 (life-cycle data list, e.g. 11.1 Plan for Software Aspects of Certification, 11.9 Software Requirements/Design/Code Standards, 11.14 SW Verification Cases & Procedures, 11.15 SW Verification Results, 11.16 SW Configuration Index, 11.17 SW Configuration Mgmt Records, 11.18 SW Quality Assurance Records, 11.19/11.20 Problem Reports); Table A-1 through A-10 objectives
- **Requires recorded:** Bidirectional requirements-to-design-to-code-to-test traceability data (per DAL A-D structural coverage obligations, incl. MC/DC at Level A); every review/analysis/test procedure AND its result, with pass/fail and reviewer identity; every Problem Report from anomaly discovery through disposition/closure with rationale; Software Configuration Index enumerating exact baselined configuration items and their versions; Configuration Management records documenting every approved change with rationale and impact assessment; SW Accomplishment Summary tying the whole life-cycle data set to the certification basis; SQA records showing independent conformity checks against the approved plans/standards.
- **Confidence:** high

### RTCA/EUROCAE DO-333 — Formal Methods Supplement to DO-178C and DO-278A
- **Clause/locus:** Sections on Software Verification Process objectives modified for formal analysis (e.g. FM-specific objectives replacing/augmenting Table A-5/A-6/A-7 test-based objectives); Annexes on soundness argument and formal analysis cases
- **Requires recorded:** The formal method(s) used and justification of its soundness for the specific verification objective claimed (a documented soundness argument, since DO-333 lets formal proof substitute for testing-based coverage only where soundness is established); the formal model/specification itself plus the traceability between formal requirements statements and the natural-language requirements they formalize; every formal analysis case and procedure (the property proved, assumptions/preconditions, and scope/limitations of the proof, e.g. abstractions made); the analysis RESULT record — proof completion status, any unproved obligations or discharged-by-review items, and justification for incompleteness; verification of the formal analysis tool itself is either qualified per DO-330 or the analysis is independently reviewed — this decision and its basis must be recorded.
- **Confidence:** high

### RTCA/EUROCAE DO-330 — Software Tool Qualification Considerations
- **Clause/locus:** Tool Qualification Level (TQL-1 through TQL-5) determination per Tool Qualification Criteria (TQC-1/2/3) crossed with software DAL; Tool Qualification Plan / Tool Operational Requirements / Tool Verification Cases and Results / Tool Configuration Index
- **Requires recorded:** For any tool whose output is trusted without independent verification (this includes formal-methods proof engines/model checkers/theorem provers used to satisfy a DO-333 objective) — the assigned TQL and the TQC rationale; Tool Operational Requirements defining exactly what the tool is qualified to do; Tool Qualification Plan, Verification Cases/Procedures/Results for the tool itself; Tool Configuration Index pinning the exact qualified tool version/build used to produce evidence in the aircraft-level certification data.
- **Confidence:** high

### SAE ARP4754A / EUROCAE ED-79A — Guidelines for Development of Civil Aircraft and Systems
- **Clause/locus:** §5 Requirements Determination/Validation, §6 Assignment of FDAL/IDAL, System Safety Assessment records per ARP4761 interface (FHA, PSSA, SSA), end-to-end requirements traceability
- **Requires recorded:** Functional Hazard Assessment results and the Failure Condition classifications derived from them; the assigned Functional Development Assurance Level (FDAL) and Item Development Assurance Level (IDAL) for every function/item plus the rationale (including any DAL-assignment credit taken for architectural mitigation, e.g. dissimilarity/monitoring, which itself must be independently justified and recorded); validation evidence that requirements are correct/complete relative to higher-level need (review, analysis, similarity argument, or test — whichever was used, and why); bidirectional traceability records: hazard → system requirement → item (SW/HW) requirement → verification evidence, with gaps explicitly flagged not silently absent; System Safety Assessment (SSA) documentation showing safety objectives are satisfied by the as-built architecture, not just the as-designed one.
- **Confidence:** high

## Loggable items

- Record every Problem Report at the moment an anomaly (test failure, review finding, in-service report, or a proof obligation that fails to discharge) is detected — not retrospectively at closure — including who found it, the exact artifact/version it was found in, its severity/DAL impact, and its full disposition trail.
- For every requirement (system, then software HLR, then LLR), record the specific verification method claimed for it (review / analysis / formal proof / test with structural coverage) and keep this a per-requirement fact, never a blanket process-level claim.
- When a formal method (DO-333) is used to close a verification objective in place of testing, record: the formalized property statement, the soundness argument for why the proof discharges the DO-178C objective, every assumption/abstraction the model makes relative to the real code, and the proof-tool's identity+version+qualification status (DO-330 TQL) at time of use.
- Record structural coverage results (statement/decision/MC-DC per DAL) per source-code baseline version, and explicitly record and justify any coverage gap closed by analysis rather than test (dead code, deactivated code, or defensive code rationale).
- Record every configuration-managed baseline transition (SCI update) with the identity of every changed configuration item, the approved Problem Report or Change Request driving it, and the re-verification scope re-run against that new baseline — partial-reverification decisions must be justified and archived, not just executed.
- Record independent SQA conformity findings against the approved plans/standards as their own artifact, distinct from and not overwritten by engineering verification records — SQA independence is evidentiary, so its records must be separately attributable.
- At project closeout, record the Software Accomplishment Summary linking every life-cycle datum above to the specific certification basis (DAL, applicable DO-178C/DO-333 objectives) it was produced to satisfy, so an auditor can reconstruct compliance without re-deriving it.
- Record FHA/PSSA/SSA safety-assessment artifacts and the FDAL/IDAL assignment rationale (ARP4754A) at the system level BEFORE software verification evidence is treated as sufficient — software-level records without a logged link back to the hazard/DAL derivation are incomplete evidence, not merely under-documented.

## Integrity principles

- Verification method claimed per-requirement must be recorded as fact, not asserted at process level — 'we do formal verification' is not evidence; 'requirement R-042 discharged by formal proof P-17 with soundness argument S-3' is.
- A formal proof is only as trustworthy as the tool that produced it and the model's fidelity to the real code — both the DO-330 tool-qualification status and the soundness/abstraction argument must be logged as first-class artifacts, not implied by the proof's existence.
- Traceability must be bidirectional and gap-visible: an audit trail that only shows forward links (requirement→test) without the reverse (test→requirement, code→requirement) can hide silently-unverified requirements.
- Problem Reports and coverage gaps must be logged at time of detection, not laundered into a clean-looking closure record after the fact — the record of the messy middle is the auditable part.
- SQA/independent-check records must remain a separate, non-overwritable artifact from engineering self-verification, since the assurance value of SQA is specifically its independence.

## Formal-methods notes

DO-333 is the load-bearing formal-methods artifact in this domain: it does not just permit formal methods, it redefines which DO-178C Table A objectives a formal analysis can satisfy and requires that substitution be justified by an explicit, recorded soundness argument (what is proved, under what abstractions, against what real-code fidelity gap) rather than treated as a strictly-stronger drop-in for testing. Two things are easy to under-record and must be forced into the log: (1) the proof-tool's own assurance pedigree — per DO-330, any tool whose output is trusted without independent human verification of each result needs an assigned Tool Qualification Level (TQL-1..5) derived from Tool Qualification Criteria x software DAL, plus its own Tool Operational Requirements/Verification Records/Configuration Index, all pinned to the exact tool version used; (2) proof completeness — a "verified" system may in fact have a formal model with unproved obligations discharged by human review or scoped out by assumption, and DO-333 requires that residue to be visible in the record, not folded into an undifferentiated "formally verified" checkbox. ARP4754A's FDAL/IDAL split adds a system-level formal-methods-adjacent obligation: if a DAL assignment takes credit for architectural mitigation (e.g., a formally verified monitor providing independence from a less-assured function), that credit claim and its own supporting analysis must be recorded as a distinct, auditable rationale, since it is exactly the kind of reasoning most likely to be asserted without evidence under time pressure.
