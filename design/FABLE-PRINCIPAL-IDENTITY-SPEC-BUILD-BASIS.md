# FABLE-PRINCIPAL-IDENTITY-SPEC — the principal surface remediated as event-sourced identity (kernel delta family s40–s41)

**Status:** RATIFIED build basis, frozen 2026-07-18 (see the build-basis header above). Fable-authored (fresh-context, no session state), per the standing rule that nobody edits kernel/lineage without a Fable-authored, maintainer-ratified spec, with six reserved decisions (§9 a/b/c/d/f/g) subsequently ratified by the maintainer and incorporated directly (ledger rows 1425, 1426) — the same in-place-ratification pattern already used for decision (e) on 2026-07-17. Sonnet builds; nothing here is applied to any existing [world](../GLOSSARY.md#world) ([runs are strictly linear](../CLAUDE.md#orchestration--the-standing-delegation-contract-2026-07-09), maintainer ruling 2026-07-11) — the deltas reach reality only by entering a future world's [birth chain](../GLOSSARY.md#birth-chain).

**What this document is, in plain words:** the kernel's `principal` table — the record of *who* every ledger row is attributed to — is four columns with no history, no registrar, no lifecycle, and no link to any of the system's authentication machinery. It is the one load-bearing surface in the kernel exempt from the kernel's own append-only, attributed, tamper-evident record discipline, and the panel deployment — this project's own simulated downstream test-bed at `~/w/vdc/1/experience/autoharn-panel`, a separate installation of this same harness whose lived history is the primary field evidence throughout this arc — has already paid for that live (silent duplicate registrations, an unusable delegation column, sibling principals whose classes disagree). This spec remediates it with the kernel's own idiom: the principal becomes an immutable anchor; everything that can happen to an identity becomes an append-only, attributed ledger event; current standing becomes a derived view the write path consults; and attribution becomes strict-by-default for new worlds, with the default *declared at birth as an explicit act* rather than silently assumed. Two kernel deltas (s40, s41) plus CLI and scaffold changes in the same commits carry it.

**Primary inputs, all read in full:** [design/ORCH-CONSULT-PRINCIPAL-SURFACE-2026-07-18.md](ORCH-CONSULT-PRINCIPAL-SURFACE-2026-07-18.md) (the ratified consultation; its §4 is the closure universe this spec is checked against); [design/ORCH-METACONSULT-RCA-CLASS-WIDTH-2026-07-18.md](ORCH-METACONSULT-RCA-CLASS-WIDTH-2026-07-18.md) (Axes 3 and 6 bear directly here); [CLAUDE.md](../CLAUDE.md); [ADR-0000](../law/adr/0000-the-alpha-and-the-omega-type-driven-design.md) (incl. the 2026-07-02 Rule 2(a) amendment and Revisit #4), [ADR-0011](../law/adr/0011-mechanization-discipline.md), [ADR-0012](../law/adr/0012-compositional-and-structural-hygiene.md) (via its P1/P2/P8 spine), [ADR-0013](../law/adr/0013-execution-integrity.md), [ADR-0017](../law/adr/0017-the-zero-context-reader.md); `kernel/lineage/` [s13](../kernel/lineage/s13-schema.sql)–[s39](../kernel/lineage/s39-blocks-start.sql) with emphasis on [s15](../kernel/lineage/s15-schema.sql), [s17](../kernel/lineage/s17-stamp-mechanism.sql), [s18](../kernel/lineage/s18-criterion-principals.sql), [s19](../kernel/lineage/s19-trigger-search-path.sql), [s29](../kernel/lineage/s29-obligation-item-key-and-typed-close.sql)/[s30](../kernel/lineage/s30-typed-dependency-edges.sql) detect discipline, [s31](../kernel/lineage/s31-supersession-uniform-retraction.sql), [s35](../kernel/lineage/s35-validation-decomposition.sql), [s38](../kernel/lineage/s38-bookkeeping-close.sql).

## 0. Executive summary (added for legibility, ADR-0017; compresses §1–§9, adds no new content)

**s40 and s41, named plainly.** They are the two kernel deltas this spec specifies, shipped as one ratified family. **s40** (`s40-principal-identity-events`) makes an identity's own facts — its registration, its standing (active/suspended/revoked), and which database role speaks for it by default — into append-only ledger events instead of silent columns; it also turns on strict attribution (every write must resolve to a registered principal that is neither revoked nor suspended) as the only posture, with a declared-not-silent default so a solo operator's world is never inconvenienced. **s41** (`s41-principal-bindings-and-relations`, hard-depends on s40) adds everything a registered identity can be bound *to*: roles, cryptographic keys (the empty slot, not the ceremony — key generation stays deferred), competence grants (who is believed competent for what, at what band, on what basis), and typed relationships to other principals (acts-for, dispatched-by, same-natural-person, succeeds). Both deltas reach reality only at a *future* world's birth (runs-are-linear); nothing here touches any existing deployment.

**Why.** The current `principal` table is four flat columns with no history — the one surface in the kernel exempt from its own append-only, attributed discipline. That gap has already cost the panel deployment real defects in the field: silent duplicate registrations (`ON CONFLICT (name) DO NOTHING` swallowing any re-registration by name as a no-op), sibling principals whose classes disagree, an `acts_for` column too thin to ever be used, and no recorded distinction between a `reviewer` and a `reviewer2` role assignment. The consultation grounding this spec (§1's cited primary input) traced the gap to NIST SP 800-63's identity/lifecycle/binding/relationship decomposition; this spec is the kernel-idiom fix — the same "append-only event, derived-view standing" pattern s29/s30/s31 already established, applied to the one table that predated it.

**Already ratified by the maintainer, in force in this draft (not open questions):** strict attribution ON by default with no permissive/toggle mode (row 1398; §3.6's concurring reconciliation is declared-not-silent defaults, not a switch); competence grants BUILT in v1, not merely named (row 1411); reserved decision (e) — refusal-event recording — resolved as a *direction* only ("ledger grade, of course," row 1419: a future kernel write-boundary redesign, candidate E, sequenced strictly after s40/s41, not built by this spec).

**Six decisions are still open, reserved to the maintainer at ratification (§9 has the full text):**
- **(a)** How strictly to scope "managerial/financial independence" claims between principals — the spec's default is human-attested scoping; the stricter alternative is an outright refusal of those claims in this deployment class.
- **(b)** Whether to build ANY reinstatement mechanism at all — v1 ships with none; the only escape from suspended/revoked standing is registering a fresh successor principal.
- **(c)** The role vocabulary's starter membership — the spec defaults to the three kernel-grounded roles (`author`, `reviewer`, `commissioner`); `orchestrator` and the panel's finer-grained names (`scout`, `item-countersign`) are named but deferred for lacking a v1 producer.
- **(d)** Whether s40/s41 ship as two deltas (the spec's default, matching the s37/s38 grain) or one.
- **(f)** Whether to add a cheap, additive column recording *how* a row's actor was resolved (explicit `LED_ACTOR` vs. the standing-declaration default) — named but not built in v1.
- **(g)** Whether the competence band/activity fields (currently free-text, non-empty) become a closed vocabulary, and under which band system — deferred until real deployment data exists to ground one.

**Scope discipline.** Ten items are explicitly named OUT of this spec, not silently absent (§7) — most notably the refusal-recording redesign (item 9, ratified as direction only) and all key-ceremony operations (item 3, standing deferral, unaffected by this spec). §8 states seven honest limits, the sharpest being: a superuser can always bypass the triggers, and value-continuity on release/revoke/withdrawal is enforced by the CLI verbs, not by a kernel-level cross-row check.

## 1. The defect, named at class level (ADR-0000 Rule 2)

**(a) The type question.** The defect class, in its most general form (the consultation's §2, adopted): *an authority-carrying identity whose authority, scope, and validity period are not representable, so every misuse of it is representable.* Instances already witnessed: the silent `ON CONFLICT (name) DO NOTHING` duplicate no-op (panel ids 1–27 with burned gaps; meta-consult Axis 6 — the axis that found this exact no-op still live and uncovered by the RCA's earlier fix); `acts_for` unused across 1800+ rows because it is single-valued, undated, kind-less; `reviewer` (subagent) vs `reviewer2` (model) class drift with no event recording either assignment; registration producing no record of registrar, time, purpose, or approval; a ratified GPG trust layer whose §5 assumes a principal↔key binding the schema cannot represent. The foreclosing type is the kernel's own: **identity facts as append-only attributed events; current standing as a derived view; the anchor immutable.** This is s31's supersession, s29's derived-not-stored verdicts, and s30's typed edges, applied to the one table that predates the discipline.

**(b) The operational lapse.** Executive-side, per Rule 2(b): the principal table was built before the record discipline existed and no audit ever ran registry-rooted at the IA/AC families until the consultation did (the same silent-omission mechanism ADR-0000 Revisit #4 records for read-logging). The net minted here: the deltas' own refusals plus their detect siblings; the standards mapping is carried in the consultation and cited, not re-derived.

**Confidence: high.** The diagnosis is the consultation's central finding, held there at high confidence and over-determined by three independent lines (standards, LAW, the panel's lived need).

## 2. Design overview — what a principal IS after s41

Four facts, four homes (the consultation's §2.2, the SP 800-63 decomposition):

1. **Identity** — `kernel.principal(id, name, agent_class)`: the immutable anchor. `name` immutable (rename is ruled OUT; succession instead — §5 Axis B). `agent_class` immutable at birth (a class change is not an event; it is a new principal plus a `succeeds` edge — this forecloses the reviewer/reviewer2 drift class rather than recording it politely). `acts_for` retired (§4, D-8).
2. **Lifecycle and standing** — append-only ledger events (`principal_registered`, `principal_suspended`, `principal_revoked`, `principal_standing_declared`), retraction uniformly by supersession (s31 idiom, no second retraction mechanism). Current standing is the derived function/view `principal_standing` — never a stored status column.
3. **Bindings** — role bindings (`principal_role_bound`, closed vocabulary), key bindings (`principal_key_bound`, the empty-until-ceremony slot, human principals only), db-role standing declarations (the strict-attribution channel), and **competence grants (`principal_competence_granted`) — BUILT in v1 by maintainer ruling (2026-07-17, ledger row 1411 — a numbered entry in this project's append-only decision/audit ledger, see [GLOSSARY.md](../GLOSSARY.md#provenance--ledger) — overruling the consultation's named-not-built suggestion)**: the [BRIEF](../law/briefs/safety-critical-logging/BRIEF.md)'s **G13 record** (its obligation-register entry for a competence record — who is believed competent for what activity, at what band, on what basis, granted by whom, when) lands as an ordinary attributed event, per D-5's recordable-not-gating posture (D-5, defined below in §4) — nothing checks competence before accepting an act in v1; the record exists so that when grading-against-bands arrives it has history to stand on, and so G13's obligation has a typed home today rather than a reserved name.
4. **Relationships** — s30-style typed principal↔principal edges as events (`principal_relation_asserted`, closed vocabulary `{acts-for, dispatched-by, same-natural-person, succeeds}`), each dated and attributed, retracted by supersession.

**Why events live in the LEDGER, not a new kernel table (a load-bearing choice, made here so the builder cannot fork on it):** a separate `kernel.principal_event` table would have to re-mint append-only triggers, actor attribution, [stamps](../GLOSSARY.md#stamp), the [s26](../kernel/lineage/s26-row-hash-chain.sql) hash chain, review/countersign, and supersession — a wholesale second copy of the kernel's record machinery, the exact ADR-0012 P1 violation this spec exists to end. As ledger kinds, identity events inherit all of it for free, including countersignability: a registration event is an ordinary ledger row, so the panel's invented proposal→countersign ceremony becomes expressible with **zero new review machinery** — a review row `regards` the registration event. The cost is new kind-scoped columns on `ledger`, which is the established house pattern (every `work_*` column since [s22](../kernel/lineage/s22-work-item-ledger.sql)).

**Rung structure — argued different from the consultation's ladder.** The consultation proposed rung 1 (`revoked_at`/`revoked_by` columns + refusals, class-ratifiable) and rung 2 (the event stream superseding those columns). That ladder was priced for a world where rung 1 could ship to deployments early. Under runs-are-linear it cannot: *both* rungs reach reality only at the next world's birth, in the same chain, so the interim columns would be born already-superseded scaffolding — dead weight that violates the event-sourced invariant the same chain establishes one delta later. This spec therefore collapses the ladder into two deltas ratified together as one family: **s40** (identity events, standing, strict attribution, the registration ceremony) and **s41** (relationships, role bindings, key bindings, the independence-honesty refusal, `acts_for` retirement). s41 hard-depends on s40. Shipping s40 and silently dropping s41 would be the ADR-0000 contrast-specimen patch; if s41 is deferred, it is deferred by the maintainer's explicit word and filed, never by executor attrition (ADR-0013 Rules 1–3).

**Confidence: high** on the event-sourced direction and ledger-resident events; **medium-high** on the two-delta split (a one-delta variant is defensible; two keeps each delta's closure statement and witness plan tractable, matching the [s37](../kernel/lineage/s37-violation-disposition.sql)/s38 grain).

## 3. Delta s40 — `s40-principal-identity-events`

### 3.1 New ledger kinds (widening the closed kind CHECK, s38 idiom: DROP/ADD, additive vocabulary)

- `principal_registered` — the birth event of an identity.
- `principal_suspended` — standing withdrawn. No v1 verb lifts it (§3.4 below states this precisely and why); the only v1 path back to active standing is a fresh successor principal via `succeeds`.
- `principal_revoked` — standing withdrawn. Same posture as suspension: no v1 reinstatement verb (§3.4).
- `principal_standing_declared` — the explicit act binding a database role to a principal as its standing (default) attribution. This is the declared-not-silent default the ratification requires.

### 3.2 New kind-scoped ledger columns (all nullable, no column DEFAULT — the s30 lesson)

- `principal_subject bigint REFERENCES kernel.principal(id)` — the principal the event is ABOUT (distinct from `actor`, who performed the act). CHECK `principal_subject_kind_shape`, **two-way** for the four s40 kinds plus s41's kinds (mandatory on every `principal_*` row, forbidden elsewhere). Two-way is safe here where s30 had to be one-way: no pre-existing row can carry these kinds (the kind values are born in this same delta), so ADD CONSTRAINT validates vacuously — state this in the HISTORY analysis, per the s38 pattern.
- `principal_purpose text` — mandatory non-empty on `principal_registered` (CHECK: `kind IS DISTINCT FROM 'principal_registered' OR btrim(coalesce(principal_purpose,'')) <> ''`), forbidden on every other kind (one-way kind-shape CHECK). This is AC-2's "account with a stated purpose" landed as a typed column, not decision-row prose — the panel procedure's purpose field given its home.
- `principal_db_role text` — mandatory on `principal_standing_declared`, forbidden elsewhere (two-way).

Per the s38 `work_review_bookkeeping_kind_shape` lesson (**the freshest lesson — enumerate every kind carrying every column you constrain**): each new column's CHECK enumerates its full licensed kind set explicitly; the builder verifies by grep over the delta itself that no CHECK is flat/kind-blind. `principal_subject` is licensed on exactly: `principal_registered`, `principal_suspended`, `principal_revoked`, `principal_standing_declared` (s40) and, after s41 re-widens the same CHECK, `principal_relation_asserted`, `principal_role_bound`, `principal_key_bound`, `principal_competence_granted`. No other kind, ever, without a ratified amendment.

### 3.3 The anchor coupling — a bare registration is unrepresentable

A new **deferred constraint trigger** on `kernel.principal` (AFTER INSERT, DEFERRABLE INITIALLY DEFERRED): at commit, every newly-inserted principal row must have a same-transaction `principal_registered` ledger row whose `principal_subject` is the new id. This closes the consultation's §3(a) — the kernel can now distinguish a governed registration from a bare one, because a bare one **cannot commit**. Anchor rows predating s40 (in a birth chain: exactly the s15-seeded `author`) are untouched — AFTER INSERT triggers do not fire retroactively; the scaffold discharges `author`'s event explicitly (§3.7). `agent_class` immutability needs no new mechanism at the anchor beyond the existing absence of UPDATE paths, but the builder adds an explicit append-only trigger on `kernel.principal` (UPDATE/DELETE refused for every role, the ledger's own `append_only()` reused — ADR-0012 P1) so the anchor's immutability is enforced, not assumed. TRUNCATE likewise.

### 3.4 Standing, derived — and the finding-45 foreclosure

`kernel.principal_standing(pid bigint) RETURNS text` is the new function. (A disambiguation, stated once because the word is overloaded: "standing" here means an IDENTITY's current status — active, suspended, revoked — and is unrelated to s36's `standing_decisions`/`led standing` vocabulary, which denotes decision-grade durability; the new verb is namespaced `led principal declare-standing` and shares nothing with `led standing`.) It is `SECURITY DEFINER`, following s17's `stamp_valid` precedent (`s17-stamp-mechanism.sql:65`, itself `SECURITY DEFINER` with no volatility marker, i.e. VOLATILE by default). `principal_standing` adds `STABLE` on top of that precedent: unlike `stamp_valid` — which consults a per-session apparatus secret — nothing this function reads changes within one statement, so the stricter marker is a correct addition, not a borrowed property. The two functions share the SECURITY DEFINER escape and its rationale, not an identical signature. EXECUTE is granted to the writing role(s), for the same reason `stamp_valid` needed it: the standing computation reads ledger rows (`kind`, `principal_subject`, `supersedes`), and a zero-SELECT writer in an s18-style deployment (a deployment with INSERT-only, no-table-SELECT writer roles, per `kernel/lineage/s18-criterion-principals.sql`'s two criterion-reviewer roles) must not be refused inside the trigger chain. This is the finding-45 class: a SECURITY-INVOKER trigger-chain refusal failure mode where a zero-SELECT writer cannot pass validation because the chain's own functions read structural columns the writer has no grant on (per `kernel/lineage/s18-criterion-principals.sql`'s finding-45 comments). It is foreclosed structurally here, rather than by widening column grants. Returns `'active'` (registered or legacy-anchored, no unsuperseded suspension/revocation), `'suspended'`, `'revoked'`, or `'unregistered-legacy'` (an anchor row with no registration event — representable only for pre-s40 seeds in a partially-migrated scratch; named honestly, treated as active for write purposes so a legacy `author` is never bricked). **Precedence, stated because nothing prevents both an unsuperseded `principal_suspended` and an unsuperseded `principal_revoked` row coexisting for the same principal** (a fixpoint-review finding — this spec's own internal self-critique pass conducted before ratification, catching an underspecification in an earlier draft rather than an external review — 2026-07-17: suspend-then-revoke and revoke-then-suspend are both legal write sequences, and the function's return was previously unspecified for that case): **`'revoked'` always dominates `'suspended'`** — the function checks for an unsuperseded revocation first; only absent one does it check for an unsuperseded suspension. This is a strict severity ordering, not an event-recency ordering — revoking an already-suspended principal is a clean escalation; suspending an already-revoked principal is legal to write (it may carry its own useful rationale in the row) but changes nothing observable. **Reinstatement, honestly: NOT built in v1, for either kind.** An earlier pass of this spec said revocation's reinstatement was merely "representable by supersession" without naming what row does the superseding — on inspection, nothing does: there is no kind whose write, superseding a `principal_suspended` or `principal_revoked` row, means "this no longer holds" (unlike the four s41 kinds, which got exactly that mechanism via `principal_binding_active` above). Adding one is a real design question — same-kind supersession reads as "re-suspend"/"re-revoke," not "lift" — and is deliberately left to a future amendment rather than solved under review pressure here. In v1: a suspended-or-worse principal has no path back to active except registering a fresh principal and asserting `succeeds` (D-2) between the two — a real, if heavier, escape hatch that already exists. A companion read view `principal_standing_current` (one row per principal: name, class, standing, registered_at, registrar, purpose) is the human/SPA surface; `security_invoker`, SELECT granted to `:role`.

### 3.5 `principal_role` becomes a derived view (one fact, one home)

The s15 table `kernel.principal_role` is dropped and re-created as a **view** of the same name and column shape (`db_role`, `principal_id`) over unsuperseded `principal_standing_declared` events (latest unsuperseded declaration per db_role wins; the view enforces one-current-binding-per-role by construction). Same shape means `set_actor` and every existing reader keep their query text; the meta-consult's Axis 3 finding — that `set_actor` is itself re-declared at the lineage head, by s19, so any remediation must target that live body rather than a frozen one — is honored in full: **the delta re-issues `set_actor` against the lineage head's declaration — s19's search_path-carrying body, not s15's frozen text** — and s40's detect fingerprints *behavior* (the new refusal's marker text searched over every function the schema owns, plus view-vs-table for `principal_role` via `information_schema`), never a pinned object name, per the migrate-detect-drift ruling of 2026-07-16 (recorded in commit `77e14dc`'s message — a git-history referent, cited by hash).

### 3.6 Strict attribution — the re-issued `set_actor`, and exactly what strict means

There is **no mode toggle**. In an s40+ kernel, strict is the only posture; the reconciliation with ergonomics is the declared default, not a switch. (A dormant permissive branch would be untestable dead code and an invitation to silent flips.) Existing worlds keep their posture by construction — deltas are never applied to them (runs-are-linear); no retroactive flip is possible, and none is licensed.

The re-issued `set_actor` (s19 idiom: `SET search_path = :"schema", :"kern", pg_temp`; body reads unqualified):

- **Strict refuses, with teach-text:**
  1. *An undeclared write*: `NEW.actor IS NULL` and no `principal_role` (view) row for `current_user` → `RAISE EXCEPTION` teaching `./led principal declare-standing` — a legible refusal replacing today's bare `NOT NULL` violation. (Today an unmapped role is also refused, but by an untaught constraint error; the semantics tighten from "any hand-inserted map row silences this" to "only a declared, attributed, dated event does".)
  2. *A write under a revoked or suspended principal*: after resolution (explicit or defaulted), `principal_standing(actor) IN ('revoked','suspended')` → refuse, naming the standing event row id.
  3. *A write under an unregistered principal*: already unrepresentable by the FK (verified, not new — stated so the closure statement can claim it without claiming new work). The CLI's unregistered-`LED_ACTOR` refusal in `resolve_actor` stands unchanged.
- **Strict never refuses:** a `NULL` actor under a connection role with a current standing declaration (it resolves to the declared principal — this IS the sanctioned default); an explicit actor that is registered and active. The scaffold's birth declaration guarantees every new world starts in the never-refused regime — strict-on costs the solo operator nothing, which is the ratified reconciliation.

Trigger name, timing, and alphabetical position are unchanged (`set_actor` still fires first in the BEFORE INSERT chain, before `set_stamp` and the validators — verified against the s17/[s21](../kernel/lineage/s21-session-aware-distinctness.sql)/s26 ordering notes).

### 3.7 The registration ceremony (CLI + scaffold, same commit)

`led register-principal <name> <class> --purpose "<text>"`:

- The `ON CONFLICT (name) DO NOTHING` is **deleted**. A duplicate name is refused loudly *before* the INSERT by an existence check whose teach-text quotes the existing principal's id, class, and registration purpose. This closes two things at once: the panel's ask #2 (the panel's Finding 1 item asking that `register-principal`'s silent no-op on a duplicate name be fixed), and the meta-consult's Axis 6 finding — that this exact no-op was the one sibling the RCA's earlier fix had silently left uncovered. Both close in both polarities: **same-class duplicate → refused** (the name exists; re-registration is never a no-op); **class-mismatch duplicate → refused**, with text naming the mismatch explicitly (*"registered as <class-A>; classes are immutable — a different class is a different principal: pick a new name and, if this succeeds the old identity, record `./led principal relate <new> succeeds <old>` once s41 has landed"*, or, if s41 is deferred past s40's own ship, the teach-text instead points at filing a decision row naming the successor relationship until `relate` exists — s40's teach-text must not emit a verb its own kernel cannot yet honor). Named and **ruled: refused**, not merely named. The kernel backstop is the existing UNIQUE constraint, now no longer suppressed, plus the sequence no longer burning ids on silent conflicts.
- A registration is one transaction: anchor INSERT + `principal_registered` ledger row (actor via `resolve_actor`; purpose mandatory — a bare `register-principal` without `--purpose` is refused at the CLI with usage text). The kernel's deferred constraint trigger (§3.3) makes the event half non-optional even for a hand-driven psql registration.
- Approval: v1 records the **registrar** (the event's `actor`) and leaves approval as an optional countersign — a review row regarding the registration event, exactly the panel's ceremony shape, now expressible with typed parts. Making a countersigned proposal a *precondition* of registration is deliberately **not** built in v1 (it would gate the solo world's every registration on a second principal that registration exists to create); named as the rung the panel's procedure can standardize on top of, future amendment.
- `led principal suspend <name>`, `led principal revoke <name>`, `led principal declare-standing <name>` — thin verbs writing the corresponding event kinds, each through `resolve_actor`, each binding `actor` and honoring `--event-time` per the [s24](../kernel/lineage/s24-declared-event-time.sql) convention. Per the meta-consult's Axis 1 — which found roughly 8 operator-intent channels across non-generic `led` verbs silently dropped the same way `LED_ACTOR` once was — named here so the new verbs do not widen that same debt: every new verb either honors or loudly refuses each shared-loop channel; the *general* channel-coverage gate remains OUT of this spec (§7).
- **Scaffold (`bootstrap/new-project.sh`, same commit):** after the lineage apply, the birth sequence becomes explicit acts in this order — (1) `author`'s `principal_registered` event (the one **genesis exception**: self-attributed, `actor = author`, purpose text naming it the scaffold connection principal; the self-attribution is named in the event's own statement, mirroring the hash-chain genesis-seed precedent for a first link that cannot reference a predecessor); (2) `principal_standing_declared` binding the world's role to `author` (actor = author, explicit); (3) `reviewer` and `commissioner` registered through the full ceremony (actor = author, purposes stated). The `ON CONFLICT` re-run-idempotence the scaffold comments rely on is replaced by scaffold-side existence checks that print "already registered" — idempotent at the scaffold, never silent at the verb.

## 4. Delta s41 — `s41-principal-bindings-and-relations` (hard prerequisite: s40)

**D-1. New kinds** (kind CHECK re-issued wider): `principal_relation_asserted`, `principal_role_bound`, `principal_key_bound`, `principal_competence_granted` (the fourth added by maintainer ruling 2026-07-17, ledger row 1411 — see §2's item 3, the "Bindings" list above). Retraction of any of the four: supersession, uniformly (s31); no `*_revoked`/`*_released` kinds are minted — one retraction mechanism, one home.

**Withdrawal-with-no-replacement, closed here (a fixpoint-review finding, 2026-07-17):** a first pass of this spec said a pure retraction (release a role binding, revoke a key, withdraw a competence grant with no successor value) is "just a supersession" without specifying what a builder writes into that superseding row's own mandatory content columns — genuinely underspecified, since "supersede with no new content" is not directly representable when a kind's columns are unconditionally mandatory-non-empty. The fix: each of the four new kinds' columns split into **identity fields** (which binding this row is about — always mandatory, on every row of the kind, active or not, since a retraction that doesn't say what it retracts isn't a retraction) and **value fields** (the content of an active binding — mandatory when the binding is asserted, forbidden when it is not). One shared boolean carries the distinction: `principal_binding_active boolean`. It is nullable, with NO column DEFAULT (§3.2's own house rule). This column follows that rule exactly — contrary to a fixpoint-review-caught contradiction in an earlier draft of this paragraph, which gave it a literal `NOT NULL DEFAULT true`. (A column-level default on a column shared by every row of the single `ledger` table would populate `true` on every OTHER kind's writes too, which the mandatory-on-four/forbidden-elsewhere CHECK below would then refuse; the CHECK is the only place "mandatory" is enforced.) The column is mandatory (kind-shape CHECK, same two-way discipline as `principal_subject`) on the four new kinds, and forbidden on every s40 kind and the generic path (added to D-2 below). A fresh assertion's INSERT explicitly supplies `true` in its VALUES list — nothing defaults it; an INSERT of one of the four kinds that omits the column is refused by the mandatory-when-kind CHECK, exactly like any other missing kind-scoped column in this spec. A fresh assertion is `active = true` with every value field populated; a retraction is a NEW row of the SAME kind, `supersedes` the prior row, `active = false`, restating only the identity field(s) with every value field NULL. A structural CHECK refuses `active = false` with `supersedes IS NULL` — an inactive-from-birth binding (one that was never asserted) is nonsensical and unrepresentable. For `principal_relation_asserted`/`principal_role_bound`/`principal_key_bound`, ALL of their content columns (`principal_relation` AND `principal_object` together, for relation_asserted; `principal_role_name`; `principal_key_fingerprint` respectively) ARE identity fields — there is no separate value field to make optional, so these three kinds are unaffected beyond gaining the shared boolean. Only `principal_competence_granted` (D-1a) splits: `principal_competence_activity` is identity (always mandatory — a principal may hold grants for several activities, so the retraction must say which one); `principal_competence_band`/`principal_competence_basis` are value (mandatory when active, forbidden when not).

**D-1a. The competence grant's columns** (same discipline as D-2 below; forbidden on every other kind, per-column kind-shape CHECKs; split per the withdrawal fix above):
- `principal_competence_activity text` — the safety activity or role the competence regards (G13's "safety activity" field). **Identity field: mandatory non-empty on every `principal_competence_granted` row, active or not.**
- `principal_competence_band text` — the integrity band claimed (G13's ASIL/SIL/DAL-style field). **Value field: mandatory non-empty when `principal_binding_active = true`, NULL when `false`.** Free text, non-empty, in v1 — **ratified 2026-07-18 (§9(g), ledger row 1426) as a placeholder architecture only, not a considered final design**; whether this becomes a closed vocabulary (and with which band system) is deferred until the deployment population's band conventions exist and future executive bandwidth allows extending it properly. A closed CHECK minted now, before those conventions exist, would be guessing.
- `principal_competence_basis text` — the basis for believing the subject meets the band (G13's "competence basis": training, track record, certification reference). **Value field: same mandatory-iff-active shape as band.** Quality unenforceable beyond non-emptiness; the countersign path is the control (same honesty as registration purpose, §8).
The grant's subject is `principal_subject` (any agent_class — model principals are exactly who competence records will matter for); the grantor is the event's `actor`. A derived view `principal_competences` (unsuperseded, `active = true` grants: subject, activity, band, basis, granted_by, at) joins D-5's list (D-5, defined below) — a withdrawal row (`active = false`) is superseding, so it drops the withdrawn grant from this view by construction, same as any other supersession; the terminal `active = false` row itself remains in raw history. Two CLI verbs join §3.7's family, same channel discipline: `led principal grant-competence <name> --activity "<a>" --band "<b>" --basis "<c>" [--supersedes <old-id>]` (fresh grant, `active = true`; refused if an active grant already exists for that exact activity/subject pair and `--supersedes` is omitted — same duplicate-active guard as `bind-role`/`bind-key`; supplying `--supersedes <old-id>` naming the currently-active grant turns the call into a re-band/re-basis correction rather than a refused duplicate, the s39-depends-correction idiom applied here); `led principal withdraw-competence <name> --activity "<a>" --supersedes <old-id>` (retraction — activity only, `active = false` implied, refused if `<old-id>` is not currently present in `principal_competences` — D-5's view, the definition of "active" used throughout this section, not the row's own `principal_binding_active` column read in isolation — for that exact activity/subject pair; a mismatched activity is refused the same way a mismatched role name or fingerprint is, below). Recordable, NOT gating: nothing in v1 consults competence before accepting any act — enforcement, like entitlement enforcement, is a named follow-on amendment.

**D-2. New kind-scoped columns** (same discipline as §3.2, and the `principal_subject` kind-shape CHECK from s40 is re-issued here widened to license all FOUR new kinds — `principal_relation_asserted`, `principal_role_bound`, `principal_key_bound`, `principal_competence_granted` — enumerated in full in the re-issued CHECK, never patched by a second constraint; a re-issue licensing only three and omitting `principal_competence_granted` is a defect, not a valid reading — D-1a's grant machinery requires it):
- `principal_binding_active boolean` — mandatory (NOT NULL) on all four new kinds, forbidden on every s40 kind and the generic path (the withdrawal-fix column named in D-1 above). CHECK `principal_binding_active OR supersedes IS NOT NULL` — an inactive-from-birth binding is unrepresentable; only a superseding row may set it false.
- `principal_actor_resolution text` — **ratified 2026-07-18 (§9(f), ledger row 1426).** Nullable, kind-agnostic (no per-kind CHECK — every ledger row, of any kind, may carry it), shape CHECK `IN ('explicit','declared-default')` when non-NULL. Set by `set_actor` (§3.6) at write time: `'explicit'` when the write supplied a non-NULL `actor` before resolution (an explicit `LED_ACTOR`), `'declared-default'` when `set_actor` resolved a NULL actor via the standing declaration (§3.1's `principal_standing_declared`). NULL only on pre-s40 history, where the distinction cannot be reconstructed. This closes the intent-forensics gap named in §9(f): today the two paths are indistinguishable on the row, which is exactly what forced the panel's misattribution investigation to reconstruct intent from session transcripts instead of the ledger itself. Recordable, not gating — nothing refuses on this column's value; it exists so the declared-default regime is auditable, not merely correct.
- `principal_object bigint REFERENCES kernel.principal(id)` — mandatory on `principal_relation_asserted` (identity field, mandatory active or not — see D-1), forbidden elsewhere.
- `principal_relation text` — mandatory on `principal_relation_asserted` (identity field), closed vocabulary CHECK `IN ('acts-for','dispatched-by','same-natural-person','succeeds')`. Forbidden elsewhere. **Creation verb, formally specified here** (this is the kind §3.7's registration teach-text names informally as `led principal relate <new> succeeds <old>` — that mention is a forward-reference to this spec, not a second specification): `led principal relate <subject> <relation> <object>` — writes a fresh `principal_relation_asserted` row, `principal_binding_active = true`; refused if `<subject> = <object>` (D-3's self-edge check); refused if an active (D-5-view, D-5 defined below) assertion already exists for the exact `subject`/`relation`/`object` triple (no duplicate active truth — same guard `bind-role`/`bind-key` carry below). **`same-natural-person` is symmetric by definition (a fixpoint-review finding, 2026-07-17: a write-side "check both orderings" duplicate guard alone leaves read and retract silently order-dependent — a caller querying only `subject = X` misses X's same-natural-person facts recorded with X as `object`, and `unrelate` in the "wrong" order would be refused as a false mismatch). Closed by CANONICALIZING at write time instead of patching each consumer separately:** for this one relation, `relate` always stores the LOWER-`id` principal as `subject` and the higher as `object`, regardless of the order the operator names them — `relate B same-natural-person A` (where `id(A) < id(B)`) silently writes `subject = A, object = B`. One canonical row shape per unordered pair means the duplicate guard above needs no special-casing (it is already a plain equality check on the canonical triple), and `unrelate` accepts either ordering, canonicalizing identically before comparing against its `--supersedes` target — the CLI performs the same normalization on both ends, so a caller never needs to remember which order was originally used. Reading remains a caller-side concern the same as any bidirectional fact table: a query for "is X the same natural person as anyone" checks `subject = X OR object = X` filtered to that relation; `principal_relations` (D-5) itself stays a raw, ordered projection with no added view machinery for one relation kind out of four. The other three relations (`acts-for`, `dispatched-by`, `succeeds`) are directional — no canonicalization, the guard compares the stated ordering only. Retraction: `led principal unrelate <subject> <relation> <object> --supersedes <old-id>` — same CLI-refusal shape as `release-role`/`revoke-key`: refused if `<old-id>` is not in `principal_relations` for that exact triple (canonicalized first, for `same-natural-person`), or if the restated triple mismatches the target's.
- `principal_role_name text` — mandatory on `principal_role_bound` (identity field, mandatory active or not), CHECK non-empty (`btrim(principal_role_name) <> ''`), **free text — NOT a closed vocabulary** (maintainer ruling, 2026-07-18, ledger row 1426: role naming is organizational configuration, not a kernel-structural fact — nothing in this delta's own CHECKs or triggers branches on *which* role name a binding carries, unlike `principal_relation`'s four values, which D-3's self-edge refusal and the `same-natural-person` canonicalization CHECK key off directly. An earlier draft of this spec closed this column to `IN ('author','reviewer','commissioner')`, citing only the kernel's own existing role surfaces as grounding — not any external standard; the grounding consultation itself (`ORCH-CONSULT-PRINCIPAL-SURFACE-2026-07-18.md` §2.2.2) flagged the specific vocabulary as a maintainer/spec-author judgment call it deliberately left open, not a standards derivation, and even suggested a broader starter slate. On inspection, no NIST 800-53 control or OSCAL construct mandates a fixed role-name enumeration: AC-2 requires account *types* be defined, approved, and reviewed as a governance *process* — it does not require the type strings themselves to come from a small set closed at the database level. Corrected here to the same free-text-non-empty pattern already used for `principal_purpose` and the competence `basis` field (§3.2, D-1a) — a deployment names its own roles; the kernel enforces only that a name was given.) Forbidden elsewhere. **Creation:** `led principal bind-role <name> --role "<role-name>"` — writes a fresh `principal_role_bound` row, `active = true`; refused if an active (D-5-view) binding already exists for that exact `principal_subject`/`principal_role_name` pair (no duplicate active truth — bind twice for two DIFFERENT roles, or release first to rebind the same one). **Release:** `led principal release-role <name> --role "<role-name>" --supersedes <old-id>` — CLI-refused, same shape as competence's withdrawal (below), if `<old-id>` is not a row currently present in `principal_role_bindings` (D-5's view — "active" is defined there, not by the row's own `principal_binding_active` column read in isolation, which never changes once set even after superseding: a second release attempt against an already-released row's id is refused because that id has left the view, not because the column changed) for that exact `principal_subject`/`principal_role_name` pair; the new row restates the SAME `principal_role_name` the target carried — a mismatched restated value is refused, never silently written as a new, unrelated fact wearing a supersession pointer.
- `principal_key_fingerprint text` — mandatory on `principal_key_bound` (identity field, mandatory active or not), shape CHECK `~ '^[0-9A-F]{40}$'` (an OpenPGP v4 fingerprint, uppercase hex, the form [MAINT-GPG-TRUST-LAYER.md](MAINT-GPG-TRUST-LAYER.md) §5's "fingerprints committed per principal" implies; a longer future-format digest is refused like any malformed witness, named not silently coerced — the s38 SHA-256 posture: s38's own commit-ref regex admits only a SHA-1-length hex digest, and names a future longer SHA-256 digest out of scope for v1, refused as malformed rather than silently truncated or coerced). Forbidden elsewhere. **Creation:** `led principal bind-key <name> --fingerprint "<fp>"` — writes a fresh `principal_key_bound` row, `active = true`; refused if an active binding already exists for that exact `principal_subject`/fingerprint pair (same duplicate-active guard as `bind-role`); refused unless the subject's `agent_class = 'human'` (D-3). **Revoke:** `led principal revoke-key <name> --fingerprint "<fp>" --supersedes <old-id>` — same CLI-refusal shape as `release-role`: refused if `<old-id>` is not currently present in `principal_keys` (D-5) for that exact `principal_subject`/fingerprint pair, or if the restated fingerprint does not match the target's.

**D-3. Structural (kernel-level) refusals** (scoped to two of the four new kinds — `principal_relation_asserted` and `principal_key_bound`): self-edges refused for all four relation values, BEFORE INSERT trigger (a principal cannot act-for, be dispatched-by, be the same natural person as, or succeed itself); both endpoints must be registered (FKs already force existence; the trigger adds nothing further — stated, not silently relied on); `principal_key_bound` refused unless the subject's `agent_class = 'human'` — **agent keys stay refused**, the GPG spec's §6 carried into the type. **A fourth, plain CHECK (not a trigger — a same-row constraint needs no cross-table lookup): `principal_relation IS DISTINCT FROM 'same-natural-person' OR principal_subject < principal_object`.** This is D-2's canonicalization requirement made kernel-enforced rather than a CLI convention (a fixpoint-review finding, 2026-07-17: the CLI-only version left a direct-`psql` write able to construct a non-canonical `same-natural-person` row that a subsequent canonical-order `relate` would not detect as a duplicate, silently defeating the guarantee the whole redesign exists to deliver — closed here at the root, the same way the DDL contradiction and the withdrawal gap were, rather than merely disclosed). Principal `id` is an immutable bigint assigned once at registration (§2, "Identity"; never reused, never renumbered), so "lower id" is a stable, well-defined ordering for the lifetime of both principals — this CHECK cannot flip its own verdict later. **Value-continuity on release/revoke/withdrawal is CLI-side, not kernel-side, for all four kinds** (the same trust-boundary posture §8 already discloses for direct-`psql` writers generally): `release-role`/`revoke-key`/`withdraw-competence` each refuse a mismatched or non-active target through the sanctioned verb (D-2, D-1a), but nothing in the kernel's own CHECKs enforces that a superseding row's restated identity value matches its target's — a direct-`psql` write bypassing the CLI could construct a mismatched pair. Named here as an accepted limit, not a silent gap; see §8.

**D-4. The crypto reconciliation, carried in this spec's text so no future auditor re-litigates it** (consultation §1.5, adopted verbatim in substance): the standing deferral (*"key generation/signing deferred until all else banked; never re-raise as recommendation"*) governs the operational ceremony — generating keys, signing rituals, tokens. The maintainer's grounding injection names cryptographic authentication a first-class concern *for the principal-surface design*. Derived views** (`security_invoker`, SELECT to `:role`; all four filter `unsuperseded AND principal_binding_active` — the withdrawal fix means "unsuperseded" alone is no longer sufficient, since the terminal row of a retraction chain is itself unsuperseded but `active = false` and must not read as a live binding): `principal_relations` (subject, relation, object, asserted_by, at), `principal_role_bindings` (subject, role_name, bound_by, at — deliberately NOT named `principal_roles`: the singular `principal_role` view from §3.5 is the structurally unrelated db-role↔principal binding, and a near-identical plural would invite conflation in queries and future migrations; the name states the different fact it derives), `principal_keys` (subject, fingerprint, bound_by, at), and `principal_competences` (subject, activity, band, basis, granted_by, at — D-1a). ## 8. Honest limits

- Trigger/CHECK refusals bind the granted role's ordinary INSERT path only; a schema-owner/superuser can bypass — the same disclosed bound s26–s38 carry.
- Standing is checked **at write time**; history under a later-revoked principal stands (correct for a record; named so nobody reads revocation as retroactive doubt).
- Reinstatement of a suspension or revocation has no verb in v1 (§3.4) — a suspended-or-revoked principal has no path back to active standing except a fresh successor principal via `succeeds`. This is a real, deliberate limit, not an oversight: same-kind supersession cannot mean "lift" without inventing new semantics, and doing so under review pressure risked introducing exactly the kind of underspecified mechanism this document has already had to correct twice this pass.
- A declared standing default still authenticates nothing: the declaration says *who the connection speaks for*, not *who is at the keyboard*. The stamp remains a tripwire; the key slot remains empty until the deferred ceremony. Strict attribution is honest bookkeeping, not authentication — stated so the spec never overclaims IA-2 (NIST's identification-and-authentication-of-organizational-users control).
- Purpose text quality is unenforceable beyond non-emptiness; the countersign path is the control.
- In a solo world, every governance event is written by the one operator's principals; the record is complete and attributed, not adversarially independent — the same honesty s17 already institutionalized.
- Value-continuity on release/revoke/withdrawal (D-3) is enforced by the sanctioned CLI verbs, not by a kernel CHECK: a direct-`psql` write bypassing `release-role`/`revoke-key`/`withdraw-competence` could construct a superseding row whose restated identity value does not match its target's, or that targets an already-inactive row. Same accepted boundary as every other direct-writer honesty gap disclosed above — not new to this delta, but worth naming explicitly here since it is the one place D-1's general "identity field, mandatory always" rule has no kernel-level cross-row check behind it. (A related-sounding but distinct invariant — `same-natural-person`'s canonical ordering, D-2 — is NOT in this category: that one is a same-row property with no cross-table lookup, so D-3 enforces it with a plain kernel CHECK rather than accepting it as a CLI-only convention. This bullet is about cross-row value-matching specifically, which genuinely does need a lookup the kernel's own CHECK machinery cannot express.)

**The mechanical problem, real and unavoidable:** a refusal fires from inside `RAISE EXCEPTION`, which aborts the whole transaction. A `SAVEPOINT` does not survive this, so the meta-event cannot be written inside the failing transaction at all — it needs a genuinely separate one, and Postgres has no native autonomous-transaction feature.

**Two candidates were tried and both failed adversarial review.** **(B)**, CLI-side logging (`led.tmpl`'s error handler opens a fresh connection after catching the refusal), was rejected (ledger row 1413) because the RECORDING is then contingent on the CLI being the tool that issued the write, while the REFUSAL itself is kernel-enforced regardless of tool. An adversary probing the system need only avoid `led` to be refused exactly as before while leaving zero trace — a strictly better position for an attacker than for an honest operator, on precisely the axis this feature exists to cover. This is not the same trust boundary as §8's already-disclosed direct-`psql`-writer honesty boundary: dishonest data THROUGH an accepted write is a different threat model than an unrecorded refused attempt.

**(A)**, a kernel-side `dblink` loopback autonomous transaction inside the trigger, was first recommended at row 1412 with a hardening requirement (catch the `dblink` failure, append it to the re-raised exception) meant to close the objection that the loopback could itself fail silently. That hardening prevents silence but does not close the actual gap. This exact mechanism, for a near-identical refusal-journal design, was already proposed and adversarially refuted once before in this project's own history — at the design stage, before it was built. (Precision matters here: the citation is a design-panel refutation of a not-yet-built proposal, not a report of a system that shipped and then failed; the refutation's own acceptance-test language is entirely predictive — "increment 3's own acceptance test... produces an empty journal" describes what WOULD happen if built, and the referenced implementation path does not exist in this repository, consistent with a design refuted before construction.) The refutation is `judgment/engine/engine-panel/refute-architecture.md` flaw 1, titled "The refusal journal cannot be built as stated" (a prior project generation; the refuter is the same model class as this spec's own author): *"a dblink/postgres_fdw loopback is an undeclared new dependency, a connection per refusal on the subject's latency-budgeted path, and needs its own failure semantics (journal write fails → refuse anyway? then completeness is gone again)."*

A second, independent fresh-context review of THIS spec re-derived the same three objections from first principles without having read that document, plus a fourth: the hardening's own honest limit is that, on double failure, the refusal-record degrades to exactly today's "loud once to stderr, then gone" — better than silent, but not the durable queryable record (e) exists to provide. (The earlier claim that hardened (A) "has no silent-failure surface either" is corrected here as an overclaim.) Two independent refutations of the same mechanism, sessions and project-generations apart, is real signal, not noise: **(A) is retracted as a recommendation, not merely unhardened.**
