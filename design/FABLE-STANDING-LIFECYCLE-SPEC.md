# FABLE-STANDING-LIFECYCLE-SPEC — db_role unbind + suspension lift as one kernel delta (working name s45-standing-lifecycle)

**Status: RATIFIED build basis at birth (ledger row 1481, maintainer batch ratification 2026-07-18), Fable-authored, the conversion of the attested design notes' §1.3/§2.3 into a build-grade kernel delta spec.** Sonnet builds (row 1481's build split: "Sonnet builds every component post-conversion"); nothing here is applied to any existing [world](../GLOSSARY.md#world) (runs-are-strictly-linear, maintainer ruling 2026-07-11) — the delta reaches reality only by entering a future world's [birth chain](../GLOSSARY.md#birth-chain). This document is the Fable-authored, maintainer-ratified spec the [CLAUDE.md](../CLAUDE.md) ORCHESTRATION contract requires for kernel-lineage work; where it fixes a choice, the builder does not re-open it (§9).

**What this document is, in plain words.** Two small governance mechanisms the kernel lacks, ratified together because they are one mechanism twice: (1) **unbind** — a database connection role that has a standing attribution default ("anonymous writes on this connection count as principal X") can be repointed forever but never switched OFF; this spec gives it a sanctioned, attributed way back to *undeclared*, after which anonymous writes on that role refuse again. (2) **lift** — a suspended principal today has no path back except registering a whole successor identity, which makes suspension a trap rather than a precaution; this spec makes suspension genuinely temporary (a recorded, attributed lift) while making revocation's permanence *enforced by the kernel*, not merely unbuilt. Both ride the identity/value-split retraction idiom [s41](../kernel/lineage/s41-principal-bindings-and-relations.sql) already established (a superseding same-kind row with `principal_binding_active = false`), licensed onto the standing-lifecycle kinds that missed it. The conversion also found and closes one genuine hole the source notes missed: under uniform supersession, a *cross-kind* supersession of a revocation (or suspension, or declaration) could silently lift or resurrect standing with no typed act at all — §3.4 closes it at the kernel.

## 0. Executive summary (ADR-0017; compresses §1–§6, adds no new content)

**s45, named plainly.** One kernel delta, `s45-standing-lifecycle`, four re-issues and zero new objects of vocabulary: no new kind, no new column, no new table. It (1) licenses the existing `principal_binding_active` boolean on two more kinds (`principal_standing_declared`, `principal_suspended` — deliberately NOT `principal_revoked`), so an unbind and a lift are representable as the same superseding-inactive-row shape every s41 binding already uses; (2) re-issues the `kernel.principal_role` view with resurrection-proof governing-row semantics, so an unbind yields *undeclared* and can never silently re-bind the role to a stale older declaration; (3) re-issues the two standing functions with the in-force filter without which every lifted suspension would still read as suspended; (4) re-issues s43's `validate_supersession_target` trigger to give the three standing-lifecycle kinds a kernel-enforced supersession discipline (same-kind only, identity-continuous), which is what makes "revocation terminal by type" actually true — the conversion found that without it, any writer could lift a revocation by superseding it with an unrelated row. Two new CLI verbs (`undeclare-standing`, `lift-suspension`), one tightened one (`suspend` gains a duplicate-active guard), and the scaffold's birth declarations gain the now-mandatory `principal_binding_active = true`. Because this delta adds no ledger column, s42's serializer law does not fire and `compute_row_hash` is untouched — [gates/hash_coverage_gate.py](../gates/hash_coverage_gate.py) stays green with no re-issue, witnessed, not assumed (§3.7).

**Under s43's typed-verdict boundary, every refusal this delta minted in its source-notes form as a "RAISE" is now a returned verdict with a durable record:** a write under a suspended principal, an anonymous write under an undeclared role, a cross-kind supersession of a lifecycle row — each is caught by the boundary functions, journaled as a committed `write_refused` row, and returned as a typed refusal verdict, never a bare abort (§3.6). The source notes predate s43; this spec re-specifies every refusal surface in the s43 idiom.

**Packaging: ONE delta, confirmed** (the source notes' §4 ground, strengthened by conversion): unbind and lift are literally one edit to one CHECK plus shared machinery — splitting them would force the second delta to re-issue the same CHECK, the same trigger, and the same functions the first just issued, a pure double-touch with no independent revert value (unlike s40/s41 and s42/s43, whose halves each carried distinct object families).

**No reserved decisions remain.** Row 1481 resolved every question the source notes deferred that this delta touches; the forks the conversion itself discovered are decided in this spec and flagged as such (§3.4's supersession discipline is the significant one), per ADR-0013's surface-at-discovery duty — decided here, visibly, not silently absorbed and not left open.

## 1. The defect classes, named (ADR-0000 Rule 2)

**(a) The type question — two instances of one class.** The class: *a governance state with a sanctioned way in and no sanctioned way out, so every exit is either impossible or a kludge that re-mints the disease the state exists to manage.* Instance 1 (unbind): a db_role's standing declaration is repointable forever, never removable — the exits are suspending the bound principal (collateral: blocks that identity on *every* channel) or parking the role on a fabricated tombstone principal (a misattribution window — anonymous writes briefly attributed to a fake identity, the family's core disease re-minted). Instance 2 (lift): suspension is enterable and not exitable, so it degenerates into a soft revocation and stops being used as the routine reversible precaution it names; the suspended/revoked vocabulary promises a temporary/permanent distinction the machinery does not honor — a lying vocabulary (ADR-0012 P8's disease one level up). The foreclosing type is the one the kernel already owns: **the s41 identity/value-split retraction** (withdrawal = superseding same-kind row, `active = false`, identity fields restated), licensed onto the kinds that missed it — plus, discovered in conversion, **a supersession-discipline type for the lifecycle kinds** (§3.4), without which "terminal by type" and "resurrection-proof" are both defeasible by a bare cross-kind supersession.

**(b) The operational lapse.** Executive-side: s40/s41 shipped the retraction idiom for the four binding kinds and honestly named its absence on the standing kinds (basis C8, §3.4/§9(b)) rather than solving it under review pressure — the right call then, converted now under an explicit ratification. The conversion-found supersession hole (§3.4) is the same lapse class ADR-0000's closure-statement amendment names: the source notes' closure universe enumerated same-kind supersession semantics and did not check the *cross-kind* axis of the same surface. Named here, closed here, and the notes document is NOT retro-edited (ADR-0005 Rule 8; it is pinned and attested — this spec is its correction of record).

**Confidence: high** on both instances (ratified as Q1/Q2 on exactly this diagnosis); **high** on the supersession hole (verified directly against s43's built `validate_supersession_target`, which guards only `write_refused` targets, and s40's built `principal_standing`, which reads `ledger_current` — a superseded revocation vanishes from it).

## 3. The delta, per mechanism (all choices fixed; the builder does not fork)

### 3.1 E1 — `principal_binding_active` licensed on the two liftable kinds (the one vocabulary edit)

Re-issue `principal_binding_active_kind_shape` (its one home; DROP+ADD, never a second patching constraint — the `principal_subject_kind_shape` precedent, s40→s41), widened from four kinds to six:

```sql
ALTER TABLE :"schema".ledger DROP CONSTRAINT IF EXISTS principal_binding_active_kind_shape;
ALTER TABLE :"schema".ledger ADD CONSTRAINT principal_binding_active_kind_shape CHECK (
    (kind IN ('principal_relation_asserted','principal_role_bound','principal_key_bound',
              'principal_competence_granted',
              'principal_standing_declared','principal_suspended'))
    = (principal_binding_active IS NOT NULL));
```

**`principal_revoked` is deliberately absent** — that absence IS the ratified "terminal by type": a lift-shaped revocation row (`active = false`) is refused by this CHECK at construction, not by policy. The existing kind-free CHECK `principal_binding_inactive_needs_supersedes` (`active IS NULL OR active OR supersedes IS NOT NULL`, s41 as built) covers the two new members with zero edit: an inactive-from-birth declaration or suspension is unrepresentable. The two-way shape means every declaration and every suspension row in an s45 world carries the flag from the world's first write — there is no legacy-NULL population, because this delta reaches only birth chains (no data rows exist between deltas in a chain). Semantics of the flag on the two kinds, fixed:

- **Declaration** (`principal_standing_declared`): `true` = an in-force-candidate binding (fresh declaration or rotation); `false` = an **unbind** — a superseding row restating BOTH identity fields (`principal_db_role` AND `principal_subject`, both already mandatory two-way on the kind per s40; a retraction that doesn't say what it retracts isn't a retraction).
- **Suspension** (`principal_suspended`): `true` = an in-force-candidate suspension (fresh, or a rationale-correcting re-suspend that supersedes its predecessor); `false` = a **lift** — a superseding row restating the identity field (`principal_subject`). The lift row carries its own `actor`, `ts`, `statement` rationale: who lifted, when, why — the attributed governance record a bare deletion could never carry.

Writers always supply the flag explicitly in the payload/VALUES list; nothing defaults it (the s30 no-column-DEFAULT lesson; the column already has no default). The [kind-shape manifest](../gates/kind_shape_manifest_gate.py)'s `principal_binding_active` row is updated from four kinds to six in this same commit, and the gate's CHAIN gains s45.

### 3.2 E2 — `kernel.principal_role` re-issued with resurrection-proof governing-row semantics

The trap, restated once so the builder feels it: the s40 view picks the max-id unsuperseded declaration per db_role. Non-superseding declarations are legal (rotation "optionally supersedes" — s40's own COMMENT; and any direct `kernel.ledger_write` caller can omit `supersedes`), so older unsuperseded declarations can coexist under one role. If the view naively filtered `active` rows *inside* its max-id selection, an unbind of the newest declaration would make the next-oldest one the maximum — silently re-binding the role to a principal nobody chose. The ratified semantics (Q1's own words: "resurrection-proof view semantics") are therefore: **select the governing row per db_role as the latest unsuperseded declaration REGARDLESS of its active flag; emit it only if it is active.** The re-issued body, in full (owner-rights preserved — no `security_invoker` clause; basis C4's finding-45 argument is untouched by this edit):

```sql
CREATE OR REPLACE VIEW :"kern".principal_role AS
SELECT lc.principal_db_role AS db_role, lc.principal_subject AS principal_id
FROM   :"schema".ledger_current lc
WHERE  lc.kind = 'principal_standing_declared'
  AND  lc.id = (SELECT max(lc2.id) FROM :"schema".ledger_current lc2
                WHERE lc2.kind = 'principal_standing_declared'
                  AND lc2.principal_db_role = lc.principal_db_role)
  AND  lc.principal_binding_active;
```

The ONLY change against the s40/s43-head body is the final conjunct; the `max(lc2.id)` subquery is byte-identical and MUST NOT gain an active filter — that is the trap. Re-bind after unbind works by construction: a later fresh declaration (higher id, `active = true`) becomes the governing row. `GRANT SELECT ON :"kern".principal_role TO :"role"` is re-stated after the re-issue (belt-and-braces; CREATE OR REPLACE VIEW preserves grants, stated so the builder doesn't "fix" the redundancy away or wonder at it). The re-issued COMMENT ON VIEW names the s45 semantics and this spec. Two transparent consumers, named: `set_actor` reads this view by name on every default-resolved write (no edit needed for the unbind to take effect — §3.5's edit is teach-text only), and s43's `journal_write_refusal` resolves the attempted-actor default through it (an unbound role now yields a NULL `refusal_attempted_actor` with the role still recorded in `refusal_attempted_role` — correct, and witnessed in §6).

**Cross-reference, per the ratified I4:** this structural foreclosure and the defeasibility envelope's §6 retraction-resurrection discipline ([design/FABLE-DEFEASIBILITY-ENVELOPE-2026-07-18.md](FABLE-DEFEASIBILITY-ENVELOPE-2026-07-18.md), I4) are the same discipline independently derived — the binding view forecloses resurrection *structurally* because a resurrected stale binding is never intended, where the envelope's attestation-lapse case can only *surface* it because lapse-resurrection is sometimes the semantics. Recorded so neither is later "unified" into the weaker form.

### 3.3 E3 — the standing functions gain the in-force filter (correctness-critical, and the naive form is a trap)

Without this edit the lift is *worse than unbuilt*: a lift row is itself kind `principal_suspended` and unsuperseded (it is the terminal row of its chain), so s40's `principal_standing` — whose suspended leg tests bare kind-existence in `ledger_current` — would read every LIFTED suspension as still suspended, forever. Two functions, two precise edits against the s40 bodies as built (both keep `SECURITY DEFINER`, `STABLE`, the search_path clause, the REVOKE/GRANT pair, and every other line byte-identical):

1. **`kernel.principal_standing(pid)`** — the suspended-leg `EXISTS` (anchor: `e.kind = 'principal_suspended' AND e.principal_subject = pid`) gains one conjunct: `AND e.principal_binding_active`. The revoked leg is UNTOUCHED — revocation rows never carry the flag (§3.1's CHECK forbids it), and the revoked-dominates precedence is unchanged: an in-force revocation still dominates everything, and lifting a suspension under an active revocation is legal to write and changes nothing observable (the exact symmetry the basis §3.4 established for the reverse ordering). The registration and `'unregistered-legacy'` legs are untouched.
2. **`kernel.principal_standing_basis(pid)`** — the single-query form is the fork trap the source notes' conversion pinned: a bare `AND e.principal_binding_active` would DROP every revocation from the result (revocations carry NULL). The `WHERE` gains the kind-aware conjunct instead: `AND (e.kind = 'principal_revoked' OR e.principal_binding_active)`. Ordering (`revoked` preferred, then latest) unchanged.

The re-issued COMMENTs name the s45 filter. `principal_standing_current` (the human/SPA view) calls `principal_standing` and updates transparently — named, no edit.

**The decided rule the ratification attached here (I5, ratified YES in row 1481), stated where a future reader of the lift will wonder:** *lifecycle standing never conditions defeat force.* Suspending a principal gates its future WRITES; it supersedes nothing and withdraws nothing, so its past attestations and any trust grant empowering it continue to operate in the defeasibility layer, and a lift changes nothing there — the sanctioned levers over defeat force are the grant and the attestation, never standing. That rule binds the future defeat spec, not this delta's code; it is restated here because "why is the suspended principal's work still credited?" must have its one-line answer ON the mechanism that provokes the question (the ratified I6 naming duty).

### 3.4 E4 — supersession discipline for the lifecycle kinds: `validate_supersession_target` re-issued (the conversion-found closure; kernel-level, decided here)

**The hole, zero-context:** retraction in this kernel is uniformly "write a row whose `supersedes` names the target" (s31), and until s43 nothing constrained what KIND of row may supersede what. s43's `validate_supersession_target` added the first target-kind rule (`write_refused` targets refused). But the standing machinery derives everything from `ledger_current` — so a `note` row superseding a revocation makes the revocation vanish from the derived standing, silently reinstating a revoked principal with no typed act; the same move on the governing declaration resurrects a stale older one past §3.2's view (the view is resurrection-proof against *unbind*, but a cross-kind supersession removes the governing row from `ledger_current` entirely, changing which row IS the max); the same move on a suspension is a lift that bypasses the attributed lift row. The source notes missed this axis; ratified "terminal by type" is not deliverable without closing it. **Decision (fixed here, flagged as the conversion's one significant addition beyond the notes):** the three standing-lifecycle kinds get a kernel-enforced supersession discipline, carried by re-issuing s43's `validate_supersession_target` — the trigger that already performs a row-addressed read of every supersession target — rather than minting a second trigger doing the same read (ADR-0012 P1).

The re-issued body: the s43 `write_refused` refusal stays byte-identical and first; after it, one new block. Let `t` be the target row (the existing `SELECT` widens from `l.kind` alone to `l.kind, l.principal_db_role, l.principal_subject` — same row-addressed read, three columns):

- If `t.kind IN ('principal_standing_declared','principal_suspended','principal_revoked')` and `NEW.kind IS DISTINCT FROM t.kind` → **refuse**, teach-text: a standing-lifecycle row is superseded only by its own kind — rotation/re-declaration or unbind for declarations (`./led principal declare-standing`, `./led principal undeclare-standing`), re-suspend-correction or lift for suspensions (`./led principal suspend --supersedes`, `./led principal lift-suspension`), re-revoke-correction for revocations; a cross-kind supersession would silently alter derived standing with no typed act (this spec, §3.4).
- If `t.kind = 'principal_standing_declared'` (and kinds match): `NEW.principal_db_role` must equal `t.principal_db_role` (refuse on mismatch — a rotation or unbind restates the role it governs); and if `NEW.principal_binding_active = false`, `NEW.principal_subject` must equal `t.principal_subject` (an unbind restates the binding it retracts; a rotation — `active = true` — may repoint the subject by design).
- If `t.kind IN ('principal_suspended','principal_revoked')` (and kinds match): `NEW.principal_subject` must equal `t.principal_subject` (a lift or a rationale correction regards the same principal, always).

What this deliberately does NOT cover, named: `principal_registered` targets (superseding a registration → s40's disclosed `'unregistered-legacy'` corner) stay outside the discipline — registration retraction is a real open question row 1481 did not ratify, adjacent to succession semantics, and is left exactly where s40's LIMITS left it (§7 item 3). The four s41 binding kinds also stay outside (their value-continuity remains CLI-side per s41's own disclosed limit — retrofitting them is a scope expansion no ruling ordered; ADR-0004's no-sweep posture; §7 item 4). The trigger's name, timing, and position are unchanged (`validate_supersession_target`, BEFORE INSERT, still sorts before `validate_work_item` and `zz_set_row_hash`); [gates/ledger_reader_allowlist.py](../gates/ledger_reader_allowlist.py) already carries this function's row-addressed raw-ledger entry — the re-issue widens the columns read, not the read shape; the gate's CHAIN gains s45 and the builder witnesses it green rather than asserting.

**Consequence, stated for the closure statement's honesty:** with §3.1 (no flag on revocations) plus this discipline (revocations superseded only by revocations, same subject), a revoked principal's standing can never leave `'revoked'` through any granted-role write path: the lift row is unrepresentable, the cross-kind escape is refused, and a same-kind correction preserves the revoked reading. Succession (s41's `succeeds` relation + a fresh principal) is, as ratified, the only path — now as a typed fact.

### 3.6 E6 — the refusal surfaces, re-specified in the s43 idiom (this section supersedes the source notes' refusal prose wholesale)

The source notes predate s43 and described refusals as bare `RAISE` aborts. In an s45 kernel every granted-role write arrives through s43's boundary functions, so **every refusal this delta touches is a caught, journaled, returned verdict**: the trigger/CHECK still raises (SQLSTATE P0001 for the taught refusals, 23514 for the CHECK shapes), the boundary catches it (journaled classes 22/23/P0), commits a `write_refused` row (surface `'ledger'` for all of this delta's paths; attributed to the `write-boundary` tool principal, with the attempted actor/role and the teach-text verbatim in `refusal_message`), and returns `('refused', NULL, refusal_id, sqlstate, message)`. Concretely, the standing-lifecycle refusal inventory an operator can now AUDIT rather than lose to an abort: anonymous writes under an undeclared/unbound role; writes under suspended or revoked principals; cross-kind or continuity-violating supersessions of lifecycle rows; inactive-from-birth rows; flag-on-wrong-kind shapes. Two compositions worth one sentence each: an attempt to *hide* a lifecycle row by illegal supersession is itself journaled (the refusal of the hide becomes a record — the same property s43 built for `write_refused` targets); and because §3.2's view governs `journal_write_refusal`'s default-actor lookup, refusals under an unbound role journal with `refusal_attempted_actor` NULL and the role recorded — correct, not a gap. CLI-side pre-checks (§3.8) remain ordinary stderr refusals that never reach the kernel — they are ergonomics, not the enforcement surface, and the kernel behind them refuses independently (witnessed both ways, §6).

### 3.8 E8 — CLI verbs ([bootstrap/templates/led.tmpl](../bootstrap/templates/led.tmpl), same commit; live-verbs doctrine throughout)

On a pre-s45 world: the two new verbs refuse with teach-text ("this world's kernel predates s45-standing-lifecycle; standing declarations there cannot be retracted / suspensions cannot be lifted"), and the existing verbs' payloads stay byte-identical (supplying the flag on an s40-kind row in an s41..s44 world would violate the OLD CHECK — the probe is what keeps old worlds' verbs honest).

1. **`led principal declare-standing <name> [--db-role <role>]`** — one change on s45 worlds: the payload gains `principal_binding_active true`. The existing rotation logic (auto-resolve and supersede the latest in-force declaration for the role) stands; its `old_id` lookup adds `AND lc.principal_binding_active` so a rotation never targets an unbind row (superseding a terminal unbind row is refused by §3.4's kind-match — kinds match, but db_role continuity holds and `active = true` needs no subject match, so a *re-declaration superseding the unbind row* is actually legal and harmless; the CLI targets the in-force row when one exists and omits `supersedes` after an unbind — both paths legal, the view's max-id semantics correct either way; the builder implements the lookup as specified and does not fork on this).
2. **`led principal undeclare-standing [--db-role <role>]`** (NEW; role defaults to the world's `$ROLE`, same as declare-standing) — resolves the governing declaration for the role via the same `ledger_current` lookup declare-standing uses, additionally requiring it active; if none: refuse, teach-text "role '<role>' has no in-force standing declaration — nothing to unbind." Payload: `kind principal_standing_declared`, `principal_binding_active false`, `supersedes <governing-id>`, `principal_db_role` and `principal_subject` restated from the target row (the CLI reads them from the target — the operator never retypes them, so CLI-side mismatch is impossible by construction and §3.4's kernel continuity check is the backstop for non-CLI callers), `statement` naming the unbind and the outgoing principal. The unbind may be written as a NULL-actor write under the very role being unbound — `set_actor` resolves BEFORE INSERT while the declaration is still governing, so the unbind is attributed to the outgoing principal: its last act under the role is relinquishing it (fixed in the source notes, unchanged by s43 — stated so the builder doesn't "fix" it). **Operational guidance printed on success (the ratified I3 naming duty, verbatim in substance):** the unbind is forward-only — rows already written under the old declaration stay attributed to the outgoing principal forever; if the reason for unbinding is that past rows were MISattributed, the sanctioned backward-looking correction is the defeasibility layer (a sentry/attestation defeat discounting those rows in derived views), never retro-editing attribution.
3. **`led principal suspend <name> [reason...] [--supersedes <id>]`** — gains the duplicate-active guard every s41 bind verb carries: refuse if an in-force suspension (unsuperseded, `active = true`) already exists for the subject, unless `--supersedes` names exactly that row (a rationale-correcting re-suspend — payload then carries `active true`, `supersedes <id>`; §3.4 enforces subject continuity). Payload gains `principal_binding_active true` on s45 worlds. The `write-boundary` CLI guard stands untouched. `revoke` is UNCHANGED in every respect (no flag, no guard change — its terminality machinery is kernel-side).
4. **`led principal lift-suspension <name>`** (NEW) — resolves the in-force suspension for the subject (`ledger_current`, kind suspended, subject, `active`); refusals with teach-text: not suspended ("principal '<name>' has no in-force suspension"); suspended AND revoked (the lift is legal and this verb still writes it, but it prints the revoked-dominates warning: standing will remain `'revoked'`; witnessed in §6). Payload: `kind principal_suspended`, `principal_binding_active false`, `supersedes <in-force-id>`, `principal_subject` restated from the target, `statement` = the lift rationale (`[reason...]` args, non-empty encouraged, not enforced). The lift requires another active principal's write (a suspended principal's writes refuse) — the verb does not special-case this; the kernel refusal (now a journaled verdict) is the teacher. One narrowing of an s43 dead-end, named: a `write-boundary` principal suspended by a direct (non-CLI) write previously required schema-owner repair; `lift-suspension` written by any active principal now repairs it through the sanctioned surface (an ACCEPTED write needs no working journaler).
5. **Multi-suspension residue, disclosed:** the duplicate-active guard is CLI-side (a scan-shaped cross-row check the kernel CHECK machinery cannot express and this spec declines to put in a trigger — the row-addressed §3.4 read is cheap; a per-write scan is not). A direct boundary caller can still stack in-force suspensions; each then needs its own lift; standing reads suspended until all are lifted (fail-safe polarity). Same accepted boundary as s41's value-continuity disclosure, inherited and named (§8).

## 4. Closure statement (ADR-0000 Rule 2(a), 2026-07-02 form)

**Invariant:** in every world born under s45, a db_role's standing default and a principal's suspension are withdrawable only by an attributed, dated, superseding same-kind event restating the identity it retracts; the current binding and the current standing are derived reads that treat a retraction chain's terminal inactive row as absence, never as force, and can never silently promote a stale older row; a revocation admits no lift row (unrepresentable by CHECK) and no supersession except an identity-continuous same-kind correction, so a revoked principal's derived standing is `'revoked'` under every granted-role write path until succession; and every refusal enforcing the above is a committed, journaled `write_refused` row returned as a typed verdict, never an unrecorded abort.

**Quantification universe** (checked outward, the presumption-of-narrowness inverted):

- **Kinds × the active flag:** exactly six kinds carry it (the four s41 kinds + the two licensed here), enumerated in the one re-issued CHECK; `principal_revoked` and every other kind excluded by the same two-way test. No second constraint, no column, no kind minted.
- **The supersession axis, both directions** (the axis the source notes missed, now enumerated): *as target*, the three lifecycle kinds accept only same-kind identity-continuous supersessors (§3.4), `write_refused` stays unretractable (s43, byte-identical), `principal_registered` and the four s41 binding kinds are named NOT covered (§7 items 3/4); *as supersessor*, an inactive row requires a target (the s41 CHECK), and rotation/unbind/lift/correction shapes are each enumerated in §3.4's rules with their identity-match obligations.
- **Derived readers of standing/bindings, disposed one by one:** `principal_role` (re-issued, §3.2); `principal_standing`/`principal_standing_basis` (re-issued, §3.3); `principal_standing_current` (transparent via the function — named); `set_actor` (reads the view + function — teach-text re-issue only, §3.5); `journal_write_refusal`'s attempted-actor default (reads the view — transparent, witnessed); the D-5 binding views and `principal_competences` (already active-filtered by s41 — untouched); SPA/`pickup`/display surfaces (read the views — display-only, named).
- **Write surfaces:** all granted-role writes arrive via the s43 boundary (verdicts + journal — §3.6); CLI verbs enumerated §3.8; scaffold birth acts §3.9; direct owner/superuser DML — the standing disclosed bound (§8).
- **Gates/engine/hash:** §3.7/§3.10's enumeration — zero columns (gate-witnessed), zero kinds, zero `.lp` predicates, the I2 ASP debt filed on the envelope successor by name.

**Denomination:** standing and bindings in dated, attributed lifecycle EVENTS, computed at read, never stored; in-force-ness in the two-conjunct governing-AND-active test (the I1 notion, SQL home = the re-issued view/functions); terminality in a CHECK's kind-set membership plus a supersession-discipline trigger — never in CLI convention or prose; continuity in immutable principal ids and the target row's own restated columns, never names. No bound is a bare literal.

## 8. Honest limits

- Trigger/CHECK refusals bind the boundary-function path; the schema owner/superuser bypass stands (s26..s43's disclosure) — including dropping the very triggers §3.4 adds; the externally-held signed head remains the closing move.
- Standing is checked at write time only; a lifted principal's refused-era attempts exist only as `write_refused` rows (correct — there are no accepted rows to re-judge), and a suspended principal's PAST accepted rows keep full credit unless positively defeated — the coherent I6 asymmetry, named here so it is never "fixed."
- The duplicate-active suspension guard is CLI-side (§3.8 item 5); stacked direct suspensions each need their own lift, fail-safe polarity.
- The unbind is forward-only; misattributed history is the defeat layer's to discount (I3), never this delta's to rewrite.
- In a solo world whose only active principal is suspended, the lift itself cannot be written through the sanctioned surface (no active writer) — the C7-shaped dead-end, narrowed (a second active principal now suffices where before nothing did) but not closed; recovery below that line remains a schema-owner act, disclosed.
- The two-conjunct in-force notion has no ASP twin until the envelope successor pays I2; until then the judge differential's coverage of principal machinery remains what it was at s40 (kind-generic entry flow only) — witnessed in AGREE, with the deeper pairing debt filed, not hidden.
