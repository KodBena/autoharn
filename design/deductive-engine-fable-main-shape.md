# Deductive engine — Fable main-loop's own shape (committed BEFORE the design panel runs)

Blind-consult discipline (the ratified never-again protocol): this is my independent
shape, committed first; the panel's briefs are facts-only and contain neither this note
nor the maintainer's framing. Synthesis compares AFTER both exist. (Fable, session
7be3443d, 2026-07-07 afternoon.)

## What the engine IS

The live half of what the marriage built retrospectively. The marriage derives T_now +
judgments AT CLOSE over a banked record; the engine derives them DURING the run — at
write time (trigger tier) and at hook/orchestration time (consult tier) — and acts on
them under the ratified posture: measure novel classes, refuse-and-teach specimen-proven
ones. e17 is the existence proof at n=1 judgment classes (stamp-distinctness); the
engine is that pattern generalized to the whole judgment vocabulary.

## The five load-bearing commitments (my shape)

1. **Replay-driven development against adjudicated ground truth.** The e-series corpus
   (acts streams, subject ledgers, rulings, findings 1–37 WITH their dispositions) is
   LABELED DATA no one has to invent: the engine's fitness function is deriving, from
   the raw record alone, the judgments the humans/Fable actually adjudicated —
   proxy_written(12), the F52 block-holds, verdict-aware staleness, claim-inflation,
   vacuous passes — and NOT deriving the ones adjudication rejected (the struck row-7
   SoD reading). Engine development = replay + differential against the findings ledger.
   No new experiments needed to start; every future e-run grows the corpus for free.
2. **Assign-don't-compete, with the differential discipline extended.** ASP/clingo owns
   nonmonotonic T_now, defaults, obligations; SQL owns the independent floor (the
   two-producer differential is already ratified practice); SMT enters ONLY where
   arithmetic/temporal bounds genuinely demand it. Each judgment family has exactly one
   home plus one independent check — never two competing homes.
3. **The compilation seam is the research question.** Two enforcement tiers are already
   ratified (trigger-consultable compiled rows vs engine-side loud judgments). The open
   design problem is WHICH judgments compile down to write-time (bounded, row-local,
   teachable in an error message) and which stay solver-side (global, nonmonotonic,
   close/consult-time) — and the PROMOTION path between them (a solver-side judgment
   with a captured specimen compiles down; measure-once-then-mechanize as an engine
   lifecycle, not a one-off).
4. **Derivations cite law.** Every derived judgment carries the rule id / F-number it
   derives under (F52, F53, ...), so an engine verdict is an auditable citation chain,
   not an oracle pronouncement. The rules file is versioned and sha-anchored in
   acts.ruling like any law text; rule changes are rulings.
5. **The engine is itself a BRIEF subject.** I8 tool qualification applies to it
   (DerivationRecord already exists — extend: engine version + rules hash + input EDB
   hash per derivation); its own failure modes (vacuous pass on empty substrate —
   finding 36's class; stale grounding; solver timeout treated as silence) get the same
   seen-red + declared-exclusion treatment as everything else. An unqualified checker is
   an unverified verifier.

## What I deliberately do NOT commit to yet

Hook-latency numbers (measure, don't assume); incremental-ASP machinery (clingo
multi-shot vs re-ground — an experiment, not a doctrine); any expansion of the judgment
vocabulary beyond what the corpus's adjudicated findings support; online learning of any
kind (rules change by ruling, never by fit).
