# FABLE-RESERVED-DESIGNS-2026-07-18 — banked design judgments on three deferred principal-surface designs

**Status: DESIGN NOTES, non-binding, Fable-authored fresh-context 2026-07-18 — explicitly NOT a spec, explicitly awaiting a future commissioning act.** Nothing here licenses touching `kernel/lineage/`; the standing rule (nobody edits kernel lineage without a Fable-authored, maintainer-ratified spec, [CLAUDE.md](../CLAUDE.md) ORCHESTRATION) is untouched, and each design below reaches reality only as its own future spec entering a future [world](../GLOSSARY.md#world)'s [birth chain](../GLOSSARY.md#birth-chain) (runs-are-strictly-linear, maintainer ruling 2026-07-11 — no delta is ever applied to an existing world).

**What this document is, in plain words.** The principal-identity delta family s40/s41 — specified by [design/FABLE-PRINCIPAL-IDENTITY-SPEC-BUILD-BASIS.md](FABLE-PRINCIPAL-IDENTITY-SPEC-BUILD-BASIS.md) (the frozen build basis, read here in full as corrected by its amendments C1–C13) and built as [kernel/lineage/s40-principal-identity-events.sql](../kernel/lineage/s40-principal-identity-events.sql) and [kernel/lineage/s41-principal-bindings-and-relations.sql](../kernel/lineage/s41-principal-bindings-and-relations.sql) — deliberately left three small designs undone, each named honestly in the basis rather than silently absent. This document banks the design analysis for those three, so that a future spec author (or the maintainer directly) can act on them without re-deriving it: (1) a sanctioned way for a database role to return to *undeclared* standing (basis amendment C8); (2) reinstatement of a suspended or revoked principal (basis §3.4 / §9(b), ratified not-built-in-v1 with an explicit revisit condition); (3) the closed competence-band system the §9(g) free-text placeholder awaits. Each section opens with a plain-words motivation preamble (what the mechanism is, what is stuck today, and what concretely goes wrong without the design — before any apparatus vocabulary), then restates its problem from zero context, enumerates the design space, states one recommended design at builder precision, names what it deliberately does not decide, and closes with a single yes/no ratification question. A final section states how the three interact — notably that designs 1 and 2 share one mechanism, and design 3 does not.

Everything below was verified against the s40/s41 DDL as actually built (constraint names, CHECK shapes, view bodies, trigger bodies), not only against the spec's prose. Where a recommendation depends on a specific line of the built DDL, that dependency is cited.

## 1. db_role standing UNBIND — a sanctioned path back to *undeclared*

*(Plain words first, before any apparatus vocabulary — non-normative, in the [ADR-0000](../law/adr/0000-the-alpha-and-the-omega-type-driven-design.md) plain-words-preface idiom; it changes nothing below. Every world's database has connection roles — the named accounts that tools actually connect as. A **standing declaration** is the recorded rule "writes arriving on this connection count as principal X whenever no explicit author is named"; it is what lets a solo operator work all day without typing an identity on every command. Today that rule is a one-way door: once a role has ever been pointed at a principal, you can repoint it to a different principal forever, but you can never turn the default OFF — there is no way to say "this connection speaks for nobody now; go back to refusing its anonymous writes." Where that bites, concretely: decommissioning a role you no longer use or trust; tightening a world after an incident so that, on one connection, every write must name its author explicitly again; a role whose principal was just revoked and that should now refuse rather than be handed to whichever principal is convenient. The two workarounds that exist today are worse than they look — one silences the person instead of the connection (suspending the principal blocks ALL of its writes, on every channel, not just this role's default), and the other requires inventing a fake principal to park the role on, which briefly ATTRIBUTES anonymous writes to that fake identity: the precise misattribution disease this whole delta family was built to end. The closing question is worth the maintainer's minutes because the fix is small — the retraction mechanism already exists for four sibling event kinds, and this only licenses it on the fifth — while every role decommissioning until then is either collateral damage or a kludge.)*

### 1.1 The problem, from zero context

In an s40 kernel, every ledger write must be attributable to a registered principal (strict attribution, [s40](../kernel/lineage/s40-principal-identity-events.sql) Element 6). The sanctioned default is a **standing declaration**: a `principal_standing_declared` ledger event binding a database connection role (e.g. the world's `vsr_rw`) to a principal, so a write that supplies no explicit actor resolves to the declared principal. The current binding per db_role is the derived view `kernel.principal_role` — the latest unsuperseded declaration event per role.

Rotation works: a newer declaration (optionally superseding the old) moves the role to a different principal. What does **not** exist is any way to return a role to *undeclared* — the state where a NULL-actor write under that role is refused rather than attributed to anyone. The basis's amendment C8 names this honestly: `principal_standing_declared` carries no retraction shape — the four s41 binding kinds got an identity/value-split retraction mechanism (`principal_binding_active`, a superseding same-kind row with `active = false`), but that column is *forbidden* on every s40 kind by the built kind-shape CHECK (`principal_binding_active_kind_shape`, [s41](../kernel/lineage/s41-principal-bindings-and-relations.sql) D-2), and "a declaration is only ever superseded by a newer declaration."

C8 lists two workarounds. Examined here, both are genuinely insufficient as a permanent answer:

- **Suspend or revoke the bound principal.** This blocks that principal's writes *everywhere* — including via explicit `LED_ACTOR` and via any other role it is declared for. Decommissioning one role by executing its principal is collateral damage whenever the principal has any other legitimate channel; it conflates "this role no longer speaks for anyone" with "this identity no longer acts," which are different facts.
- **Supersede with a declaration binding a different principal.** This needs a principal to bind. If none is honestly right, the operator must fabricate a parking identity ("tombstone") and then suspend it so the role refuses — a two-step kludge that (a) fabricates a category (the [ADR-0008](../law/adr/0008-classification-discipline.md) failure shape), and (b) opens a misattribution window between the declaration and the suspension during which NULL-actor writes are silently attributed to the tombstone. Misattribution is the exact disease the whole s40/s41 family exists to end; a workaround that re-mints it, even briefly, is not a workaround.

So the status quo is not permanently sufficient; the unbind wants a real mechanism.

### 1.2 The design space

1. **Do nothing, permanently** — rejected above; both workarounds fail on their merits, one of them by re-minting the family's core disease.
2. **A new retraction kind** (`principal_standing_undeclared` or similar) — rejected: the family's ratified posture is one retraction mechanism, uniformly supersession ([s31](../kernel/lineage/s31-supersession-uniform-retraction.sql) idiom; s41's header: "no `*_revoked`/`*_released` kinds are minted — one retraction mechanism, one home"). A dedicated lift-kind for one binding would be a second retraction mechanism minted for the one kind that missed the first.
3. **Sanction bare/cross-kind supersession of a declaration** (supersede it with a row that "means nothing follows") — rejected: `principal_subject` and `principal_db_role` are mandatory on the kind (two-way CHECKs), so a "declaration of nothing" is unrepresentable, and cross-kind supersession as a semantic ("a `note` retracts a declaration") would be an untyped convention no CHECK enforces — exactly the load-bearing-knowledge-in-unenforceable-convention shape [ADR-0012](../law/adr/0012-compositional-and-structural-hygiene.md) cancer G forbids.
4. **Extend the s41 identity/value-split idiom to `principal_standing_declared`** — recommended, below.

### 1.3 The recommended design (builder precision)

The type-level diagnosis first ([ADR-0000](../law/adr/0000-the-alpha-and-the-omega-type-driven-design.md) Rule 2(a)): a standing declaration **is a binding** — db_role ↔ principal — that happened to be homed among s40's lifecycle kinds because strict attribution needed it in the first delta. It has exactly the shape of three of the four s41 binding kinds (`principal_role_bound`, `principal_key_bound`, `principal_relation_asserted`): all content columns are identity fields, no value fields — unlike the fourth, `principal_competence_granted`, whose identity field pairs with band/basis value fields under the same active-flag retraction idiom. The fix is therefore not a new mechanism but licensing the existing one onto the one binding kind that missed it:

1. **Re-issue `principal_binding_active_kind_shape`** ([s41](../kernel/lineage/s41-principal-bindings-and-relations.sql) line-of-record: the two-way CHECK currently licensing exactly the four s41 kinds) widened to include `principal_standing_declared` — the same one-home re-issue discipline `principal_subject_kind_shape` already follows (s40 mints it four-wide, s41 re-issues it eight-wide; never a second patching constraint). Because this reaches reality only in a future birth chain, every declaration row in any world born with the delta carries the column from the world's first write; there is no legacy-NULL population to grandfather. The existing kind-free CHECK `principal_binding_inactive_needs_supersedes` (`active IS NULL OR active OR supersedes IS NOT NULL`) then covers declarations for free: an undeclared-from-birth row is unrepresentable.
2. **Retraction shape**, exactly the s41 idiom: an unbind is a NEW `principal_standing_declared` row, `principal_binding_active = false`, `supersedes` the in-force declaration, restating both identity fields (`principal_db_role` and `principal_subject` — both remain mandatory on the kind, active or not; a retraction that doesn't say which binding it retracts isn't a retraction). A fresh declaration explicitly supplies `true` in its VALUES list; nothing defaults it (the s30 no-column-DEFAULT lesson, unchanged).
3. **Re-issue the `kernel.principal_role` view with resurrection-proof semantics.** This is the one place a naive builder forks, so it is fixed here: the view must select the **latest unsuperseded declaration per db_role regardless of its active flag, and then emit the row only if that governing row is active**. Concretely, against the built view body (s40 Element 5): the `max(lc2.id)` subquery stays exactly as built (it must NOT filter on `principal_binding_active`), and the outer WHERE gains `AND lc.principal_binding_active`. Rationale: rotation is "a newer declaration, *optionally* superseding the old" (s40's own COMMENT), so older unsuperseded declarations can exist; if the max-id computation filtered inactive rows, an unbind would resurrect the next-oldest declaration — silently re-binding the role to a principal nobody chose. With the governing-row semantics, unbind yields *undeclared*, and a later fresh declaration (higher id, active) re-binds normally. The view stays owner-rights (basis C4's finding-45 argument is unchanged by this edit).
4. **CLI verb**, joining the §3.7/D-2 family with the same channel discipline: `led principal undeclare-standing <db_role> --supersedes <old-id>` — refused if `<old-id>` is not the current in-force binding for that role in the (re-issued) `principal_role` view's underlying selection; the new row restates the target's `principal_db_role` and `principal_subject`, a mismatch refused at the CLI. Value-continuity remains CLI-side, not kernel-side — the same disclosed direct-`psql` limit s41's LIMITS block already carries for all four binding kinds; this adds nothing new to that boundary and inherits its disclosure.
5. **Self-referential write ordering, stated so the builder need not re-derive it:** the unbind row may itself be written as a NULL-actor write under the very role being unbound — `set_actor` resolves BEFORE INSERT, while the declaration is still in force, so the unbind is attributed to the outgoing principal, which is correct (the outgoing principal's last act under the role is relinquishing it). After commit, NULL-actor writes under the role refuse with s40's existing declare-standing teach-text; explicit `LED_ACTOR` writes are untouched. No interaction with the C7 total-revocation dead-end is created: unbinding never blocks explicit attribution, so a world with one active principal and zero declarations is inconvenienced, not bricked.
6. **`set_actor` needs no edit.** It reads `principal_role` by name; the view's re-issued body governs. The refusal teach-text it already emits ("declare this role's standing principal") remains correct for the undeclared state this design makes reachable.

### 1.4 Deliberately not decided here

Whether the same active-flag licensing should extend to `principal_registered` (an "unregister" event — today a superseded registration yields the odd `'unregistered-legacy'`-treated-as-active state s40's LIMITS names as "deliberately NOT specified in v1"); that is registration-lifecycle territory adjacent to §2's reinstatement question and should be decided with it, not smuggled in here. Also not decided: whether `undeclare-standing` warrants a [countersign](../GLOSSARY.md#obligation) ceremony — a second, distinct principal attesting the act, which otherwise stands as visible unreviewed debt under the kernel's review-gap machinery — (registration's optional-countersign posture suggests not, but the question is the maintainer's).

### 1.5 Ratification question

**Shall `principal_standing_declared` adopt the s41 identity/value-split retraction — `principal_binding_active` licensed on it, with the resurrection-proof `principal_role` view semantics specified in §1.3 — so a db_role has a sanctioned, attributed path back to undeclared?**

## 2. REINSTATEMENT of a suspended or revoked principal

*(Plain words first, same idiom. A principal can be **suspended** or **revoked** — recorded events after which every write it attempts is refused. What is missing is any way to say "the suspension is over": no verb and no event means "this no longer holds," so both states are forever. The only path back to a working identity is registering a brand-new principal that formally succeeds the old one — the identity equivalent of replacing an employee's badge, number, and file because their badge was temporarily deactivated. Where that bites, concretely: suspend a principal while investigating an anomaly and, once it is cleared, you cannot un-suspend — you either abandon the identity and its accumulated history behind a succession pointer, or you learn to never suspend at all because the cost is total; a precaution that cannot be undone stops being used as a precaution. And that reveals the deeper defect: today "suspended" and "revoked" behave identically — the vocabulary promises a temporary/permanent distinction the machinery does not honor, so the words themselves mislead the reader of the record. The recommendation makes suspension genuinely temporary (liftable, with an attributed who/when/why record of the lift) while making revocation's permanence a typed fact rather than a v1 gap. The closing question is worth the maintainer's minutes because it decides whether suspension is a usable operational tool or a trap — and it costs one licensing edit to the same mechanism §1 already asks about.)*

### 2.1 The problem, from zero context

s40 gives a principal three lifecycle standings beyond active: `principal_suspended` and `principal_revoked` events withdraw standing (writes under that principal refuse at write time), with a ratified precedence — revoked always dominates suspended, a severity ordering, not recency. Ratified decision §9(b): **no reinstatement mechanism exists in v1**, for either kind. The basis is honest about why this is hard rather than merely unbuilt: superseding a `principal_suspended` row with another row of the same kind reads as *re-suspend*, not *lift* — no kind exists whose write means "this no longer holds" for a lifecycle event. The only v1 path back to activity is registering a fresh successor principal and asserting a `succeeds` relation. The maintainer's ratification note keeps the question open: "on its face this is not believed necessary, but the question stays open to a future standards read."

The pressure point a future standards read will most likely find: **suspension that cannot be lifted is not suspension.** In every standards family the grounding consultation ([design/ORCH-CONSULT-PRINCIPAL-SURFACE-2026-07-18.md](../vestigial_documentation/design/ORCH-CONSULT-PRINCIPAL-SURFACE-2026-07-18.md)) rooted this arc in, suspension/disabling is definitionally temporary (NIST SP 800-53 AC-2(3) account disabling; X.509's `certificateHold` is the one revocation reason that is liftable) while revocation is terminal. s40's vocabulary carries that temporary/terminal distinction; its mechanism does not honor it — a suspended principal and a revoked one differ today only in which word the refusal prints and which event dominates. A vocabulary whose distinction the mechanism cannot honor is a lying signature at the vocabulary scale ([ADR-0012](../law/adr/0012-compositional-and-structural-hygiene.md) P8's disease, one level up). So the succession-only posture is fully defensible for *revocation* and structurally misleading for *suspension*.

### 2.2 The design space

1. **Succession-only, permanently, for both kinds** — coherent and simple, but leaves the lying vocabulary above: either suspension should then be removed/redefined as "revocation with a softer name," or the distinction should be made real. Keeping both words with identical semantics is the dishonest middle.
2. **A new lift kind** (`principal_reinstated`) whose in-force presence neutralizes prior suspensions — rejected on the same one-retraction-mechanism ground as §1.2 option 2, and it forces a second question (which suspension does a reinstatement neutralize? all? by recency?) that supersession answers structurally for free.
3. **Recency-ordered standing** (latest lifecycle event wins; add a reactivation kind) — rejected: it discards the ratified revoked-dominates severity precedence, which was minted precisely because event-recency ordering was found underspecified (the fixpoint-review finding §3.4 records), and it makes `suspend-then-revoke-then-suspend` sequences load-bearing puzzles again.
4. **Extend the s41 active-flag idiom to `principal_suspended` only; revocation stays terminal by type** — recommended, below.

### 2.3 The recommended design (builder precision)

The same mechanism as §1, applied to exactly one lifecycle kind:

1. **License `principal_binding_active` on `principal_suspended`** (the same widened re-issue of `principal_binding_active_kind_shape` as §1.3 step 1 — if both designs ship, one re-issue licenses both kinds). **Deliberately do NOT license it on `principal_revoked`.** Revocation's terminality then remains enforced by the type — a lift row for a revocation is *unrepresentable*, not policy-refused — which is the strongest surface available ([ADR-0011](../law/adr/0011-mechanization-discipline.md)'s vocabulary: construction-time, not review or CLI). Succession stays revocation's only escape, exactly as ratified, now as a typed fact rather than a v1 gap.
2. **Lift shape:** a suspension is asserted with `active = true` (explicit in the VALUES list); a lift is a NEW `principal_suspended` row, `active = false`, `supersedes` the in-force suspension, restating the identity field (`principal_subject` — already mandatory two-way on the kind). The lift row carries its own `actor`, `ts`, rationale: who lifted, when, why — attributed governance, which is precisely what a bare "delete the suspension" could never record. A re-suspend remains representable as a fresh `active = true` row (optionally superseding, for a rationale correction — the superseding-same-kind-row-as-correction idiom [s39](../kernel/lineage/s39-blocks-start.sql) established).
3. **`principal_standing` gains one predicate, `principal_standing_basis` the same.** Against the built function bodies (s40 Element 4): the suspended-leg `EXISTS` gains `AND e.principal_binding_active`; the revoked leg is untouched (revocations never carry the column — its kind-shape CHECK forbids it there). Without this filter the function is *wrong*, not merely stale: the lift row is itself kind `principal_suspended` and unsuperseded, so the unfiltered `EXISTS` would read every lifted suspension as still suspended. `principal_standing_basis`'s `WHERE` gains the same conjunct (it must name the governing *in-force* event). Precedence is unchanged: an active revocation still dominates everything; lifting a suspension under an active revocation is legal to write and changes nothing observable — the exact symmetry §3.4 already established for the reverse ordering.
4. **CLI verb:** `led principal lift-suspension <name> --supersedes <old-id>` — refused if `<old-id>` is not an in-force (unsuperseded, `active = true`) suspension of that principal; identity restated; mismatch refused. Companion tightening: `led principal suspend` gains the same duplicate-active guard every s41 bind verb carries — refused if an in-force suspension already exists for the subject unless `--supersedes` names it (a rationale update, not a stack). This matters because multiple coexisting unsuperseded suspensions are legal at the kernel today, and a lift supersedes exactly one row; the guard keeps the sanctioned surface single-suspension so "lift" has one target. A multi-suspension state constructed by direct `psql` remains the standing disclosed direct-writer limit (each stacked suspension then needs its own lift) — named, not silently absorbed.
5. **Write-time-only standing is unchanged and needs no carve-out:** history under suspension does not exist (writes were refused), and the lift restores acceptance forward-only. The one operational corner: a suspended principal cannot write its own lift (its writes refuse), so a lift requires another active principal — in a solo world whose only principal is suspended, recovery is the schema-owner act C7 already discloses. This design narrows C7's dead-end (a lift verb exists where none did) but does not close it; the last-active-principal refusal C7 considered and declined stays declined here.
6. **Teach-text update:** s40's revoked/suspended refusal in `set_actor` currently teaches only the successor path for both standings. With this design it should branch: suspended → teach `lift-suspension` (and by whom); revoked → the existing successor text unchanged.

### 2.4 Deliberately not decided here

Whether lifting a suspension should require a countersign or a second principal's attestation (governance ceremony vs. thin verb — a maintainer policy call, and the recordable-not-gating posture suggests shipping the thin verb first); whether `principal_registered` joins the licensed set to give retraction of a registration defined semantics (named in §1.4; it shares this mechanism but is a separate question with its own blast radius on `'unregistered-legacy'`); and whether the standards read the maintainer conditioned §9(b) on changes the revocation half — this design deliberately leaves revocation exactly where the ratification put it.

### 2.5 Ratification question

**Shall suspension become liftable by licensing the s41 active-flag retraction on `principal_suspended` alone (lift = superseding same-kind row with `active = false`, the standing functions gaining the in-force filter per §2.3), while `principal_revoked` remains terminal by type with succession as its only escape?**

## 3. The COMPETENCE BAND SYSTEM — the architecture the §9(g) placeholder awaits

*(Plain words first, same idiom. s41 lets the record say "principal P is believed competent for activity A, at band B, on basis C" — but band and activity are today free text: "band 3", "III", "senior", and "high" are all accepted, and none is comparable to any other. So the record can be WRITTEN but never USED: any future rule of the form "countersigning safety-critical work requires at least band X" needs bands a machine can rank — it must be able to compute that one band is at-or-above another — and free text cannot be ranked. That un-usability is exactly why the maintainer's ratification stamped the current shape a loud placeholder rather than a design. What stays annoying until this is addressed: competence grants accumulate as good-faith prose that no gate, report, or audit can ever aggregate or compare, and the longer the free-text habit runs, the more idiosyncratic the entries a deployment's operators invent. This section sketches what the placeholder awaits — where the band vocabulary should live (in the ledger, as deployment-declared, ranked, attributed events: not hardcoded into the kernel, and not imported wholesale from automotive/avionics schemes that grade the hazard of functions, not the competence of persons), how grants would validate against it, and what an eventual grading gate would consult — and shows that the feared free-text migration problem mostly evaporates under the runs-are-linear ruling. Nothing is decided: the ratified precondition (real deployment data) still governs. The closing question is worth the maintainer's minutes because it asks only whether this architecture CLASS is the direction, so the eventual spec starts from a settled shape instead of an open field.)*

### 3.1 The problem, from zero context

s41 built the G13 competence record (from the [safety-critical-logging BRIEF](../law/briefs/safety-critical-logging/BRIEF.md)'s obligation register): `principal_competence_granted` events record who is believed competent for what *activity*, at what *band*, on what *basis*, granted by whom, when — recordable, not gating. Ratified decision §9(g) made `band` and `activity` free non-empty text, with the maintainer's required loud note: **placeholder architecture only, not a considered final design** — the closed vocabulary (and which band system) is deferred until deployment conventions exist and executive bandwidth allows. This section is the sketch that deferral awaits. Per the commission: the thinking is banked; **nothing here is decided**.

### 3.2 Vocabulary shape: house scheme vs imported ASIL/SIL/DAL

The imported schemes (ISO 26262 ASIL A–D, IEC 61508 SIL 1–4, DO-178C DAL A–E) grade the *hazard severity of functions and artifacts*, not the competence of persons; personnel-competence frameworks in the same standards families (IEC 61508's competence clauses, IEC 62443's roles, SFIA-style capability levels) are process requirements, not a portable closed enumeration. Importing ASIL literally as a people-grading scale would be a category borrow that buys familiar letters and wrong semantics. The honest shape is a **small, ordered, house scheme** (three to five ranked bands), with any mapping to an external scheme recorded per-deployment as documentation, not encoded in the kernel. But which bands, how many, and what their names are is exactly the question the deferral says must wait for deployment data — this note deliberately proposes no membership.

The structural question this section CAN answer is *where the vocabulary lives*, and here two ratified precedents pull in opposite directions and must be reconciled:

- §9(c) ruled role names **free text** because role naming is organizational configuration, not a kernel-structural fact — nothing in the kernel branches on which role name a binding carries.
- But the entire point of a band system is that something eventually DOES branch on it: grading-against-bands needs bands to be **ordered and comparable** ("activity X requires band ≥ B"), and free text cannot order. The relation vocabulary shows the other pole: `principal_relation` is a closed kernel CHECK precisely because kernel refusals key off its values.

Bands sit between: deployment-owned naming (the 9(c) property) *and* kernel-computable ordering (the relation property). A hardcoded kernel CHECK gets the ordering and violates the 9(c) principle; free text gets the naming and forecloses grading. The reconciliation is the house idiom the whole s40/s41 family runs on: **the vocabulary itself becomes event-sourced data**.

### 3.3 The sketched architecture (three declared-event layers; a sketch, not a spec)

1. **Band definitions as ledger events.** A new kind (working name `competence_band_defined`) declares a band into existence per deployment: identity field the band name, value field an integer rank (the ordering), attributed and dated like every identity event, retractable by the same active-flag supersession as every s41 binding. The scaffold can seed a deployment's bands at world birth as explicit acts, exactly as it seeds principal registrations. The vocabulary is thereby closed *per deployment* — deployment-owned membership (the 9(c) principle honored), append-only history of who defined which band when (the family's record discipline), and a computable order (the grading requirement).
2. **Grants validate against in-force definitions.** `principal_competence_granted`'s band field stops being free text and must match an in-force band definition. This is a cross-row lookup, so it is a BEFORE INSERT trigger leg (the `validate_principal_binding` family), not a CHECK — the same accepted mechanism/limit split D-3 already documents. Activity plausibly gets the same treatment later (`competence_activity_defined`) but can stay free text in the first cut; band is the field grading orders on, activity is only matched by equality.
3. **Requirements as declared events, and what grading consults.** The third layer — working name `activity_band_required` (activity, required band, declared by whom) — is what future grading-against-bands would consult: at the enforcement point, join the actor's in-force grants (`principal_competences`) to the in-force requirement for the activity, compare ranks via the definitions. The natural first enforcement point is the already-named entitlement-enforcement follow-on (basis §7 item 4 — refusing a countersign by an unbound principal); competence gating (§7 item 6) is the same shape one join wider, and the two should be designed as one amendment when either arrives. Until then all three layers remain recordable-not-gating, matching D-5's ratified posture.

### 3.4 The migration question dissolves under runs-are-linear

The commission asks how a vocabulary transition migrates existing free-text grants under append-only law. Checked against the actual constitutional posture: **mostly, it doesn't have to.** Deltas reach reality only in a birth chain; a birth chain applies its full lineage before the world's first data row, so a world born with the band delta has its band-validation trigger in force before any grant exists — there is never a mixed free-text/closed population inside one world. Free-text grants in *prior* worlds are dust with their worlds (read-only evidence, never patched). The real transition cost is operational, not mechanical: at the first band-delta world's birth, still-believed competences are **re-granted** as fresh attributed events whose `basis` field cites the prior world's record — which is the honest shape anyway, since a competence grant is a dated belief, and re-attesting it at a vocabulary boundary is a feature of the record, not a migration burden. No backfill, no mapping table, no retro-edit; the append-only problem the question anticipates is foreclosed by the runs-are-linear ruling before it arises.

### 3.5 Deliberately not decided here

Everything substantive, on purpose: band count, names, ranks; whether an external-scheme mapping is ever recorded and where; whether activity joins band as a declared vocabulary in the first cut; whether requirement declarations need a countersign; when (or whether) grading is ever armed, and at which enforcement points; and whether the three working kind-names survive contact with a real spec. The one thing this section asserts is the *architecture class* — deployment-declared, event-sourced, ordered vocabulary — as the shape that reconciles the 9(c) precedent with the grading requirement. Deployment data remains the ratified precondition; this sketch does not substitute for it.

### 3.6 Ratification question

## 4. How the three designs interact (stated honestly)

*(Plain words first. This section exists so the packaging facts are on the record before any commissioning, not discovered during it: a yes to §1 and a yes to §2 are one build, not two — same column, same constraint, sensibly one spec — while either can also be ratified alone; §3 can proceed or wait independently of both; and two nearby-sounding extensions are deliberately NOT carried along by a yes to §1/§2, so they cannot ride in by accident.)*

**Designs 1 and 2 are one mechanism, and one delta could carry both.** Both are "license `principal_binding_active` onto an s40 kind and give it the same-kind `active = false` supersession lift" — a single widened re-issue of `principal_binding_active_kind_shape` serves both, and the basis's C8 already noted the unbind is "a future amendment on reinstatement's footing." They are nonetheless *separable*: §1 stands alone (a declaration is structurally a binding; licensing it is idiom-completion), and §2 stands alone (suspension needs lift semantics whether or not unbind ships). If ratified together they should ship as one delta family with one spec; if ratified apart, whichever ships second re-issues the same CHECK wider, per the established one-home discipline.

**What deliberately does NOT join that shared mechanism:** revocation lift (§2 keeps `principal_revoked` outside the licensed set by type — the terminality is the design, not an omission) and registration retraction (named in §1.4/§2.4 as a real open question that should ride whichever of these is specced first, decided rather than inherited by accident).

**Design 3 is mechanically orthogonal but philosophically downstream of the same move.** It touches neither the standing function nor the binding CHECKs of designs 1–2; no ordering constraint exists between it and them. Its one dependency on this family's precedents is conceptual: §3.3's "vocabulary as declared events" is the same reconciliation the 9(c) free-text ruling forced, extended to a vocabulary the kernel must eventually *order*. A reinstatement interaction worth one sentence: if grading ever gates acts (§3.3 layer 3), a lifted suspension (§2) restores write acceptance but says nothing about competence — the two standings are independent records, and a future grading spec should not conflate "active again" with "competent."

**Common constitutional path for all three:** each is kernel-lineage work — a Fable-authored, maintainer-ratified spec, birth-chain-only, scratch-witnessed both-polarity with the SQL/ASP differential in AGREE, per the standing ORCHESTRATION contract. None is class-ratified fail-safe as sketched (designs 1 and 2 re-issue live function/view bodies; design 3 mints a validating trigger), so each routes through full ratification. The C12 Fable-builds ruling was scoped to the s40/s41 family only; the builder for these future families is the standing default unless the maintainer rules otherwise at commissioning.
