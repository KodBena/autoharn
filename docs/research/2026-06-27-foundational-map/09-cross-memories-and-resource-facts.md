## Summary

The maintainer maintains two parallel knowledge surfaces: passive Claude memories (37 files across three project memory dirs, indexed by per-project MEMORY.md) and an authoritative, machine-local resource SSOT file `omega/services_local.gitignore`. The resource/reference FACTS (postgres hosts, redis topology, tensorboard root, x11 headed-display, perf-capture, venvs) live canonically in `services_local.gitignore`; the memories deliberately carry only the how-to-use protocol and DEFER connection facts to that file — an already-existing "registry vs. behavior" split that Pillar 1 should formalize. The structural problem: memory is PUSH (passively injected, low-fidelity, no freshness guarantee), and the AI leans on it hardest exactly when context is thinnest, the opposite of the intent for long sessions. The maintainer has already built the antidote once — omega's `todo` Postgres work-status store replaced a hand-edited `docs/work-status.json` with a SQL anti-corruption layer that you QUERY on demand, with constraint-enforced validity, a `work_status_violations` invariant gate, and `audit_log`/`table_asof` time-travel. That migration (passive editable file → queried SQL with invariant gates + audit) is the exact template for Pillar 1: a capability registry the agent PULLS at point-of-need rather than a memory that gets stale-injected. The human analog is explicit in omega CLAUDE.md: FEATURES.md exists because the surface "outgrew the project author's ability to enumerate features from memory."

## Gaps (where the haphazardness lives)

- Resource facts live in a passive gitignored markdown file (services_local.gitignore) — there is no query interface and no liveness check; an agent must remember the file exists and read it, and nothing detects when a fact (e.g. the per-session-changing X11 cookie host, or a redis port) is stale. This is precisely the 'memory leaned on under context pressure' failure mode for resource facts.
- The split is bifurcated and informal: connection facts in services_local, how-to-use protocol in separate Claude memories that 'point here' by name (reference_dev_resources, project_research_redis_topology, reference_perf_capture_normalization, feedback_tensorboard_default, reference_x11_headed_display_access). The cross-references are prose, not machine links — an agent loading the protocol memory may never resolve to the connection fact, and vice-versa.
- Memory injection is PUSH with no provenance or freshness stamp: memory files carry an originSessionId but no validated-against-commit or last-verified timestamp; an empirical interpretation banked as a memory (e.g. core-3 pinning) can be later REFUTED yet the stale belief persists until manually rewritten (host-cpu-isolation-pinning.md shows the core-3 benefit refuted in-line, but only because a human re-edited it).
- OR/automated-reasoning capabilities (Z3, OR-Tools) are NOT enumerated anywhere in services_local or memory — the registry gap that lets agents neglect provable tools is total here; there is no machine-readable 'what venv/solver is installed' fact for the agent to pull.
- The standing 'assume up, but surface a failed check loudly' rule (services_local lines 12-15) describes a verification seam that exists only as prose advice — there is no executable health-probe the agent can call to honor it.

## Harness hooks (where a registry / ledger / logic-gate plugs in)

- Build the capability registry as a QUERIED store (SQL table or logic-program facts) mirroring services_local.gitignore — one row per service/resource/venv/solver with address, liveness-probe command, owning-memory id, and last-verified stamp — so the agent PULLS `SELECT addr,probe FROM capability WHERE name='research_redis'` at point-of-need instead of being PUSHED a stale memory. Generalize the omega work-status anti-corruption pattern (constraints + `*_violations` gate + audit_log) directly.
- Promote the prose cross-links between connection-fact (services_local) and how-to-use (memory) into machine-resolvable FK/relation rows so a protocol query always joins to its current connection fact — closing the bifurcation gap.
- Add a registry-backed health-probe tool the agent can invoke to honor the 'surface a failed check loudly' rule (services_local lines 12-15): a probe per row (e.g. `psql ... -c 'select 1'`, redis PING, `xset q` with the derived XAUTHLOCALHOSTNAME) returning live/stale, making liveness a queried fact not a cached belief.
- Register OR/automated-reasoning tools (Z3, OR-Tools) and the catch-all venv (/home/bork/w/vdc/venvs/kataproxy/) as first-class capability rows with invocation examples, so agents discover provable-result tools by querying the registry rather than defaulting to the StackExchange-shaped statistical path.
- Stamp every registry fact with a validated-against (commit/timestamp) column and a provisional|confirmed|retracted status, reusing the tlab_reading⊥tlab_finding measurement/interpretation discipline (exp_db.py) so a refuted resource belief (like the core-3 pinning) is SUPERSEDED queryably, never silently stale — the 'pull not push' design gives the agent freshness metadata the memory injection cannot.
- Treat the per-session-volatile X11 cookie host as the canonical example of a fact that MUST be a derived-at-query-time probe in the registry, never a stored value — encode it as a `derive` command rather than a literal.
