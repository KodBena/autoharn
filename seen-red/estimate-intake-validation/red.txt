THE FEATURE, both halves proven live below:

  - bootstrap/templates/led.tmpl now validates an `estimate:` statement (whitespace-normalized
    copy: every run of \n/\r plus following indentation collapsed to one space) against
    design/USER-RETROSPECTIVE-RECIPE.md's "Estimate statement grammar" section BEFORE the
    INSERT -- exactly six `|`-separated fields (TASK-SLUG, TOOL-CALLS, SUBAGENT-SPAWNS,
    WALL-CLOCK, TOKEN-OOM, BASIS), each with its own shape check: TASK-SLUG matches
    `^[a-z0-9][a-z0-9-]*$`; TOOL-CALLS/SUBAGENT-SPAWNS match a non-negative integer or an N-M
    range; WALL-CLOCK matches a duration (`20m`/`2h`/`1d`) or a range; TOKEN-OOM is one of the
    closed vocabulary `1K | 10K | 100K | 1M | 10M+`; BASIS is non-empty. On failure: REFUSED,
    exit nonzero, NOTHING written (atomicity by refusal-before-write), teach-text naming the
    grammar and the specific field that failed. On success: the statement is INSERTed AS TYPED,
    byte-exact (never the normalized copy).
  - bootstrap/templates/pickup.tmpl's new estimates() reader renders every on-record `estimate:`
    row under its own `### SECTION: ESTIMATES` header, sorted by TASK-SLUG then ledger row id,
    using the identical newline-normalization SELECT and MALFORMED-flagging posture
    resources() already established for `resource:` — never silently dropping a bad row.
  - COHERENCE: led's validator trigger and pickup's SELECT filter define "an estimate
    declaration" IDENTICALLY, the same coherence-partner contract `resource:`/resources()
    already keep — led triggers on the statement after stripping leading POSIX [:space:]
    starting with `estimate:`; pickup's filter names the same class:
    `statement ~ '^[[:space:]]*estimate:'`.
  - The grammar's ONE documented home is design/USER-RETROSPECTIVE-RECIPE.md's new "Estimate
    versus actual" section (§6), which both parsers cite in their own comments rather than
    restating the grammar a second time (ADR-0012 P1) — the same doc also carries the
    estimate-vs-actual retrospective method (deliberately NO audit family, NO gate, NO exit
    code: a missed estimate is retrospective data, never a violation).
