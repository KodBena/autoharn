0. THE BUG THIS FIXTURE FOUND AND THE FIX IT PROVES (both polarities in one finding -- a
   FIXTURE-HARNESS bug, not a `serving/` production bug; still a genuine red->green transition
   captured live, not fabricated). The FIRST live run of this fixture was RED on all three WM3
   legs:

     === wm3-unknown-top-level-key-refuses-before-bind ===
       [FAIL] port refused-to-connect=True, exit=-15, stderr tail='' (expected to name
       'unknown top-level key')
     === wm3-missing-required-key-refuses-before-bind ===
       [FAIL] port refused-to-connect=True, exit=-15, stderr tail='' (expected to name
       'missing required key')
     === wm3-zero-deployments-refuses-before-bind ===
       [FAIL] port refused-to-connect=True, exit=-15, stderr tail='' (expected to name
       'ZERO deployments')

   exit=-15 (SIGTERM) with EMPTY stderr is the tell: the fixture called
   `bs_fixtures.stop_server(proc_wm3)` -- which unconditionally `terminate()`s -- IMMEDIATELY
   after `Popen`, racing the child's own Python-interpreter-startup + module-import window. The
   child was killed before it ever reached `boundary_multiplex_config.load_multiplex_config`'s
   own refusal path, so the SIGTERM exit code merely LOOKED like a fast refusal while proving
   nothing about `serving/boundary_service.py`'s actual construction-time behavior (confirmed
   by hand: `python3 -m serving.boundary_service --config <same-bad-toml> ...` run directly,
   unhurried, exits 2 with the correct named message every time -- see item 1 below). Fixed by
   replacing the immediate `terminate()` with `proc.communicate(timeout=10)` -- waiting for the
   process's OWN natural exit (a construction-time config refusal has no DB round trip and no
   network I/O, so it is fast on its own) -- before ever checking the port or the exit code, and
   tightening the exit-code assertion from `!= 0` to `== 2` (this service's own documented
   config-refusal exit code, `_parse_args`'s sibling bind-guard refusal uses the same `return 2`
   convention). The GREEN run below re-exercises exactly the path that was red (all three
   malformed-config legs) and stays green.
