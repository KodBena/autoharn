THE FEATURE, proven live below:

  - bootstrap/templates/led.tmpl now validates an `outcome:` statement (whitespace-normalized
    copy: every run of \n/\r plus following indentation collapsed to one space) against
    design/USER-RETROSPECTIVE-RECIPE.md §7's "The `outcome:` statement grammar" section BEFORE
    the INSERT -- exactly five `|`-separated fields (TASK-SLUG, MODEL, SEAM-VERDICT,
    DEFECTS-FOUND-AT-SEAM, NOTES), each with its own shape check: TASK-SLUG matches
    `^[a-z0-9][a-z0-9-]*$` (the same rule `estimate:`/`actual:`/`taxon:` already use); MODEL
    matches `^[a-zA-Z0-9][a-zA-Z0-9._-]*$` (a bare identifier, no spaces, so a future grouping-
    by-model read is an honest exact-string match); SEAM-VERDICT/DEFECTS-FOUND-AT-SEAM/NOTES are
    each non-empty free text. On failure: REFUSED, exit nonzero, NOTHING written (atomicity by
    refusal-before-write, ADR-0002), teach-text naming the grammar and the specific field that
    failed. On success: the statement is INSERTed AS TYPED, byte-exact (never the normalized
    copy).
  - UNLIKE `estimate:`/`actual:`/`taxon:`, `outcome:` deliberately has NO `./pickup` display
    section -- Section 7's own "consumption surface" states this is RETROSPECTIVE-ONLY, per the
    standing never-police invariant (the same invariant §6 states twice for `estimate:`/
    `actual:`): grouping outcomes by model is a retrospective's own read (documented method,
    §7's "read surface" subsection), never a live ranking display. This fixture therefore proves
    the accepted row is readable straight off `ledger_current` (the same view every `pickup.tmpl`
    reader already queries) instead of proving a pickup section renders it.
  - COHERENCE: the grammar's ONE documented home is design/USER-RETROSPECTIVE-RECIPE.md §7,
    which led.tmpl's validator cites in its own comments rather than restating the grammar a
    second time (ADR-0012 P1).
