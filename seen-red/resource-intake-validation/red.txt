Work item `resource-intake-validation` (BACKLOG 2026-07-12). Three defects were witnessed live
on 2026-07-12 when the maintainer declared the first live resource in world run12:

  1. `bootstrap/templates/led.tmpl`'s generic `<kind> <statement...>` write path performed NO
     intake validation of a `resource:`-prefixed decision statement -- a malformed statement
     (missing fields, an embedded newline from a wrapped terminal paste) landed silently.
  2. `bootstrap/templates/pickup.tmpl`'s `resources()` reader shredded a statement containing an
     embedded newline into multiple spurious rows, because it iterated psql's own PHYSICAL output
     lines (`-t -A`) rather than logical result rows -- one fake row even printed
     `id=experiments`, the tail of the pasted text misread as a row id.
  3. Neither write path was audited for atomicity beyond Postgres's own single-statement
     guarantee (the maintainer's ruling: malformed adds must fail loudly at write time, and
     ledger writes should carry atomic all-or-nothing semantics in general).

THE FIX, both halves proven live below:

  - led.tmpl now validates a `resource:` statement (whitespace-normalized copy: every run of
    \n/\r plus following indentation collapsed to one space) against design/USER-BLESSED-TABLE-
    TEMPLATE.md's "The statement grammars" section BEFORE the INSERT -- exactly six `|`-separated
    fields, CLASS in {solver, service, backend, binary, library}, TIER one of
    `available | blessed: <task-shape> | mandated: <task-shape>`. On failure: REFUSED, exit
    nonzero, NOTHING written (atomicity by refusal-before-write). On success: the statement is
    INSERTed AS TYPED, byte-exact (never the normalized copy).
  - pickup.tmpl's resources() SELECT now applies the identical newline-collapse SERVER-SIDE
    (`regexp_replace(statement, '[\n\r]+[ \t]*', ' ', 'g')`), so one query row is always exactly
    one physical `-t -A` output line, regardless of what the statement's own text contains.
  - COHERENCE (follow-up, same day): led's validator trigger and pickup's SELECT filter define
    "a resource declaration" IDENTICALLY. led triggers on the statement AFTER stripping leading
    POSIX [:space:] starting with `resource:`; pickup's filter names the same class:
    `statement ~ '^[[:space:]]*resource:'` (it was `statement LIKE 'resource:%'` at first --
    position-0 only, which would have let a validated-but-indented declaration vanish from the
    read side). Each file's comment names the other as its coherence partner; changing either
    side alone is flagged there as the defect it would be.
  - Atomicity audit of every write path in led.tmpl (907 lines): every path but one (`led
    review`) is a single INSERT/DELETE statement, atomic by Postgres semantics with no code
    change needed. `led review` already wraps its two INSERTs (ledger + review_detail) in an
    explicit BEGIN/COMMIT -- pre-existing, confirmed correct, untouched.
