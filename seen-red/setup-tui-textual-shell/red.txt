LIVE (not a fixture artifact) -- backward navigation worked in InteractiveUi/ScriptedUi (the

This fixture is inherently mixed-shape: most cases (WX1, WX2, WX6) drive the real Textual
application through Textual's own headless test harness (run_test/Pilot) to a real, complete
outcome and assert on it -- a failure here (an exception, a hang, a wrong checklist row, a
transcript that stops matching the plain backend) IS this gate's red case, the same both-
polarity-adjacent shape seen-red/setup-tui-scripted-smoke's own red.txt already banks for this
package. WX5 is closer to classic red-before-green: it delivers a REAL SIGTERM to a REAL
tools.setup_tui.app._run_textual process (via a pty, a synthetic-but-real child process standing
in for a live boundary_service) and asserts the child is reaped and the process exits promptly --
the failure mode this specifically guards against (a worker thread parked on a prompt hanging
interpreter shutdown, or the App's own terminal-restore teardown never completing) was WITNESSED
red during this build's own construction, before it was fixed:

REAL RED, witnessed during this build (2026-07-19, this commit's tree, before the fixes below
landed) -- WX5 hung past its own generous wait budget, reproducibly. A faulthandler all-threads
stack dump caught mid-hang showed the process stuck inside Textual's OWN shutdown path
(App._shutdown -> LinuxDriver.close -> WriterThread.stop -> Thread.join), not in this package's
code -- root-caused to this FIXTURE's own pty-draining discipline (the test stopped reading the
pty's master fd once it found the "ready" marker, so Textual's background WriterThread's
write() to the now-undrained pty slave blocked once the kernel buffer filled, and
WriterThread.stop()'s own .join() waited on a thread that could never finish). The fix landed in
this fixture's own _run_wx5 (keep draining the pty continuously while waiting for exit, never a
blind proc.wait()) -- confirmed stable across 6 consecutive full-suite runs after the fix, 0/6
failures (was intermittent, ~1-in-2 to 1-in-3, before). Two independent PRODUCT hardenings were
also made during the same investigation, defensible on their own merit regardless of which one
was the actual trigger: tools/setup_tui/ui_textual.py's _call_from_thread_safe (bounds every
worker-to-main-thread bridge call, where the vanilla App.call_from_thread's own future.result()
has no timeout and could otherwise hang a worker forever if scheduled during the App's shutdown
transition) and tools/setup_tui/app.py's SIGTERM handler ordering (request_shutdown before the
potentially-5-seconds-blocking _terminate_boundary_proc, so a parked worker is woken before, not
after, that wait).

WX4's suspend-wiring case is real too: TextualUi.suspend() is driven from an actual worker
thread (App.run_worker(..., thread=True)) and observed to raise the REAL SuspendNotSupported
Textual's own headless driver raises -- proving the bridge reaches App.suspend() for real,
never a silent no-op standing in for it.
