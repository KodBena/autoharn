Work item / tracker item `maintainer-review-queue` (2026-07-13). Maintainer commission: an
EXECUTIVE LEDGER — a maintainer-facing review queue where items awaiting his EXPLICIT review can
be ticked off, ranked. Design constraint from house law: the queue lives in the LEDGER, never in
a prose file (the BACKLOG-retirement lesson: a live task queue narrated in immutable prose rots
in seconds — ADR-0005 Rule 6 corollary). Deliverable was a `review:`/`review-done:` statement
grammar pair, refusal-grade like the five existing precedents (`resource:`/`estimate:`/`actual:`/
`taxon:`/`interface:`), plus a `MAINTAINER-REVIEW-QUEUE` section in `./pickup` rendering open
entries rank-ascending with the exact tick command.

THE FEATURE, both halves proven live below:

  - bootstrap/templates/led.tmpl now validates a `review:`-prefixed decision statement
    (whitespace-normalized copy: every run of \n/\r plus following indentation collapsed to one
    space) against design/USER-RECIPES-FAQ.md's "Your review queue" section BEFORE the INSERT --
    exactly four `|`-separated fields (SLUG, RANK, WHAT, POINTER): SLUG matches
    `^[a-z0-9][a-z0-9-]*$`; RANK is a positive integer (1 = most important); WHAT and POINTER are
    each non-empty. A `review-done:`-prefixed statement is validated identically against its own
    two-field grammar (SLUG, DISPOSITION), with SLUG held to the SAME slug grammar `review:` uses.
    On failure: REFUSED, exit nonzero, NOTHING written (atomicity by refusal-before-write),
    teach-text naming the grammar and the specific failing field. On success: the statement is
    INSERTed AS TYPED, byte-exact.
  - bootstrap/templates/pickup.tmpl's new review_queue() reader renders the MAINTAINER-REVIEW-
    QUEUE section by DERIVING each SLUG's current state from the highest-ledger-row-id row among
    BOTH prefixes: a `review:` row opens or RE-RANKS an item (no `--supersedes` flag needed -- the
    latest row for a SLUG simply wins), a `review-done:` row removes the SLUG from the rendered
    open queue (nothing deleted -- append-only, just no longer the latest state), and a LATER
    `review:` row for the same SLUG RE-OPENS it. Open entries print rank-ascending (tie-broken by
    SLUG), each block carrying the exact `./led decision "review-done: <slug> | <disposition>"`
    one-liner to tick it. An empty queue prints its own explicit "(no open review-queue entries
    on record)" line, never silence. A malformed row (wrong field count, non-integer RANK) is
    flagged MALFORMED, never silently dropped or allowed to corrupt another SLUG's derived state.
  - COHERENCE: led's `review:`/`review-done:` validator triggers and pickup's combined SQL filter
    (`statement ~ '^[[:space:]]*review(-done)?:'`) define "a review-queue declaration"
    IDENTICALLY, the same coherence-partner contract every earlier grammar pair already keeps.
  - The grammar's ONE documented home is design/USER-RECIPES-FAQ.md's new "Your review queue"
    section -- a deliberate deviation from that page's usual "point elsewhere" convention (stated
    inline, in the section itself), because this recipe is the grammar's actual SSOT rather than
    a pointer to it (ADR-0005 Rule 1).
  - No collision with the pre-existing `led review <entry-id> <verdict> ...` subcommand: that is
    a distinct top-level dispatch on argv[0] (countersigning an earlier ledger row), never
    reached from this generic `<kind> <statement>` validation path, and the shell case patterns
    `review:*` / `review-done:*` partition every statement disjointly by construction (the latter
    never matches the former's literal prefix).
