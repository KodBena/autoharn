A HAZARD FOUND AND FIXED DURING THIS BUILD (worth banking alongside the two refusals above,
since the fix IS what makes GREEN-FORCE below safe): the first version of this script applied
the full birth-chain DDL unconditionally under --force, including against a schema that
ALREADY carried the complete chain. Postgres refused with "cannot drop columns from view" —
CREATE OR REPLACE VIEW cannot shrink a view's column list, and re-running an EARLIER
kernel/lineage file's own (shorter) view definition against a view that later files had
already grown (s20/s21/s22's own "column-complete" re-issue idiom) hits exactly that wall.
This is a general property of the birth chain, not specific to this script — `new-project.sh
--new-world --force` against an already-migrated schema would hit the identical failure — and
it is flagged in track-work.sh's own comments as a hazard out of THIS offering's scope to fix
upstream (new-project.sh is under this session's own collision constraint; kernel/lineage/*.sql
is frozen, maintainer-ratified-spec-only). The fix taken here, in this script's own scope: skip
the DDL re-apply entirely when the kernel schema already exists — --force re-derives
deployment.json + the verb shims + the principals only, never re-runs kernel DDL a second time.
