RED case (embedded in run_fixtures.py's case 1, "negative self-check"): a synthetic BROKEN act
class (`BrokenEagerAct`) whose own `.render()` resolves a Hole's `extract` callable EAGERLY --
exactly the purity violation `Plan.render()`'s contract exists to forbid (a decision-phase
rendering must never touch a hole's real-value extractor; that extractor may only run at commit,
against a step's REAL output, per ADR-0000's ban on a fabricated/prematurely-materialized value
standing in for a real one). Rendering a plan containing `BrokenEagerAct` DOES trip the poison
`extract` (an `AssertionError`), proving the purity check this fixture applies to the REAL
`CommandAct`/`WriteAct`/`BackgroundAct` types is not vacuous -- it can and does catch the shape of
defect it exists to catch.

A second, independent ordering hazard was caught and fixed DURING this module's own build, before
this fixture was ever run against it (so no separate "old commit" RED replay exists for it, unlike
setup-tui-write-file-atomicity's git-history replay -- there is no prior committed version of this
module to diff against): `commit_executor.execute`'s original draft called the caller-supplied
`on_result` UI callback BEFORE marking the journal entry DONE. A callback exception (a rendering
bug, nothing to do with whether the act itself succeeded) occurring immediately after a REAL
write/command already took effect would leave the journal still naming that entry PENDING -- a
resumed run would then RE-RUN an act that had already, truly, happened (a double keygen, a double
`led decision` write, a second `CREATE ROLE`). Fixed by marking the journal DONE immediately after
the act succeeds, before the callback runs (commit_executor.py's own "ORDER IS LOAD-BEARING"
comment names this explicitly). Case 3 below proves the FIXED ordering: a callback that raises
right after entry 0's real write still leaves the journal correctly marking entry 0 DONE, and a
resumed execute() does not re-run it (verified by an mtime marker that a re-write would disturb).
