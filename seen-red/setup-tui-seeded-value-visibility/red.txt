Maintainer-witnessed finding (ledger row 1130): a live in-UI load of known-good-blank.toml
reported "seeded 21 field default(s)" including hydration.durable_decisions/adopt_adrs
(MultiChoiceField checkbox groups) and several ConfirmFields, but the sections did not show the
values as set. Root cause, established empirically before writing this fixture: it is NOT
field-kind-dependent -- get_field_value/set_field_value/every widget builder were verified
correct for all four field kinds once a section is genuinely unblocked. The real defect is that
hydration (like boundary/observability/birth/principals-authority/signed-genesis) stays BLOCKED
until a destination directory is set elsewhere (Fork/target), and the OLD SectionPane.compose
returned immediately on a blocked reason, rendering ONLY the "BLOCKED -- ..." banner -- nothing
about the section's own fields, seeded or not. An operator who loads a config and immediately
checks a gated section (without first visiting Fork/target) saw a blocked banner with zero cue
that seeded values were sitting underneath, for every field kind that section carries.

Case RED loads the OLD SectionPane straight from commit 3cc769d (the last commit before this
fix, via git show, executed in an isolated namespace) into the REAL running app, reproduces the
maintainer's own path exactly (in-UI Load, then visit hydration without ever visiting
Fork/target first), using a known-good-blank.toml variant seeded with NON-EMPTY
durable_decisions/adopt_adrs (the shipped template seeds both as [], which cannot distinguish
"seeded correctly to nothing" from "seeding is broken"). Cases GREEN 2-3 prove the fixed
SectionPane: the same blocked visit now names every field whose value differs from its own
default (both MultiChoiceFields), and once dest is set (unblocking hydration), the ConfirmField
and both MultiChoiceFields all render their real seeded values.
