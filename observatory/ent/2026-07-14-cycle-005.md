## 1. STOP-SPAM VERDICT (headline)

**The strict-subset-inherit fix (`stop-breaker-progress-reset-defect`, commit `e537701` /
its worktree twin `c6207f2`, landed in this repo's checked-out `next` HEAD at
2026-07-14T01:33:35Z) IS live and IS firing correctly on every single transition observed
since the session restart. The continuing spam is neither the old pre-fix pattern surviving

**not** the "unfixable in-world, unfixable per runs-are-linear" shape the question offered:
the `review_gap` debt that dominates the largest burst below was fully discharged in-world,
twice, within the same session (two genuine `clean_allow` events, §1.2). So: real, ongoing,
high-frequency block/fail-open cycling — but it is the mechanism doing exactly what its own
code says, not breakage.

### 1.2 Every journal transition since the restart matches the shipped code, line for line

Read `hooks/stop_clean_exit.py`'s actual `_breaker_transition()` (lines 517-539) rather than
trusting the fix commit's own prose: the merged code is **simpler** than the original
`0cd0a6f` commit message described. The message spoke of a distinct `breaker_inherit`
outcome that allows a subset-continuation immediately without incrementing further; the
code that actually shipped (superseding commit, same message text, hash `e537701`/`c6207f2`)
does not implement that outcome at all (`grep -c breaker_inherit hooks/stop_clean_exit.py` =
0) — it does something narrower: an **identical** debt-hash always increments (`count =
prior + 1`); a **strict subset** of the prior entries also increments (inherits, does not
reset); anything else (an ADDED entry, anywhere — a pure addition, a swap, or a
subset-plus-addition) resets to 1. Once `count >= 3` the call fires `breaker_fail_open`
(allow) EVERY time thereafter, unconditionally, for as long as the identical-or-subset
condition keeps holding — there is no "first fail-open only" special case. This is a
legitimate, simpler design than the commit's own prose promised (the prose is stale
relative to the code — flagged below as a hazard, §5), but it is coherent and it is what
runs.

framed "unfixable in-world, runs-are-linear" debt class; it was fixable, and was fixed,
inside this same run.

**What is actually producing the frequency the maintainer is seeing**: the debt set behind
this mechanism is churning — new `review_gap` obligations and new/swapped `work_open` items
keep arriving at a rate that frequently outpaces pure closure. The fix specifically only
relieves the PURE-SHRINK case (items only leaving); it does nothing for the much more common
case in an actively-working session where new debt keeps entering the set alongside old debt
leaving it. Every addition — even one item added while nine others close in the same breath —
throws the count back to 1 and restarts the 3-strike climb. That is intentional (a genuinely
new obligation deserves fresh scrutiny, per the fix's own stated rationale) but it means a
session generating debt nearly as fast as it discharges it will keep re-triggering the full
block→block→fail-open cycle every few minutes, which is what "still essentially broken" is
observing. This is STRUCTURAL-TENSION, the same class cycle-004 named, now with a concrete
mechanical explanation for why the fix didn't make it visibly quieter: the fix targets a
narrower case (pure progress) than the one actually recurring (progress interleaved with new
debt).

## 4. A DOC/CODE DIVERGENCE HAZARD (flagged in passing, not fixed here)

While tracing the fix (§1.2), the merged commit's own message (and the module docstring it
installed, `hooks/stop_clean_exit.py` lines 91-109) describes the design in terms that the
actual shipped code does not implement: no `breaker_inherit` outcome, no `inherited_count`
marker, no "first-subset-after-fail-open allowed immediately, all others block" distinction
— `grep -c 'breaker_inherit\|inherited_count' hooks/stop_clean_exit.py` returns 0 in both
matches. The code that runs is simpler and, on this cycle's evidence, correct on its own
terms — but a reader trusting the docstring's or the commit message's narrative (rather than
the function body) would expect a THIRD outcome string in the journal that never appears and
would misread the observed `breaker_fail_open`-keeps-incrementing behavior as a bug rather
than the intended (if under-documented) design. This is a hazard within reach of what this
cycle touched (the same file, the same mechanism, read in full to answer the maintainer's
question) — not something this observatory report is authorized to fix (this repo's own
constraint: hooks/ is gated on the live ent session; this is a read-only cycle) — so it is
flagged loudly here for the next builder who touches this file, rather than silently routed
around. Concretely: either the docstring should be corrected to match the shipped
increment-forever behavior, or the `breaker_inherit`/`inherited_count` behavior described in
prose should actually be implemented — the current state is a lying-docstring-in-miniature,
the same defect class ADR-0012/ADR-0013 already name for code, applied here to a doc that
asserts an outcome vocabulary the code doesn't produce.

## 6. METHOD CANDIDATES

- **A mechanical "which commit is actually checked out when a hook fires" trace** (§1.1) —
  this cycle needed `git reflog` plus a file-mtime/hash cross-check to establish that the fix
  was live at the moment in question, because the deployment model (ent execs this repo's
  working tree directly, with no deploy step) means "is the fix live" is a moving,
  wall-clock-dependent question, not a static fact once a commit merges. A small script that
  answers "for hook file F, at timestamp T, which commit's content was checked out in the
  executing repo" (reflog-walk + `git show <rev>:<path>` diff against the journal's own
  timestamps) would generalize this cycle's single-purpose investigation into a reusable
  observatory tool for the next "did the fix actually land" question. Candidate for the
  recipes/tooling corpus; not designed or built here.
- **Docstring/code conformance check for hook files** (extends cycle-004's "paired-journal
  integrity check" candidate into a sibling shape): §4 found a hook whose own module
  docstring names an outcome vocabulary (`breaker_inherit`, `inherited_count`) the code
  never emits. A grep-based lint (does every "journaled with outcome X" claim in a hook's
  docstring correspond to a `_journal({"outcome": "X", ...})` call site in the same file?)
  would have caught this mechanically at review time rather than requiring a live-behavior
  trace to surface it. Flagged as a candidate; not designed or adjudicated here.
- Carried forward, unchanged, not evaluated this cycle (same as cycle-004): the
  iterate-to-approval review convention, and the "stall vs. dead" diagnostic shape — both
  would need picom-side task narrative this report does not include.
