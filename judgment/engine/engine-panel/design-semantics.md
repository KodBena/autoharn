# The Deductive Engine — Judgment Semantics and Obligation Logic

**Lens:** what, precisely, must the engine derive; the logical character of each judgment; how judgments cite law; how the vocabulary grows without drift. Design for a builder, from the primary record (LEDGER-LOGIC-MARRIAGE.md incl. Appendix A, the safety-critical-logging BRIEF, BRIEF-CONFORMANCE-MAP, `soundness.lp`/`core_a.lp`/`ledger_tnow.lp`, FINDINGS F1–F53, consult 35, ADR-0000/0012).

---

## 0. The one-sentence thesis

A judgment is a **typed, law-citing, frontier-pinned verdict derived from the record by a named engine** — and the engine is nothing more than the closed set of such judgments, each admitted with a paid-for specimen and a mutation that flips it, computed under one semantics at three schedules (write-time, turn-time, close-time). Everything below is the working-out of that sentence against what the fourteen experiments actually proved.

## 1. What a judgment IS — the type, first (ADR-0000 Rule 1)

The record already contains the negative specimens that dictate the type. A boolean verdict lied by silence (F49: three mandatory close lines silently did not run); an uncited flag was un-adjudicable (F51's four alias flags needed a human to reconstruct *which rule's kind-test* produced the false positive); an unpinned derivation was unreplayable (the pre-A.4 posture); an un-deduplicated finding would re-nag every turn (marriage §6.1). So the judgment type has **five mandatory components**, and a value missing any of them is not constructible:

```
Judgment ::= (family     : FamilyId,          -- from the closed registry (§2)
              verdict    : Verdict[family],   -- from the family's OWN closed enum, never a shared boolean
              subject    : SubjectRef,        -- row ids / edge / span / (stream, id-range); id-pinned, never text
              law        : LawRef,            -- the governing law atom (§4); a judgment with no law is a bug
              frontier   : Frontier)          -- the record point it was derived AT (§3.2)
```

plus, attached at banking, the **DerivationRecord** already mechanized in marriage A.4 ({engine+version, config, EDB/program/output hashes}) — a verdict without both records is NO RESULT, inherited unchanged. Identity for storage is the marriage's `FindingIdentity` discipline: a persisting judgment is one stored row **re-observed at a new frontier**, never re-injected.

Three consequences worth stating because they are load-bearing:

- **Per-family verdict enums, no global truth lattice.** I9 says discharge status is never conflated into "verified"; the engine applies that to itself. `observed_currency`'s verdicts ({OBSERVED-CURRENT, UNOBSERVED-AT-CITATION, STALE-OBSERVED}) are epistemic-status values, not truth values; the differential's ({AGREE, DIVERGE_BY_DESIGN, DIVERGE_DEFECT, QUARANTINED}) are qualification values. Forcing these into one lattice is exactly the ADR-0008 category error Specimen 2 (`CellLedger`) names — three currencies read as one. Cross-family aggregation happens in exactly one place, the close manifest, through **declared** verdict→{OK,RED,REQUIRED-ABSENT} mappings per line, never by implicit coercion.
- **Silence is unrepresentable.** Every family has a QUARANTINED/REQUIRED-ABSENT-shaped member for "the derivation did not run," and a non-run turns the run red (F49's fix, A.5/A.8 generalized: an empty model over a non-empty EDB is a quarantine, never an empty result set).
- **Subject confidence never keys a verdict.** `confidence` is subject-authored data; a verdict conditioned on it prices a self-declared label and corrupts it as a measurement (the F26 law, extended from `kind`/`concern` to `confidence`). Reports may sort by it; verdicts never branch on it.

## 3. Positions on the formal-logic choices — taken, and justified from the record

### 3.1 Negation and model semantics: stratified default negation, with generate-and-test quarantined into a separate lane

The T_now judgments are default negation in its native habitat (`in_force(Id) :- entry(Id,…), not superseded(Id)`). But the engine relies on a stronger property the substrate gives for free and must **check as an invariant**: append-only + earlier-target validation makes every defeater edge strictly backward in id, so the whole judgment program (families A–D, E-discharge) is **stratified — one unique model**, where stable-model and well-founded semantics coincide. The EDB layer therefore carries an acyclicity check (every edge target < source id) as a Family A judgment; a violation is DEFECT, and the run is red before any nonmonotone rule fires. We never depend on answer-set multiplicity for a judgment.

Where multiplicity is the *point* — `core_a.lp`'s choice rules, minimal-repair `#minimize` — that is a different logical act: a **probe about dischargeability or blame**, not a judgment about the record. Position: keep the two in separate programs with separate verdict enums (E-probe verdicts are SAT/UNSAT-shaped; D-blame output is an explanation report). A probe result banked as if it were a record judgment would be a category error of exactly the CellLedger kind. `core_a.lp`'s own header already practices the boundary ("encoding a paraphrased synthesis as hard logic manufactures contradictions the sources may not contain"); the design ratifies it as a structural rule: **stratified layer = judgments; choice layer = probes and explanations; no atom crosses from the second into the first.**

### 3.2 Temporal semantics: frontier-indexed derivation over the id order; ts is a clock, never an order

Ratified law (the F-D retrofit, consult 17 §5.3): **id is the record order; every precedence keys on strict id-`<`; ts never keys defeat.** The design's addition is the *live* consequence: every nonmonotone judgment is true only **at a frontier**. `in_force(4)` held at frontier 21 and fails at frontier 22 — both derivations are correct, and a banked judgment that does not say which frontier it was derived at is unreplayable (an I11/F16 violation inside the engine). So:

- The EDB export carries one fact `frontier(N)` (max id per stream at export), hash-covered by the DerivationRecord.
- A **live** judgment is always as-of the current frontier; a **banked** judgment pins its frontier; replay at that frontier must reproduce it bit-identically (this becomes a standing fixture class: replay s10 at every frontier 1..32 and confirm the known flip of `in_force(4)` at 22).
- Cross-stream judgments (Family F) have a **frontier pair** (ledger-id, acts-id); the sound join semantics under interleaved writes is an open question (§8.3), so until fixed by fixture, Family F runs close-time only, where both streams are quiescent.

**ts has exactly two sanctioned judgment roles**, both comparisons against an external clock, neither an ordering within the record: (i) Family B contemporaneity metrics (lag distributions, M-trigger gap detection against the acts stream — the BRIEF's "trigger fired and entry absent = detectable defect," scoped to M triggers only); (ii) I7 validity bounds (`valid_until < now → EXPIRED`, the `ledger_assumes.lp` closure). Deliberate rejection: **no LTL/CTL/temporal-modal layer.** The record is not a Kripke trace of world states; it is an append-only history of speech acts, and the two-theory split (monotone T_event / derived T_now) *is* the temporal semantics — proven over fourteen experiments, simpler than any modal alternative, and with termination by construction. Where genuinely metric-temporal properties arise they are arithmetic over (id, ts) pairs and belong to the SQL/SMT lanes.

### 3.3 Deontic logic: the Anderson reduction, not modal operators

Position: **no O/P/F modal operators anywhere in the engine.** Obligation enters the system only as a *recorded fact* — an other-assigned row (`obliged(Scope, AssignedBy, ObligesActor)` in the EDB; F26's line: obligations are other-assigned, never self-declared-and-priced) — and the engine derives exactly one thing about it: its **discharge status** (Family E), which is closure over recorded facts. Violation is not a modal falsum; it is a derived flag routed to the review queue (Anderson's reduction — obligation-violated ≡ obligation-recorded ∧ ¬discharged-in-record — which is *precisely* the flag architecture the project already ratified in F28). The classical deontic paradoxes never arise because the engine never derives obligations from obligations: contrary-to-duty structure ("upon violation, X is owed") is the review queue's recorded routing plus, where mechanized, the write-time refuse-and-teach — again recorded acts, not derived norms.

Justification against the record, which is unusually rich here:

- **F26 (three instances, both signs + recovery):** any mechanism cost keyed on a self-declared label corrupts the label. A modal-deontic layer whose derived "forbidden" fed enforcement back onto subject-declared fields would industrialize this failure. The engine therefore derives status and routes flags; the only deny surfaces remain the write-boundary triggers, and those refuse on **apparatus-verified** facts (HMAC stamps, target existence, quotation checks), never on subject-declared labels.
- **F28 (the launder proof, executable):** auto-resolving a violation rewrites the record into coherent-looking falsehood. A deontic layer with consistency-restoring dynamics (as in some defeasible-deontic frameworks) is auto-resolve by another name. Flag-never-auto-resolve is a semantics decision, not a UX one.
- **e17 (the positive proof):** the one deontic judgment promoted to write-time — F53 independence — worked *because* it was a refusal keyed on an apparatus-owned fact (stamp distinctness) with a teaching payload, and the subject repaired the record by **making the claim true**. "A write-time teaching refusal out-performs any pre-run prose prime" is the strongest deontic-design datum the project owns; it argues for exactly this architecture (facts + refusal + taught alternative) and for nothing more modal.
- **The dischargeability probes stay separate** (§3.1): `core_a`'s k-phase result ("the SoD triple is UNSAT at this principal population — a model-size failure, not a contradiction") is a statement about what the obligation set *can* demand of this world, and its honest exit is a recorded waiver row. That is the engine's entire concession to deontic modality: SAT/UNSAT of the strict kernel under population parameters, with waivers as first-class recorded outcomes.

### 3.4 Paraconsistency vs consistency-repair: neither, at the kernel — the quotational stance dissolves the need

Position, restating and grounding the marriage's hard rule: **the kernel's derived validity stays two-valued and row-granular; consistency-repair as record mutation is banned; the FDE lens is a report-side, interim rendering.** The reason there is no explosion problem to solve at the base: the engine's atoms are `entry(5, …)` — facts *about assertions*, not the assertions themselves. "Row 5 says X and row 29 says ¬X" is a perfectly consistent theory of what was said; contradiction in the record is **data** (a Family C/D flag over T_now), never inconsistency in the engine's own theory. The paraconsistent machinery is needed only where a consumer must render a genuinely aspectual state — F44's triple: a row in force whose load-bearing clause is defeated — and there the two-bit FDE `both` value is a **queryable report value for the review queue**, explicitly interim (each covered case is superseded by first-class DTO fragments as they land), never feeding a gate, never a third truth value in T_now. Minimal-repair `#minimize` is sanctioned as *blame explanation only* ("which smallest retraction restores soundness") — its output is a report artifact; the launder proof is the permanent negative control against ever applying it.

Deliberate rejections, each in one line: **truth-maintenance/retraction systems** (retraction is UPDATE wearing a theory; I3 forbids it); **priority-based defeasible argumentation frameworks** (priorities between laws would be a second, un-ratified law register — precedence between laws is the maintainer's ruling surface, recorded as supersession in the law register itself, §4); **probabilistic/graded logics** (the record's `confidence` is subject data; N=1 apparatus-building has no distribution to ground weights in, and a graded verdict is a boolean that learned to mumble — I9's enemy).

### 3.5 Strict vs defeasible, summarized as a two-tier rule

All derivation is **strict given the EDB at a frontier**. "Defeasibility" exists in exactly two sanctioned places, both *in the data, not the logic*: (i) default negation over the growing record (a T_now judgment can flip at the next frontier — that is the record defeating, not the rule bending); (ii) **flag adjudication** — a Family D/F flag is a defeasible claim about the world (alias_surface has proven false positives, F51; unbound_row has proven instrument artifacts, e17) that awaits an appended human ruling, and the ruling is a new fact the next derivation consumes. No rule is ever overridden by another rule at run time.

## 4. How judgments cite their governing law

The law register exists (F1–F53; the binding subset filed as `acts.ruling` rows, sha-anchored, e.g. F52/F53 as rulings 28/29) but is prose-homed in FINDINGS.md. The engine needs it as data, under the same discipline as everything else:

1. **One authority module** (`judgment_authority.py`, fact-mining side, the kb_ledger idiom): the closed registry of families, per-family verdict enums, law references (F-number, invariant id, ADR id, and — where ratified — the `acts.ruling` row id), engine assignment, mode ceiling, and fixture pointers. From it are **generated**: the `.lp` law constants, the SQL floor's citation column, the judgment-store DDL, and the human-readable registry doc. A parity test pins the live DB against it (ADR-0012 P1/P7: one writer, N derived readers; never two hand-authors of the law list).
2. **Law atoms in the EDB**: `law(LawId, RulingRow, Status)` exported at derivation time, hash-covered. Every ASP rule head that emits a judgment carries the law constant: `judgment(unsound_derivation(E,D), law(f28), frontier(N)) :- …`. In the Python/SQL lanes the citation is a NOT NULL column with an FK into the generated law table. **A judgment atom with no law term does not exist as a type** — the boundary constructor refuses it (ADR-0000: the uncited-flag class made unrepresentable, not reviewed-for).
3. **Laws are themselves ledger-shaped, so law currency is a T_now judgment.** Rulings are append-only and supersedable; the engine applies its own Family C closure to the law register. A banked judgment citing a law that has since been superseded is flagged STALE-LAW (a Family G judgment) — not invalidated (its DerivationRecord pins the law version it lawfully used; that is I3 for judgments) but routed for re-derivation under the successor. This closes a loop the subjects kept failing and the apparatus must not: **the engine's citation of law is record-observed at derivation time by construction** (the law EDB is exported and hash-pinned in the same DerivationRecord), meeting the F42/F46 bar mechanically where memory-grounding failed.
4. **Where letter and spirit diverge** (the CLAUDE.md clause): the engine derives from the letter — that is all a machine honestly can — and the registry marks each law with its ratification depth ({RATIFIED-RULING(row), FINDINGS-ENTRY, BRIEF-INVARIANT, ADR}), so a consumer can see how settled the letter is. Spirit disputes are Family H by definition.

## 5. Vocabulary growth without semantic drift

The growth protocol is the ADR-0008/ADR-0011 discipline applied to judgments, made checkable:

**Admission of a new judgment family (or verdict) requires, jointly:**
1. a governing law reference (at minimum a FINDINGS entry; a gate-feeding judgment requires a ratified ruling);
2. a **banked specimen** — evidence someone already paid for (the §5-marriage windfall pattern: rows 25/27, event 61, the F44 triple, e17 rows 12/17/18). No consumer, no edge; no specimen, no judgment — this is the "no kind without a consumer" law pointed at the engine itself;
3. a **mutation that flips the verdict** (a gate never seen red is a claim, not a net) plus, where the family is nonmonotone, a frontier-replay fixture;
4. an **engine assignment justified by semantics** (assign-don't-compete: monotone closure→SQL, defeater closure→ASP, arithmetic/cores→SMT, aspectual rendering→FDE lens, meta→shell) and either a second independent producer for the differential or a **declared exclusion with its reason**;
5. a **mode ceiling** declared at admission (§6) — a judgment is born retrospective and promoted, never born as a gate.

**Drift guards, mechanical:**
- A family's meaning **is** its fixture set plus its law citation. Any change to a program that alters a banked fixture's output without a corresponding law amendment is DRIFT → red. (The four-arm bit-identical reproduction discipline, standing.)
- Verdict enums are closed and generated from the one authority; adding a member is a reviewed schema change with a consumer, and the differential vocabulary A.5 stays the template.
- Names are minted once; no aliases. Two families that need "the same" verdict word get two namespaced members — shared spelling with divergent meaning is how drift starts (the F26 lesson at the meta level: the moment a name is priced or overloaded, it stops measuring).
- The registry ↔ conformance-map link is generated both ways, so a family silently dropped from the registry surfaces as a REQUIRED-ABSENT row in the map, not as silence (F49's class, foreclosed structurally).

## 6. Live derivation — one semantics, three schedules

The endpoint is not a second, live logic; it is the **same program text** evaluated at three trigger points with different EDB frontiers and different consequence routing. e17 proved the top rung exists; the design's job is the ladder's shape:

| Mode | Trigger | EDB scope | Consequence | What is proven |
|---|---|---|---|---|
| **Write-time** | PreToolUse hook / DB trigger on an attempted row | delta + the bounded closure it touches | REFUSE-AND-TEACH (frozen taught text; refused act lands visibly — a stub or an id gap, never a silent drop: I4) | F53 independence (e17, live); target-existence (e17 row-15 refusal); kind-vocabulary (F37) |
| **Turn-time** | Stop/PostToolUse hook (the L1′/L2′ ladder) | full session record at current frontier | FLAG + inject-if-novel (FindingIdentity-gated), currency **warnings** never gates | designed (HOOK-DESIGN re-founded); the instruments all run retrospectively today |
| **Close-time** | run close | both streams, quiescent; full differential | manifest lines {OK, RED, REQUIRED-ABSENT}; QUARANTINED turns the close red | mechanized and exercised (close_manifest, e17) |

**Promotion criteria, write-time** (each necessary): (i) the judgment is computable on the delta plus a bounded closure inside the trigger budget (§8.1 is the open measurement); (ii) the dishonesty class is **specimen-proven**, never speculative; (iii) the refusal can *teach* — a frozen text naming the honest alternative, because e17's mechanism-reading arc shows the subject converts a legible refusal into genuine compliance, and F26 shows an illegible one (e9's neutral deny) converts into label evasion. An undiagnosable refusal is worse than a flag; **legibility is a promotion requirement, not a nicety**; (iv) the refused act remains visible in the record. Judgments that fail (iii) stay flags forever — most of Family D should, on current evidence.

**The dataflow for a builder:** `ledger_edb.py` (exists) + `acts_edb.py` (exists) + the law/frontier facts (§4.2, new) → `ledger_tnow.lp`/`ledger_assumes.lp`/`ledger_acts.lp`/`judgment.lp` (last one new: the law-citing heads layered over the existing closures) → `ledger_differential.py` (exists; extend verdict routing) → a **judgment store** (new table, FindingIdentity-keyed, columns exactly the §1 type + derivation FK) → consumers: review queue, hook injector, close manifest, conformance map.

## 8. What is unknown, and how to find out

1. **Write-time latency at scale.** Every live record is ≤ 55 rows; clingo cold-start per PreToolUse call at 10³–10⁴ rows is unmeasured. Find out: synthetic-ledger benchmark sweep. If it misses the budget, the candidate is incremental/multi-shot grounding — but that admits a new defect class (incremental state diverging from from-scratch), so the differential discipline must extend: **incremental vs from-scratch bit-identity at every frontier** becomes a standing fixture before any incremental mode ships. Until then, write-time judgments stay delta-local (the current trigger design already is).
2. **The trigger budget itself** is a UX-safety tradeoff with no measurement: how slow can a refusal be before subjects route around the instrumented path? Observable in the acts stream (harness threat model: fix passive primes; pre-register active avoidance as an assumption, don't import an adversarial-subject model).
3. **Cross-stream frontier semantics** (§3.2): needs the interleaved-writes fixture before Family F goes turn-time.
4. **Law-register census.** How much of F1–F51 has acts.ruling backing vs FINDINGS-only status is unenumerated; increment 1 forces the census, and the ratification-depth marking (§4.4) carries the honest answer rather than assuming uniformity.
5. **DTO arrival.** The closures are DTO-ready (built on the scratch lineage, A.6); the open item is inbound re-key *judgment* semantics on the live lineage — conservative re-key debt is a verdict ({REKEY-DEBT(ambiguous-between)}), per the maintainer's ruling that these flags are true information, not false alarms.
6. **Obligation specimens beyond independence.** Family E's vocabulary is BRIEF-sized now (per the standing rule), but only the F53 class has a live refuse-and-teach specimen; G5-shaped (approval-precedes-change) and G7-fixpoint judgments await their first organic specimen before any promotion past retrospective — the `review_fixpoint` line is built and armed for e18, which is exactly the right next datum.

## 9. Honest limits
